# 薄弱点分析

分析用户当前面试题积累情况，结合简历找出薄弱环节。

## 触发

用户使用 `/weak-analysis` 或要求分析薄弱点时触发。

## 会话目标

- 看清“题库覆盖”与“简历要求”之间的差距
- 找出高频方向中的薄弱点与空白点
- 给出可执行的补题清单和优先级
- 分析口径以当前简历技术栈为主，题库方向可动态扩展

## 执行流程

1. 读取 `C:\Users\76741\Desktop\面试\题库\*.md`（排除 `README.md`），按文件维度统计覆盖情况
2. 读取用户简历 `C:\Users\76741\Desktop\面试\简历.pdf`，了解技术栈
3. 对比分析：
   - 简历中提到但面试题未覆盖的技术点
   - 各方向的覆盖深度（基础/中级/深入）
   - Java 后端面试高频考点中尚未涉及的内容
   - 若简历出现新方向且题库无对应 `md`，标记为“应新建方向”
4. 输出分析报告：
   - 当前各方向题目统计
   - 已覆盖的核心知识点
   - 建议补充的方向和具体题目
   - 优先级排序（根据面试高频程度）

## 报告模板

输出时按以下结构组织：

1. 覆盖总览：方向 -> 题目数 -> 覆盖深度
2. 简历匹配差距：简历重点但题库不足的点
3. 高频缺口：市场高频但当前缺失的点
4. 优先级清单（P0/P1/P2）：
   - P0：短期必须补齐（面试高频 + 你简历强相关）
   - P1：次优先（高频但简历相关性一般）
   - P2：扩展项（中低频或偏进阶）
5. 建议新增题目：每个方向给 2-3 题，题干尽量偏原理和排障
6. 目录建议：对缺失方向给出“建议新建的题库文件名”

## 优先级判断规则

- 高频程度越高，优先级越高
- 与简历项目越相关，优先级越高
- 现有覆盖越薄弱，优先级越高
- 题目质量偏定义背诵时，建议替换为原理/排查型题目

## 分析维度

结合简历技术栈，重点关注以下方向的覆盖情况：
- Mysql（`题库/mysql.md`）

- JVM（`题库/jvm.md`）
- Spring/SpringBoot（`题库/spring-springboot.md`）
- Redis（`题库/redis.md`）
- 分布式（`题库/distributed.md`）
- 系统设计（`题库/system-design.md`）
- 计算机网络与操作系统（`题库/network-os.md`）

- ClickHouse（`题库/clickhouse.md`）
- Elasticsearch（`题库/elasticsearch.md`）

## 异常与边界处理

- 若 `题库` 为空：直接输出“当前无可分析题库”，并给出初始化建议
- 若简历无法读取：改为通用 Java 后端岗位基线分析，并明确说明
- 若某方向题目很多但深度不足：标记为“数量覆盖、深度不足”
