# I/O 与网络编程

## 阻塞 I/O、非阻塞 I/O、I/O 多路复用、异步 I/O（AIO）核心区别是什么？请按“发起读请求 -> 数据到达 -> 拷贝到用户态 -> 线程何时返回”解释
### 标准面试回答
一次读 I/O 可以拆成两步：先等待数据到达内核缓冲区，再把数据从内核拷贝到用户缓冲区。  
四种模型差异主要在于“谁等待、何时通知、线程何时可继续执行”：
- 阻塞 I/O：`read` 调用后线程一直阻塞，直到数据就绪并完成拷贝才返回。
- 非阻塞 I/O：`read` 立即返回；无数据时返回 `EAGAIN`，应用需反复轮询。
- I/O 多路复用：先在 `select/poll/epoll_wait` 等就绪事件，就绪后再调用 `read` 拷贝数据。
- 异步 I/O（AIO）：提交请求后立即返回，内核后台完成等待和拷贝，完成后回调/事件通知应用。

服务端高并发场景通常选择“非阻塞 + 多路复用（常见 `epoll`）”，因为可以用少量线程管理大量连接；  
AIO 并非必选，需结合平台生态、复杂度和可观测性综合评估。

### 详细讲解整理
理解这题最稳的方法是抓住“两阶段模型”：
1. 等数据到内核（网卡收包、协议栈处理、放入 socket 接收缓冲区）
2. 从内核拷贝到用户缓冲区（应用可直接消费数据）

阻塞 I/O 在这两步里都会让当前线程等待；  
非阻塞 I/O 不会让系统调用长时间卡住，但会把等待成本转为应用轮询成本；  
多路复用把“等待哪个 fd 就绪”集中交给内核，应用只处理活跃连接；  
AIO 则把“等待 + 拷贝”都交给系统后台，应用在完成通知后消费结果。

工程落地上，常见误区有三点：
1. 把“非阻塞 I/O”误认为“异步 I/O”。
2. 把 `epoll` 误认为 AIO（`epoll` 本质是就绪通知机制）。
3. 认为用了 `epoll` 就一定快，忽略业务线程池、锁竞争、序列化等真实瓶颈。

## 解释一次 read 调用到底发生了什么（从 Java 到网卡）
### 标准面试回答
一次网络 `read` 可以理解为两条并行时间线：收包线和取数线。  
收包线是网卡收包后通过 DMA 把数据搬到内存，内核在中断/软中断中跑协议栈，把 payload 放入目标 socket 的接收队列。  
取数线是 Java 线程调用 `read`，经 JNI 进入系统调用 `read/recv`，内核检查 socket 接收队列：有数据就拷贝到用户缓冲区并返回；无数据则阻塞（阻塞 I/O）或返回 `EAGAIN`（非阻塞 I/O）。

关键点是：`read` 不是直接从网卡取数据，而是从内核 socket 缓冲区取数据。  
`epoll` 优化的是“等待就绪”阶段，不消除普通 `read` 的内核到用户拷贝成本。

### 详细讲解整理
先纠正一个高频误区：应用线程调用 `read` 时，通常不会直接和网卡寄存器交互。  
网卡收包、协议栈处理和 socket 入队往往已经发生，`read` 做的是“从内核缓冲区拷贝到用户缓冲区”。

可以按下面时间线复述：
1. Java 层发起调用  
   - BIO 常见入口：`SocketInputStream.read(byte[])`  
   - NIO 常见入口：`SocketChannel.read(ByteBuffer)`  
2. JVM/JNI 桥接  
   - Java 方法进入本地实现（JNI），最终触发 OS 的 `read/recv` 系统调用。  
   - 此处发生一次用户态 -> 内核态切换。  
3. 内核检查接收队列  
   - 内核定位到该 socket，检查 Recv-Q 是否有可读数据。  
   - 若无数据：  
     - 阻塞 socket：线程挂起，等待网络栈唤醒；  
     - 非阻塞 socket：立即返回 `-EAGAIN`。  
   - 若有数据：进入 copy 阶段。  
4. 网卡收包与协议栈（收包线）  
   - 对端报文到达网卡；  
   - 网卡通过 DMA 把数据写入主内存 ring buffer；  
   - 硬中断做最小工作后，softirq/NAPI 批量处理数据包；  
   - 包经过 L2/L3/L4（以太网/IP/TCP）处理，按四元组找到目标 socket；  
   - payload 进入该 socket 接收队列，并触发等待线程唤醒或 epoll 就绪。  
5. 内核 copy 到用户缓冲区（取数线）  
   - `read/recv` 将数据从 socket 缓冲区拷贝到用户提供的 buffer；  
   - 返回字节数并切回用户态，Java 拿到结果后进入解码/业务处理。  

哪些地方会阻塞或切换：
- 阻塞点  
  - BIO 常阻塞在 `read` 等待数据；  
  - NIO/Reactor 常阻塞在 `epoll_wait`，而非单个连接 `read`。  
- 切换点  
  - 系统调用导致用户态/内核态切换；  
  - 线程睡眠与唤醒导致调度切换；  
  - 中断与软中断带来内核执行上下文切换。  

线上排查建议（面试加分）：
1. 用 `strace -tt -p <pid>` 看 `read` 调用耗时，判断是否长期阻塞在等数据。  
2. 用 `ss -tin` 看 Recv-Q 是否积压，确认“包到了没被及时消费”还是“包根本没到”。  
3. 用 `sar -n DEV`、`ethtool -S` 看网卡层是否丢包。  
4. 用 `perf top` 看 CPU 是否大量消耗在 `softirq`、协议栈或 copy 路径。  

常见误区纠偏：
1. 误区：`read` 直接从网卡读取。  
   - 纠正：普通路径下从 socket 接收队列读。  
2. 误区：NIO 就不阻塞。  
   - 纠正：通常阻塞在事件等待（如 `epoll_wait`），不是完全无阻塞。  
3. 误区：用了 `epoll` 就没有拷贝开销。  
   - 纠正：`epoll` 管的是事件通知，`read` 的数据拷贝仍然存在。  

## Java NIO 视角下，单 Reactor 单线程、单 Reactor 多线程、主从 Reactor 多线程分别怎么工作？如何做选型？
### 标准面试回答
Reactor 的本质是“事件分发器 + 事件处理器”：`Selector` 负责等待 I/O 事件并分发给处理逻辑。  
三种模型是按并发能力和复杂度逐步升级：
- 单 Reactor 单线程：一个线程处理 `accept + read/write + 业务`，实现最简单，但慢业务会阻塞整个事件循环。
- 单 Reactor 多线程：Reactor 线程只做 `accept + I/O 分发`，业务投递到 worker 线程池，吞吐更高，但单个 Reactor 仍可能成为瓶颈。
- 主从 Reactor 多线程：MainReactor 只 `accept`，多个 SubReactor 分别处理各自连接的 `read/write`，业务交给线程池，扩展性和隔离性最好。

选型建议：
1. 低并发、快速上线：单 Reactor 单线程。  
2. 中并发、业务处理变重：单 Reactor 多线程。  
3. 高并发长连接、低延迟要求高：主从 Reactor 多线程（生产主流）。

### 详细讲解整理
可以把三种模型理解为“拆职责”的三个阶段：

第一阶段：不拆（单 Reactor 单线程）
- 一个线程做所有事：监听连接、收发数据、解析协议、执行业务。
- 优势：代码最短、排查容易。
- 风险：任一环节慢都会卡住全局，尾延迟容易放大。

第二阶段：拆业务（单 Reactor 多线程）
- Reactor 线程专注 I/O；业务逻辑移到 worker 池。
- 好处：I/O 循环不再被重业务长期占用。
- 风险：若连接数继续增长，单 Reactor 的事件分发仍会成为瓶颈。

第三阶段：拆接入与读写（主从 Reactor 多线程）
- MainReactor 专注 `accept`；
- 新连接按策略（轮询/最小负载）分发到 SubReactor；
- SubReactor 维护各自 selector，处理读写与编解码；
- 重业务继续交给 worker。
- 好处：横向扩展自然，某个 SubReactor 抖动不会拖垮全部连接。

面试里建议补充两点工程化细节：
1. `EPOLLOUT` 不常驻，只在发送队列非空时订阅，写完立刻取消，避免空转。  
2. 每连接发送队列要有上限，并有背压策略（限流/降级/断开慢连接），防止慢客户端拖垮事件循环。
