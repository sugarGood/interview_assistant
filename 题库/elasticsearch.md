# Elasticsearch

## Elasticsearch 的倒排索引是什么？为什么全文检索快？

倒排索引本质上是 `词项 -> 文档列表` 的映射，而不是 `文档 -> 词项`。\
ES 会先对文本做分词，再为每个词项建立索引，记录它出现在哪些文档中。\
查询时可以先根据词项快速定位命中文档，而不是逐条扫描全文，所以全文检索效率很高。

关系型数据库常见索引更适合结构化字段的精确匹配和范围查询，例如 `id`、`age`、`create_time`。\
而 ES 天生面向全文检索场景，适合标题、正文、描述这类长文本搜索。\
因此像 MySQL 的 `like '%xx%'` 往往容易退化为扫描，而 ES 可以直接基于倒排索引命中文档集合。

## Elasticsearch 里 `text` 和 `keyword` 有什么区别？各自适合什么场景？为什么聚合、排序、精确匹配通常更适合用 `keyword`？

`text` 是全文检索字段，写入时会经过分词器拆成多个 term，适合标题、内容、描述这类自然语言文本。\
它通常用于 `match`、`match_phrase` 这类查询，重点是“搜相关内容”。

`keyword` 是精确值字段，整体作为一个 term 存储，不分词，适合用户名、手机号、状态、标签、订单号这类结构化字段。\
它通常用于 `term`、`terms` 查询，以及聚合、排序、去重等场景。

聚合、排序、精确匹配更适合 `keyword`，核心原因是不分词，字段值保持完整一致，语义稳定。\
如果对 `text` 做这些操作，往往会因为字段先被分词而导致结果不符合预期。\
另外 `text` 默认也不适合直接做聚合和排序，强行开启相关能力通常会带来额外内存开销。

生产中常见做法是一个字段同时保留两种能力，例如 `name` 用 `text` 负责搜索，再增加 `name.keyword` 用于精确匹配、排序和聚合。

## 线上 Elasticsearch 检索接口突然变慢，你会怎么系统性排查？为什么 `keyword` 误配成 `text`、条件没放进 `filter`、以及滥用 `wildcard` 都可能导致查询变慢？

### 详细讲解整理

这类题建议按固定框架回答：`现象 -> 指标 -> DSL -> mapping -> 分片与资源 -> 优化方案`。

先看现象，确认是全部查询都慢，还是只有某个接口慢；是高峰期慢，还是全天都慢；是突然变慢，还是随着数据量增长逐渐变慢。\
这一步是为了快速区分更偏业务查询问题，还是更偏集群资源和索引布局问题。

再看指标，重点关注 `QPS`、平均耗时、`TP95/TP99`、CPU、JVM Heap、GC、磁盘 I/O、搜索线程池、拒绝数、segment 数量、分片大小和节点负载分布。\
如果 CPU 很高，通常说明查询或聚合太重；如果 Heap 高且 GC 频繁，往往说明聚合、字段加载或大结果集导致内存压力；如果磁盘 I/O 高，可能是查询扫描、segment merge 或冷热数据混放导致。

然后看查询 DSL。常见慢查询写法包括：

- 使用高成本的 `wildcard`，尤其是前缀带 `*` 的模式，如 `*abc`
- 使用 `regexp`
- 使用很深的 `from + size` 分页
- 对高基数字段做大范围聚合
- 使用脚本查询、脚本排序
- 把本该放进 `filter` 的精确过滤条件写进 `must`
- 一次返回过多字段或过大结果集

`filter` 更适合纯过滤条件，因为它只负责筛选，不参与相关性评分。\
如果把状态、租户、时间范围这类条件放进 `must`，ES 往往需要同时做匹配和打分，执行成本会更高。\
因此经验上是：能不打分的条件就不打分，能放 `filter` 的尽量放 `filter`。

`wildcard` 会慢，是因为它不能像精确的 `term` 查询那样直接命中单个词项。\
尤其当前缀是 `*` 时，ES 往往需要扫描大量 term，再逐个做匹配，导致候选词项爆炸，CPU 消耗明显增加。\
所以 `wildcard` 应该谨慎使用，更好的方案往往是通过合理的字段建模、前缀搜索、`ngram` 或 `keyword + term` 来替代。

`keyword` 误配成 `text` 也会带来性能和语义问题。\
`keyword` 表示一个完整值，适合精确匹配、过滤、聚合、排序；`text` 会先分词，更适合全文检索。\
如果把手机号、证件号、状态码、业务编码这类本应完整匹配的字段建成 `text`，查询会走分词和全文检索路径，不仅可能查不准，也会让过滤、排序、聚合变得更重。\
生产中常见做法是多字段映射，例如 `name` 既保留 `text` 用于搜索，也增加 `name.keyword` 用于精确过滤、排序和聚合。

接着看索引和 mapping 设计。\
要检查字段类型是否合理、是否存在动态 mapping 膨胀、字段数量是否失控、分词器是否合适，以及是否存在大量小分片。\
如果 mapping 设计不合理，即使集群资源充足，查询依然会慢。

再看分片与节点资源。\
分片太多会导致查询扇出到更多 shard，再由协调节点做汇总，调度和合并开销会上升；分片太大则会拖慢单 shard 查询和恢复速度。  \
如果存在热点索引或热点 shard，往往会出现少数节点先打满，整体响应时间抖动的情况。

最后给优化方案时，建议按“低成本到高成本”回答：

- 先优化 DSL：减少 `wildcard`、避免深分页、限制返回字段、把纯过滤条件下沉到 `filter`
- 再优化 mapping：区分 `text` 和 `keyword`，控制动态字段，多字段映射按需使用
- 再优化索引与分片：按时间分索引、合理设置分片数、减少过多小分片
- 再做架构优化：冷热分层、预聚合、归档历史数据
- 最后再考虑扩容节点或提升硬件

如果要给面试官一个简洁总结，可以直接说：\
Elasticsearch 查询变慢，通常不是单点问题，而是 `DSL 写法`、`mapping 设计`、`分片布局` 和 `节点资源` 共同作用的结果。排查时我会先看现象和指标，再拆 DSL、mapping、分片和资源，优化时优先改查询和索引设计，最后才考虑扩容。

## 为什么不建议随便把 Elasticsearch 的 `primary shard` 设很多？

### 详细讲解整理

这题的核心不是死记参数，而是理解 Elasticsearch 底层的物理结构：`Index -> Shard -> Segment`。

`Index` 是业务层的索引概念，包含 `mapping`、settings、分片数等定义。\
`Shard` 是索引的物理分片，每个 shard 本质上都是一个独立的 Lucene 索引。\
`Segment` 是 shard 底层真正落盘的数据段，Lucene 会不断把新写入的数据整理成多个 segment，并在后续做 merge。

分片的作用主要有两个：

- 把大索引拆开，分布到多个节点上存储
- 让查询和写入可以并行处理

但 shard 不是免费的并行单元。\
每个 shard 都有自己的 Lucene 结构、segment 集合、元数据、文件句柄和调度开销。\
因此 `primary shard` 不是越多越好，过多分片会带来明显的查询和运维成本。

<br />

### 为什么 shard 多了会更重

因为每个 shard 内部并不是一个简单文件，而是由多个 segment 组成。\
一次查询的真实执行链路通常是：

`一次请求 -> 扇出到多个 shard -> 每个 shard 再查自己内部多个 segment -> shard 返回局部结果 -> 协调节点 merge 全局结果`

如果 shard 过多，就会出现：

- 查询扇出增加
- 局部结果 merge 成本变高
- segment 更碎
- 元数据管理对象更多
- heap 压力更大

<br />

segment 可以理解为 Lucene 生成的一批批索引小文件。\
一个 shard 内部通常会包含多个 segment。查询时需要在这些 segment 上分别查找，再在 shard 内部做一次整理。

segment 多不一定表示数据多，很多时候表示“碎片多”。\
碎片化带来的问题包括：

- 查询要处理更多小段
- 打开和维护更多文件
- merge 时增加 I/O 和 CPU 开销
- 需要在内存中维护更多 segment 相关元信息

Elasticsearch 的分片本质上是把一个大索引拆成多个独立的 Lucene 索引，每个 shard 内部又由多个 segment 组成。一次查询会先扇出到多个 shard，再由每个 shard 在内部多个 segment 上执行检索，最后由协调节点合并结果。分片过多虽然提高了拆分粒度，但也会增加查询扇出、merge 成本、segment 碎片化和 heap 元数据开销，因此 primary shard 不能随便设很多。`mapping` 是 index 级别的字段定义，segment 保存的是按 mapping 构建后的实际索引结构，而不是完整 mapping 本身。

## Elasticsearch 是实时还是近实时？为什么不是严格实时？请结合整个写入流程说明

### 详细讲解整理

Elasticsearch 不是严格实时搜索，而是近实时搜索（Near Real Time）。

原因在于，一条文档写入成功后，并不代表它会立刻对搜索可见。\
Elasticsearch 在写入时，数据会先根据 `mapping` 做字段解析，然后写入目标 `primary shard` 的内存缓冲区，同时追加到 `translog`。`translog` 的作用主要是保证故障恢复，避免节点还没来得及把内存数据整理成索引结构时发生宕机导致数据丢失。

写入成功后，`replica shard` 也会同步这次操作。\
但是此时数据虽然已经“写进去了”，却还没有立刻变成可搜索的 `segment`。只有等到 `refresh` 发生时，内存缓冲区中的这批数据才会被生成新的 `segment`，并对查询可见。

这就意味着：

- 写入成功，不等于立刻能搜到
- 要等 `refresh` 后，数据才变得可搜索

因此，Elasticsearch 的本质是写入成功和搜索可见之间存在一个很短的时间差，所以它被称为近实时，而不是严格实时。

一条文档的完整写入流程可以概括为：

1. 客户端把写请求发送到某个 Elasticsearch 节点。
2. 协调节点根据索引、文档 id 和路由规则，计算这条数据应该落到哪个 `primary shard`。
3. 请求被转发到目标 `primary shard`，ES 根据 `mapping` 解析字段，比如判断哪些字段需要分词、哪些字段需要建立倒排索引、哪些字段需要生成 `doc_values`。
4. 数据先写入 `in-memory buffer`，同时追加到 `translog`。前者是后续生成 `segment` 的基础，后者负责故障恢复和持久性保障。
5. `primary shard` 再把这次写操作同步给对应的 `replica shard`，副本也会执行类似过程。
6. 客户端收到写入成功响应，但此时文档还不一定能被搜索到。
7. 只有在 `refresh` 触发后，内存中的这批数据才会被写成新的 `segment`，并变得对查询可见。

<br />

Elasticsearch 之所以不设计成“每次写入都立刻可搜索”，是因为那样会导致频繁生成大量小 `segment`，带来明显的磁盘 I/O、merge 开销和性能损耗。\
因此 ES 采用的是“先写入 buffer 和 translog，再通过周期性 refresh 让数据变得可搜索”的平衡方案。

面试里如果要给一版完整口述答案，可以这样回答：

Elasticsearch 不是严格实时，而是近实时。因为文档写入后，会先进入 `primary shard` 的内存缓冲区，并同时写入 `translog` 来保证故障恢复。这个时候客户端可能已经收到写入成功，但数据还没有立刻对搜索可见。只有等 `refresh` 触发后，内存中的数据才会生成新的 `segment`，搜索才能命中这条文档。所以 Elasticsearch 的特点是写入成功和可搜索之间存在一个很短的时间差，因此它是 Near Real Time，而不是 strict real time。

## Elasticsearch 里的 `refresh`、`flush`、`merge` 分别是干什么的？为什么 `refresh` 之后节点崩溃了，恢复时仍可能依赖 `translog`？

### 详细讲解整理

这三个动作都发生在 Elasticsearch 的写入链路中，但解决的问题完全不同：

- `refresh` 解决“新写入的数据什么时候能被搜到”
- `merge` 解决“segment 太多太碎，怎么合并降低查询和管理成本”
- `flush` 解决“如何形成更稳定的提交点，并推进 `translog` 生命周期管理”

先说 `refresh`。\
文档写入后，会先进入 shard 的内存缓冲区，同时写入 `translog`。\
`refresh` 的作用是把内存缓冲区里的这批数据写成新的 `segment`，并让这个 `segment` 对搜索可见。\
所以真正让数据“变得可搜索”的动作是 `refresh`，不是 `flush`。

再说 `merge`。\
由于写入持续发生、`refresh` 周期性执行，一个 shard 内会不断产生新的小 `segment`。\
如果这些小段长期堆积，查询时就要处理更多 segment，文件数、元数据和 I/O 开销也会上升。\
所以后台会通过 `merge` 把多个小 `segment` 合并成更大的 `segment`，从而减少碎片。

最后说 `flush`。\
`flush` 不是让数据第一次可搜索，而是让当前索引状态推进到一个更稳定的提交点，并让已经不再需要的 `translog` 得到清理。\
你可以把它理解成：`refresh` 更像“先让大家查得到”，而 `flush` 更像“正式记一笔账，并把临时流水账清掉一部分”。

很多人不理解的一点是：为什么都 `refresh` 了，节点崩溃后恢复时还可能依赖 `translog`？

关键在于，下面三件事不是同一时刻完成的：

- 客户端收到写入成功响应
- 数据对搜索可见
- 数据已经进入更稳定的 Lucene commit point

`refresh` 解决的是第二件事，也就是“让数据对搜索可见”。\
它的本质是让新的 `segment` 被当前搜索器看到，因此查询可以命中这批刚写入的数据。\
但 `refresh` 不等同于形成新的 Lucene commit point。

而节点崩溃后的恢复，关注的不是“宕机前能不能搜到”，而是“宕机后拿什么做可靠恢复基线”。\
从恢复视角看，Elasticsearch 依赖的是：

- 上一次 Lucene commit
- 再加上这次 commit 之后的 `translog` 回放

所以即使某条数据在宕机前已经因为 `refresh` 而能被搜到，只要它仍然处于“上一次 commit 之后、下一次 commit 之前”这个区间，恢复时依然可能需要靠 `translog` 把它补回来。

换句话说，可以把语义拆成两层：

- `refresh` 后：数据已经“看得见”
- `flush` 后：恢复基线被进一步推进，后续对长 `translog` 的依赖更小

这就是为什么我说 `flush` 更像“更稳定的提交”。\
它的“稳定”不是说 `refresh` 不稳定，而是说相对于“只是能搜到”，`flush` 会把当前索引状态推进到更正式的提交点，使恢复时不必长期依赖一段很长的 `translog`。

从整体关系上，可以把三者串成下面这条链路：

1. 文档写入 `buffer + translog`
2. `refresh` 触发，把内存数据写成新的 `segment`，让数据可搜索
3. 随着写入持续，产生越来越多的小 `segment`
4. `merge` 后台把小 `segment` 合并成更大的段，减少碎片
5. `flush` 推进提交点，并清理已经不再需要的 `translog`

为什么频繁执行这些动作会有性能问题，也要顺手会答：

- 频繁 `refresh` 会产生大量小 `segment`，导致后续 `merge` 压力更大，查询也更容易受碎片影响
- 频繁 `flush` 会增加提交和磁盘 I/O 成本，影响整体写入效率
- `merge` 本身也会消耗 CPU 和磁盘资源，所以 segment 过碎时会带来额外后台负担

面试里如果要给一版完整答案，可以这样回答：

`refresh`、`flush`、`merge` 都属于 Elasticsearch 写入链路上的关键动作。`refresh` 的核心作用是把内存缓冲区中的数据写成新的 `segment` 并让数据可搜索；`merge` 的作用是把多个小 `segment` 合并成更大的段，减少碎片和查询成本；`flush` 的作用则是推进更稳定的提交点，并清理不再需要的 `translog`。需要注意的是，`refresh` 解决的是“查得到”，不是“恢复时完全不依赖 translog”，所以节点崩溃后恢复仍可能需要结合 `translog`。三者的关系是先 `refresh` 获得可搜索性，再通过 `merge` 降低碎片，`flush` 则负责提交点和恢复成本控制。`refresh` 和 `flush` 过于频繁都会带来额外 I/O 和性能开销。

## Elasticsearch 里的 `doc_values` 是什么？为什么 `keyword`、数值、日期字段通常更适合配合它做排序和聚合，而 `text` 字段一般不这么用？

`doc_values` 本质上可以理解成一种更适合按“文档 -> 字段值”访问的列式存储结构，主要服务于排序、聚合、脚本和去重等场景。\
它和倒排索引的职责不同：倒排索引更适合回答“哪个 term 出现在哪些文档里”，也就是 `term -> doc list`；而 `doc_values` 更适合在命中文档集合上继续读取字段值，例如按某个字段排序、做桶聚合或做统计计算。

为什么排序和聚合更依赖 `doc_values`，核心原因是它们的访问模式不是“先按 term 找文档”，而是“对一批文档读取字段值，再比较、分桶、统计”。\
例如排序时，ES 需要快速拿到每个命中文档在该字段上的值并比较大小；聚合时，ES 需要快速读取字段值并做分桶统计。\
这类操作天然更适合列式读取。

`keyword`、数值、日期、布尔等字段语义稳定，字段值完整，适合直接读取和统计，因此通常更适合配合 `doc_values` 使用。\
而 `text` 字段通常会先分词，更适合全文检索；它的核心价值在于倒排索引，而不是直接拿原始完整值做排序和聚合。\
所以 `text` 一般不直接这样用，否则不仅语义容易失真，还可能引入额外内存和性能问题。

生产中更合理的设计方式通常是多字段映射，例如：

- `title` 用 `text` 负责全文检索
- `title.keyword` 用 `keyword` + `doc_values` 负责排序、聚合、去重

如果字段设计不合理，比如把本来需要排序、聚合的字段建成 `text`，线上常见问题包括：

- 排序和聚合结果不符合预期
- 查询性能下降
- 额外内存消耗
- 高并发下延迟抖动更明显

面试里可以直接概括为：\
倒排索引解决“怎么搜到相关文档”，`doc_values` 解决“拿什么高效读取字段值做排序和聚合”。两者不是替代关系，而是服务不同访问模式。`keyword`、数值、日期字段更适合用 `doc_values`，而 `text` 更适合走全文检索链路。

## Elasticsearch 里为什么深分页通常不建议直接用 `from + size`？为什么 `search_after` 更适合顺序深翻页？它的缺陷和适用场景分别是什么？

### 详细讲解整理

`from + size` 深分页的根本问题在于，Elasticsearch 是分布式分片检索系统，不是单机有序列表。\
每个 `shard` 只能先给出自己的局部有序候选结果，协调节点再把多个 shard 的局部结果做全局 `merge`，才能确定真正的全局排序结果。\
因此深分页不是“简单跳过前面很多条”，而是每个 shard 都要准备更大的局部候选集，协调节点还要汇总更多结果并统一排序。

为什么每个 shard 都要给候选结果，看起来反直觉，但本质上是因为任何一个 shard 都可能持有全局前 N 的结果。\
协调节点在没看到各个 shard 的局部 topN 之前，根本不知道全局最优结果分布在哪个 shard 上，所以只能先让每个 shard 都交出候选，再做统一归并。\
这也是为什么深分页时 shard 和协调节点都会变重。

`search_after` 更适合顺序深翻页，因为它不是基于偏移量跳页，而是基于上一页最后一条记录的排序值继续往后查。\
它本质上是一种基于“排序游标”的分页方式，绕开了 `from + size` 随页码加深而越来越重的问题。

但 `search_after` 有明显限制：

- 不能任意跳页，更适合“下一页 / 加载更多”而不是“跳到第 100 页”
- 必须依赖稳定排序字段，否则容易出现重复或漏数据
- 数据集持续变化时，如果不固定查询视图，容易出现翻页漂移
- 前后端都要保存上一页最后一条记录的排序值，实现复杂度比普通分页更高

为什么通常要配合稳定排序字段？\
因为 `search_after` 依赖的是上一页最后一条记录的排序值作为游标。\
如果排序字段不稳定、不唯一，或者很多文档排序值相同，就容易在翻页边界出现重复和漏数据。\
因此生产里一般会用“主排序字段 + 唯一辅助字段”，例如：

- `createTime desc`
- `_id asc`

它适合的场景主要包括：

- 日志检索、流水查询这类顺序往后翻的场景
- `加载更多`、无限滚动这类游标式分页
- 需要深翻页但不要求随机跳页的查询
- 大结果集顺序遍历

不太适合的场景包括：

- 传统页码分页
- 用户随机跳页
- 排序字段不稳定或无法补唯一排序键的查询

面试里可以直接概括为：\
`from + size` 是偏移分页，越翻越深时每个 shard 和协调节点都要处理更多候选结果，所以成本持续上升；`search_after` 是游标分页，更适合顺序深翻页，但代价是不能任意跳页，而且必须依赖稳定排序。

## Elasticsearch 里的 `PIT` 和 `scroll` 是什么？为什么很多深分页和导出场景会用 `PIT + search_after` 或 `scroll`？

### 详细讲解整理

`PIT` 和 `scroll` 都是为了解决“大结果集遍历时，数据边查边变导致结果不稳定”的问题，只是实现方式不同。

`scroll` 可以理解成：\
Elasticsearch 先为当前查询维持一个临时搜索上下文，然后返回一个 `scroll_id`。\
后续客户端不是重新发起普通搜索，而是拿着这个 `scroll_id` 一批一批往后取结果。\
它更像“开一个查询会话，然后沿着这次会话不断取下一批数据”。

`PIT` 是 `Point In Time`，可以理解成：\
给当前索引拍一个“时间点快照”。\
它本身不负责翻页，通常要和 `search_after` 配合使用。\
这样做的意义是：即使翻页过程中底层数据还在新增、删除或更新，当前这轮查询仍然基于同一份稳定视图，不会因为结果集变化而漂移。

为什么很多场景还要配合 `PIT`？\
因为 `search_after` 解决的是“深分页性能”问题，但不天然解决“翻页过程中数据集变化”问题。\
如果你不固定视图，第一页和第二页之间索引数据发生变化，就可能出现重复、漏数据、顺序漂移。\
而 `PIT` 正好是用来固定这份查询视图的。

为什么导出场景更适合 `scroll / PIT + search_after`？\
因为导出、全量扫描、后台批处理更关心的是：

- 稳定
- 完整
- 不重不漏
- 顺序遍历

它们通常不需要传统页码和随机跳页，所以不适合继续使用 `from + size`。\
这类场景更适合：

- 用 `scroll` 做老式批量遍历
- 用 `PIT + search_after` 做更现代、更灵活的顺序遍历

举个导出日志的例子：\
如果你要导出最近 7 天的 100 万条错误日志，用普通分页会越来越慢，而且导出过程中如果有新日志写入，结果还可能漂移。\
如果用 `scroll`，你可以开启一次遍历会话，持续一批一批取完结果。\
如果用 `PIT + search_after`，你可以先固定当前查询快照，再基于稳定排序持续往后翻页，这样既能避免深分页偏移成本，也能避免数据集变化带来的翻页漂移。

它们各自更适合的场景可以这样记：

- `scroll`：后台导出、离线任务、全量扫描、一次性批量遍历
- `PIT + search_after`：深分页、日志检索、加载更多、大结果集顺序遍历

面试里可以直接概括为：\
`scroll` 更像老式的批量遍历会话，适合后台大批量导出；`PIT` 更像固定查询快照，通常配合 `search_after` 使用。`search_after` 负责高效顺序翻页，`PIT` 负责保证翻页期间结果视图稳定，所以现代很多深分页和导出方案会优先考虑 `PIT + search_after`。

## Elasticsearch 的 `mapping` 为什么不能随便依赖动态推断？如果线上动态字段很多、字段类型还经常漂移，你会担心什么问题？又会怎么治理？

动态 `mapping` 的优势在于接入方便。\
ES 可以根据写入内容自动推断字段类型，开发初期不需要先把字段模型定义得特别完整，因此接入新数据源时很省事。

但在线上环境里，动态 `mapping` 很容易失控。\
真实数据往往来源复杂、字段不稳定、值类型会漂移，尤其在日志、埋点、半结构化 JSON、外部系统对接这类场景里，同名字段的值类型可能变化，或者字段名本身就不断膨胀。\
一旦完全依赖动态推断，索引结构很快就会失去控制。

最常见的风险主要有三类：

- 字段爆炸：动态 key、用户自定义属性、深层嵌套对象不断引入新字段，导致字段数量失控
- 类型冲突：同一个字段有时是数字，有时是字符串，或者今天是对象、明天是普通值，最终导致写入失败或索引设计被污染
- 索引膨胀：无价值字段也被建立索引，mapping 变大，segment、heap、磁盘和查询负担一起上升

这些问题带来的后果通常包括：

- 写入失败，某些批次数据突然入不进去
- 索引元数据膨胀，heap 压力增大
- cluster state 变重，管理和发布成本上升
- 查询性能下降
- 存储浪费明显

因此治理动态 `mapping` 时，核心目标不是“完全禁用灵活性”，而是“让灵活性可控”。\
比较常见的治理思路包括：

- 对核心字段显式定义 mapping，不依赖自动推断
- 使用 index template 统一索引创建规范
- 使用 dynamic template 约束某类字段的落库方式
- 对动态字段做白名单、黑名单或命名规则控制
- 给字段总数设置上限，防止字段爆炸
- 对异常数据、脏字段、临时字段做单独隔离，避免污染主索引

如果面试官追问“线上你最担心什么”，可以直接回答：

- 最担心字段爆炸导致索引元数据失控
- 最担心类型漂移导致写入失败
- 最担心大量无价值字段被建索引，拖慢查询并浪费资源

面试里可以直接概括为：\
动态 `mapping` 适合开发初期快速接入，但不适合核心生产索引完全放开。线上如果字段很多且类型经常漂移，就容易出现字段爆炸、类型冲突和索引膨胀，最终影响写入稳定性、查询性能和集群资源。治理时应该通过显式 mapping、template、dynamic template、字段数量限制和异常数据隔离，把动态能力收敛到可控范围内。

## Elasticsearch 里“候选词项（candidate terms）”为什么会存在？`term dictionary`、`postings`、`wildcard` 之间是什么关系？

### 详细讲解整理

这道题的关键是理解 ES/Lucene 检索不是“扫文档”，而是“先找词，再找文档”。

先看三个基础概念：

- `term dictionary`：字段级词典，保存这个字段里出现过的 term（通常按字典序组织）
- `postings`：倒排列表，记录 `term -> docIDs`（以及可能的 `tf/positions` 等）
- `candidate terms`：本次查询从词典里筛出来的一批“可能匹配”的 term

查询执行链路可以概括为：

1. 解析查询 DSL（`term/match/prefix/wildcard/regexp`）
2. 在 `term dictionary` 里生成候选词项（candidate terms）
3. 逐个读取候选 term 对应的 `postings`
4. 合并文档集合并做过滤/打分/排序，得到最终结果

为什么会有“候选”这个中间层？  
因为 ES 的底层结构是 `term -> docs`，不是 `query -> docs`。  
所以必须先把查询表达式转换成“哪些 term 可能命中”，再去取这些 term 的文档列表。

一个简化例子（字段 `name.keyword` 的词典）：

- `abc`
- `abcd`
- `abcx`
- `abd`
- `babc`
- `xabcx`

当查询是 `prefix: "abc"` 时，候选 term 往往是：

- `abc`
- `abcd`
- `abcx`

因为有固定前缀，候选范围相对可控。

当查询是 `wildcard: "*abc*"` 时，候选 term 可能是：

- `abc`
- `abcd`
- `abcx`
- `babc`
- `xabcx`

因为没有固定起点，通常要匹配更大词典空间，候选词项会明显增加。

为什么候选词项变多会慢？  
候选 term 越多，需要读取的 `postings` 越多，后续集合合并和去重开销越大，CPU 与延迟就越高。  
这也是为什么前缀是 `*` 的 `wildcard` 在高并发场景中常成为性能热点。

面试里可以直接总结：  
`candidate terms` 只是“候选 term 名单”，不直接存文档关系；真正的 `term -> 文档` 关系在 `postings`。查询性能好坏很大程度取决于候选词项规模，候选越可控，查询越稳定。

## Elasticsearch 里 `analyzer`、`tokenizer`、`token filter` 分别负责什么？执行顺序是什么？

### 详细讲解整理

这三个概念要按“流水线”理解，而不是三个孤立名词。

- `analyzer`：整条文本分析方案（一个组合配置）
- `tokenizer`：把原始文本切成 token
- `token filter`：对 token 再加工（如小写化、去停用词、同义词扩展）

标准执行顺序通常是：

`char filter（可选） -> tokenizer -> token filter`

也就是说，`analyzer` 本质上是“字符预处理 + 切词 + 词项后处理”的总装配置。

一个简化示例：

原文：`The iPhone is very good`

1. `tokenizer=standard` 后：
   - `["The", "iPhone", "is", "very", "good"]`
2. `lowercase` 后：
   - `["the", "iphone", "is", "very", "good"]`
3. `stop`（停用词）后：
   - `["iphone", "good"]`
4. 若配置 `synonym`（如 `iphone => apple_phone`）：
   - 可扩展为 `["iphone", "apple_phone", "good"]`（具体以同义词规则为准）

线上常见实践：

- `text` 字段走 analyzer，适合全文检索
- `keyword` 字段默认不分词，适合精确匹配、过滤、聚合、排序

面试里可直接概括为：  
`analyzer` 是分析链路的总配置；`tokenizer` 负责切词；`token filter` 负责对切出来的词做规范化和扩展。顺序通常是 `char filter -> tokenizer -> token filters`，不是反过来。

## `match` 和 `match_phrase` 有什么区别？什么时候用 `match_phrase`？

### 详细讲解整理

两者都用于全文检索，但语义约束不同：

- `match`：偏相关词匹配，只要分词后有命中即可，召回更宽
- `match_phrase`：要求词项按顺序出现，并且通常要求相邻（或在 `slop` 允许范围内）

一个直观例子（查询词：`"华为 手机"`）：

文档示例：

1. `华为 手机 旗舰`
2. `旗舰 华为 手机`
3. `华为 新款 手机 旗舰`

使用 `match` 时：  
1、2、3 通常都可能命中（因为都含相关词项）。

使用 `match_phrase` 且 `slop=0` 时：

- 文档 1 命中（词序与相邻关系满足）
- 文档 2 不命中（词序不满足）
- 文档 3 不命中（中间有间隔词）

若设置 `slop=1`，文档 3 可能命中（允许词间有 1 个位置偏移）。

什么时候更适合用 `match_phrase`：

- 品牌词、专有术语、法规标题等短语语义必须准确的场景
- 需要约束词序和邻接关系，避免结果“词都对但语义不对”
- 二阶段检索中，用 `match` 做召回，再用 `match_phrase` 做收敛或加权

面试里可直接概括为：  
`match` 适合广召回，`match_phrase` 适合短语语义精确匹配；当你关心词序和连续性时，优先考虑 `match_phrase`，并用 `slop` 平衡准确率与召回率。
