# Redis
## Redis 的单线程模型为什么还能支撑高并发？
### 标准面试回答
Redis 高并发能力来自“内存 + 事件驱动 + 高效数据结构”，而不是仅靠单线程。核心原因有四个：
1. 纯内存读写，避免了磁盘随机 IO 的高延迟。
2. 采用 IO 多路复用（epoll/kqueue），一个线程可以同时处理大量连接事件。
3. 命令执行路径短，很多操作复杂度是 O(1) 或接近 O(1)。
4. 单线程执行命令避免了加锁与线程切换开销，降低了 CPU 抖动。

### 详细讲解
“单线程”只解决了并发控制复杂度，不直接等于高性能。Redis 真正快在于它把热点路径做到极致：网络层用事件循环减少阻塞，执行层把常见命令做成常数复杂度，数据层放在内存里把访问延迟压到微秒级。

线上排查要按链路拆分：
- 命令层：看 `SLOWLOG`、大 key、是否用了 `KEYS`/`HGETALL` 这类重命令。
- 网络层：看连接数、网卡带宽、是否充分使用 pipeline。
- 资源层：看 CPU 是否被序列化/压缩/脚本执行打满。

## Redis 持久化 RDB 和 AOF 的区别？线上如何选型？
### 标准面试回答
RDB 是“按时间点快照”，AOF 是“按写命令日志”。
- RDB 优点：文件小、恢复快、适合全量备份；缺点：可能丢失快照间隔内的数据。
- AOF 优点：数据完整性更高（取决于 `appendfsync` 策略）；缺点：文件更大、恢复更慢，需要重写。

线上通常采用混合策略：`RDB + AOF(everysec)`，兼顾恢复速度与数据可靠性。

### 详细讲解
面试里建议把“恢复目标”讲清楚：
- 如果追求极小数据丢失：AOF 更合适。
- 如果追求重启恢复速度：RDB 更有优势。

`appendfsync` 常见三档：
- `always`：最安全，性能开销最大。
- `everysec`：常用折中，最多丢 1 秒数据。
- `no`：交给 OS，吞吐高但风险大。

实际工程里还要关注 AOF 重写时机、磁盘吞吐、主从复制链路延迟，否则会出现“理论可靠、实际抖动”。

## Redis 为什么会出现缓存击穿、穿透、雪崩？如何治理？
### 标准面试回答
- 缓存击穿：热点 key 失效瞬间高并发打到 DB。
  - 治理：互斥重建、逻辑过期、热点不过期。
- 缓存穿透：请求的数据本就不存在，缓存和 DB 都 miss。
  - 治理：布隆过滤器、空值缓存、入参校验。
- 缓存雪崩：大量 key 同时失效或 Redis 集群故障。
  - 治理：过期时间打散、多级缓存、限流降级、熔断兜底。

### 详细讲解
这三者的核心区别是“流量来源”不同：
- 击穿是“少量热点 key”的瞬时冲击。
- 穿透是“无效请求”持续冲击。
- 雪崩是“大面积缓存失效”系统性冲击。

治理时必须把“技术措施 + 容量措施”一起做：
- 技术上做防护（锁、布隆、降级）。
- 容量上做预案（DB 可承载阈值、降级开关、告警阈值）。

否则缓存层一旦抖动，很容易把问题传导到数据库层形成连锁故障。

## Redis 主从复制和哨兵故障切换的关键流程是什么？
### 标准面试回答
主从复制流程：
1. 从节点发起 `PSYNC` 请求。
2. 若复制积压缓冲区可覆盖断点则增量复制，否则触发全量复制（RDB + 增量命令）。
3. 全量完成后进入持续命令流同步。

哨兵切换流程：
1. 哨兵先判定主观下线，再达成客观下线。
2. 多哨兵选举 leader。
3. leader 选择新主（优先级、复制偏移量、延迟等）。
4. 通知其他从节点复制到新主，并向客户端发布拓扑变更。

### 详细讲解
关键风险点有两个：
- 全量复制代价高：大实例会带来网络和磁盘压力，容易引发抖动。
- 切换窗口不可避免：客户端若不及时刷新路由，会产生短暂失败。

工程建议：
- 控制单实例大小，避免一次全量复制过重。
- 客户端必须支持拓扑刷新和重试退避。
- 监控复制延迟、积压缓冲区命中率、故障切换耗时。

## Redis 分布式锁有哪些坑？如何设计更稳妥？
### 标准面试回答
常见问题：
1. 锁过期后业务未完成，导致并发进入。
2. 释放锁时误删他人锁。
3. 主从切换时锁状态丢失，引发一致性风险。

稳妥做法：
1. 加锁用原子命令：`SET key value NX PX ttl`。
2. value 用唯一请求标识；解锁用 Lua 脚本“比对 value 后删除”。
3. 长任务启用自动续期（watch dog）。
4. 强一致场景加入数据库唯一约束/状态机兜底。

### 详细讲解
Redis 锁是“高性能协调工具”，不是“绝对一致性协议”。  
所以面试里要明确边界：它适合控制并发、减少冲突，但不应单独承担资金扣减、库存绝对一致等强一致职责。

设计时建议两层保障：
- 第一层：Redis 锁减少并发冲突。
- 第二层：DB 唯一键/状态机保证最终正确性。

这样即使锁偶发失效，也不会产生不可逆错误数据。
