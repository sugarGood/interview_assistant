# Redis

## Redis 的单线程模型为什么还能支撑高并发？

### 标准面试回答

Redis 高并发能力来自“内存 + 事件驱动 + 高效数据结构”，而不是仅靠单线程。核心原因有四个：

1. 纯内存读写，避免了磁盘随机 IO 的高延迟。
2. 采用 IO 多路复用（epoll/kqueue），一个线程可以同时处理大量连接事件。
3. 命令执行路径短，很多操作复杂度是 O(1) 或接近 O(1)。
4. 单线程执行命令避免了加锁与线程切换开销，降低了 CPU 抖动。

### 详细讲解

“单线程”只解决了并发控制复杂度，不直接等于高性能。Redis 真正快在于它把热点路径做到极致：网络层用事件循环减少阻塞，执行层把常见命令做成常数复杂度，数据层放在内存里把访问延迟压到微秒级。

线上排查要按链路拆分：

- 命令层：看 `SLOWLOG`、大 key、是否用了 `KEYS`/`HGETALL` 这类重命令。
- 网络层：看连接数、网卡带宽、是否充分使用 pipeline。
- 资源层：看 CPU 是否被序列化/压缩/脚本执行打满。

## Redis 里说的“单线程”到底指什么？Redis 6 之后还是单线程吗？

### 标准面试回答

Redis 常说的“单线程”，主要是指 `命令执行层` 以主线程串行执行为主，而不是说整个 Redis 进程所有工作都绝对单线程。

- 在经典模型里，命令执行、核心数据结构修改由主线程负责。
- 后台任务如持久化、AOF rewrite、异步删除，本来就不是都在主线程里完成。
- Redis 6 开始，网络读写可以引入多线程辅助处理，但真正执行命令、修改核心数据结构，仍然主要是串行的。

所以更准确的表达是：`Redis 的核心命令执行是单线程的，网络 IO 和部分后台任务可以多线程化。`

### 详细讲解

这题面试官通常是想看你是否理解 Redis 的“单线程边界”。

如果你只回答“Redis 是单线程的”，很容易被继续追问：

- Redis 6 不是支持多线程了吗？
- 那为什么还说它单线程？

正确理解应该拆成三层：

1. 网络层：客户端连接、收包、发包，Redis 6 之后可以多线程优化。
2. 执行层：命令解析后，真正对 key 空间和数据结构做修改，仍然以主线程串行为主。
3. 后台任务：如 RDB、AOF rewrite、惰性删除/异步释放等，本身就可能由后台线程或子进程参与。

Redis 团队没有把核心命令执行彻底改成多线程，原因是：

- 共享数据结构并发修改会引入锁竞争。
- 系统一致性和可预测性会变复杂。
- 对于大多数短命令场景，串行执行的收益仍然更高。

工程上，这是一种很典型的权衡：`把网络层能并行的部分并行化，把最容易出并发复杂度的核心修改路径继续串行化。`

## 20 个并发请求同时到 Redis，Redis 是怎么处理的？

### 标准面试回答

如果 20 个并发请求同时到达 Redis，`连接事件的感知` 可以几乎同时发生，但 `命令真正执行` 仍然主要由主线程串行处理。

也就是说：

- IO 多路复用可以同时发现很多连接已经可读/可写。
- Redis 事件循环把这些就绪事件取出来。
- 主线程按顺序读取请求、解析命令、执行命令、返回结果。

所以不是“20 个命令并行执行”，而是“20 个请求能被同时感知，然后被快速串行处理”。

### 详细讲解

这题的核心是区分 `并发到达` 和 `并行执行`。

很多人误以为：既然 Redis 用了 IO 多路复用，那 20 个请求是不是就能同时执行？

答案是否定的。IO 多路复用解决的是：

- 如何高效监听大量连接
- 如何避免一个连接阻塞一个线程

它不解决：

- 命令执行时如何并行修改共享数据结构

举个例子，如果这 20 个请求都是：

- `INCR counter`

那 Redis 的效果更像是：

1. 操作系统通知 Redis，这 20 个连接都已经有数据可读。
2. Redis 把这些连接放进事件处理流程。
3. 主线程依次执行 20 次 `INCR`。

也正因为是串行执行，所以像 `INCR` 这种操作天然具有很好的原子性，不需要业务再加锁。

但如果 20 个请求里有一个特别重的命令，比如：

- `KEYS *`
- 大 key 的 `HGETALL`
- 很长的 Lua 脚本

那主线程就会被这个重命令卡住，后面的请求即使已经到达，也只能排队等待。

因此你在面试里最好补一句：`IO 多路复用提高的是连接处理效率，不改变 Redis 核心命令串行执行的事实。`

## Redis 和 Nginx 都是事件驱动，它们的本质区别是什么？

### 标准面试回答

Redis 和 Nginx 在网络模型上很像，都是基于 `非阻塞 IO + IO 多路复用 + 事件循环`，用少量线程处理大量连接。

但两者的核心职责不同：

- Nginx 更偏网络代理、协议处理和流量转发。
- Redis 在事件驱动之后，还要执行真实的数据结构操作，比如 key 查找、哈希修改、跳表维护等。

所以 Redis 比 Nginx 更依赖“命令足够轻、执行足够快”这个前提，也更怕慢命令阻塞主线程。

### 详细讲解

这题考察的是你是否能区分“网络并发能力”和“业务数据操作能力”。

Nginx 的很多请求处理本质上是：

- 接收请求
- 做协议解析
- 路由或转发给 upstream
- 把结果再写回客户端

它更像一个高性能流量中转站，对共享业务数据本身操作较少。

Redis 则不同。请求进入后，它要真正操作数据：

- 查 key
- 改 hash
- 更新 zset
- 执行 Lua
- 维护过期和淘汰相关状态

因此两者虽然都采用事件驱动，但事件之后做的事情不是一个量级：

- Nginx 偏“转发型”
- Redis 偏“内存数据操作型”

这也是为什么 Redis 不能像很多纯网络框架那样，轻易把核心命令执行全面多线程化。因为一旦多个线程同时修改共享 key 空间，就会带来锁竞争、一致性和性能抖动问题。

面试里一句话概括就是：`Nginx 的强项是高效处理网络流量，Redis 的强项是高效处理网络流量之后，再高效完成内存数据操作。`

## Redis 的 LRU 和 LFU 有什么区别？为什么 Redis 不直接使用精确 LRU？

### 标准面试回答

`LRU` 是最近最少使用，核心看“最近有没有访问”；`LFU` 是最不经常使用，核心看“访问频率高不高”。

- `LRU` 更适合热点变化快的场景，比如会话缓存、临时查询结果。
- `LFU` 更适合热点长期稳定的场景，比如热门商品、长期高频配置。

Redis 没有采用严格精确的 LRU，而是采用近似 LRU。原因是精确 LRU 需要维护全局访问顺序，通常要用 `哈希表 + 双向链表`，每次访问都要更新链表位置，内存和维护成本更高。Redis 选择记录访问时间并在淘汰时随机采样一批 key，从中选最久未访问的 key，以更低开销换取足够好的效果。

### 详细讲解

#### 1. LRU 和 LFU 分别在看什么

- `LRU = Least Recently Used`
  - 看最近一次访问时间
  - 核心是“最近还热不热”
- `LFU = Least Frequently Used`
  - 看访问频率
  - 核心是“长期值不值得保留”

一句话理解：

- `LRU` 看最近性
- `LFU` 看频率

#### 2. 两者为什么会得出不同淘汰结果

假设：

- `keyA`：刚刚访问过一次，但历史上整体访问很少
- `keyB`：过去一周都很热，但最近 10 分钟没访问

那么：

- `LRU` 更可能保留 `keyA`，因为它刚被访问
- `LFU` 更可能保留 `keyB`，因为它历史频率更高

所以：

- `LRU` 更适合短期热点快速切换的业务
- `LFU` 更适合长期稳定热点业务

#### 3. 适用场景

`LRU` 适合：

- 会话缓存
- 最近浏览
- 临时查询结果
- 短周期热点页面

`LFU` 适合：

- 热门商品
- 榜单数据
- 高频配置
- 长期稳定热点接口结果

#### 4. 为什么精确 LRU 常常会提到链表

这点很容易混淆，关键在于：

- `记录最近访问时间`
- `维护全局严格访问顺序`

不是一回事。

如果只是给每个 key 记录一个“最近访问时间”，那淘汰时你仍然需要从大量 key 里找出“全局最老”的那个，代价很高。

所以传统精确 LRU 通常会用：

- `哈希表`：根据 key 快速定位节点
- `双向链表`：维护访问顺序

每次访问一个 key，就把它移动到链表头部；链表尾部永远是最久没访问的数据。这样淘汰时直接删除尾部即可，不需要全量扫描。

#### 5. Redis 的近似 LRU 是怎么做的

Redis 不维护全局严格顺序，而是：

1. 为 key 记录最近访问时间信息
2. 淘汰时随机采样一批 key
3. 在采样集合里选最久没访问的 key 删除

所以 Redis 的 LRU 不是“严格全局最老”，而是“采样集合里的最老”。

这样做的好处：

- 实现更简单
- 内存和维护成本更低
- 在大多数线上场景里效果足够好

#### 6. LFU 为什么也不是绝对精确

Redis 的 LFU 也不是无限增长的精确计数器，而是：

- 近似频率统计
- 带时间衰减

引入时间衰减的原因是：

- 老热点不能永远霸占内存
- 业务热点会迁移
- 很久不访问的数据应该逐渐降温

所以 Redis 的 LFU 更准确地说是“带时间衰减的近似频率淘汰”。

#### 7. 为什么 Redis 不直接上精确 LRU

原因不在于做不到，而在于工程上不划算：

- 每次访问都要维护链表顺序
- 访问频繁时维护成本高
- 大 key 空间下内存开销更明显

Redis 是高性能内存数据库，追求的是：

- 开销低
- 速度快
- 效果足够好

所以它选择“记录访问时间 + 采样淘汰”，而不是“全局有序 + 精确淘汰”。

#### 8. 一段可直接复述的回答

“LRU 是最近最少使用，核心看最近是否访问；LFU 是最不经常使用，核心看访问频率。LRU 更适合热点变化快的场景，LFU 更适合长期稳定热点的场景。Redis 没有使用精确 LRU，而是采用近似 LRU：为 key 记录最近访问时间，淘汰时随机采样一批 key，从里面选最久没访问的。之所以不用精确 LRU，是因为精确方案通常要用哈希表加双向链表维护全局访问顺序，每次访问都要调整链表位置，维护成本更高。Redis 更倾向于用较低成本换取足够好的淘汰效果。” 

#### 9. 一句话记忆

- `LRU`：看最近
- `LFU`：看频率
- `精确 LRU`：维护全局顺序，常用双向链表
- `Redis LRU`：记录访问时间 + 随机采样，属于近似 LRU

## 线上有个库存扣减场景，当前方案是：1. 先查 Redis 库存；2. Redis decr 预扣；3. 异步下单写 MySQL；4. 失败后再回补 Redis。最近出现两个问题：高并发下偶发“超卖”；少量请求出现“库存变负后又回正”的抖动，业务方投诉体验差。请你回答：1. 这个方案的核心风险点有哪些？2. 你会怎么设计一个“高并发下不超卖”的方案（要求可落地）？3. Redis、消息队列、MySQL 三者之间如何保证最终一致？4. 如果消息重复消费或回补失败，怎么兜底？

### 标准面试回答

这个方案的问题本质上是：

- `Redis decr` 只保证了 Redis 单点上的原子性
- 但并没有保证 `Redis -> MQ -> MySQL` 这一整条链路的一致性
- 所以在高并发下，很容易出现“预扣成功，但最终订单没落库”或者“重复消费 / 回补异常”带来的库存错乱

#### 1. 核心风险点

1. `Redis 预扣成功 != 最终下单成功`
- 用户请求到了 Redis，`decr` 成功了
- 但后面的 MQ 投递失败、消费失败、DB 写失败，都可能导致库存已经扣了，但订单没真正成功

2. `超卖窗口`
- Redis 扣减只是前置控制
- 如果 MySQL 最终扣库没有做条件更新、没有幂等控制，重复消费或并发写库仍然可能超卖

3. `库存负后回正的抖动`
- 先 `decr` 再异步回补，会导致短时间内库存数值先变负，再恢复
- 对用户和业务方来说，表现就是库存数字抖动、体验差，甚至误导监控

4. `重复消费`
- MQ 至少一次投递语义下，消息可能重复消费
- 如果消费端没有幂等控制，就会重复下单、重复扣库、重复回补

5. `回补失败`
- 如果 DB 写失败后打算回补 Redis，但回补动作本身失败了
- 最终 Redis 和 DB 库存会长期不一致

#### 2. 怎么设计一个高并发下不超卖的方案

更稳妥的思路是：

- `Redis` 做前置削峰和快速拦截
- `MySQL` 做最终库存正确性的硬兜底
- `MQ` 做异步削峰和解耦
- `幂等 + 补偿 + 对账` 保证最终一致

一个常见可落地方案如下：

1. `Redis 原子预扣，不要裸用 decr`
- 用 Lua 脚本把“校验库存是否充足 + 扣减库存 + 记录预扣流水”做成一个原子操作
- 不要只做简单 `decr`
- 否则很难处理并发下的校验和追踪

Lua 里至少做三件事：
- 判断当前库存是否 `>=` 购买数量
- 足够才扣减
- 记录 `requestId/orderId` 的预扣流水，便于后续幂等和补偿

2. `发送下单消息`
- 预扣成功后，把下单消息发给 MQ
- 消息里必须带：
  - `requestId`
  - `orderId`
  - `userId`
  - `productId`
  - `buyCount`

3. `消费端写 MySQL`
- 消费端处理消息时，最终要在 MySQL 做真正库存扣减
- 关键点是必须用条件更新兜底，例如：

```sql
update stock
set available = available - #{num}
where product_id = #{productId}
  and available >= #{num}
```

这个 SQL 非常关键。
因为即使前面 Redis 预扣逻辑偶发失效，MySQL 这里仍然能作为最终“不超卖”的硬兜底。

4. `订单、扣库、回补都做幂等`
- 订单创建要有唯一 `orderId`
- 消费消息时，先检查这个 `orderId/requestId` 是否已经处理过
- 回补库存时也必须幂等，避免重复回补

#### 3. Redis、MQ、MySQL 怎么保证最终一致

这里要明确一点：

- 三者之间很难做到分布式强一致事务
- 工程上一般追求的是 `最终一致`

常见做法是：

1. `事务消息 / Outbox 模式`
- 在本地数据库事务里，同时写：
  - 订单状态
  - 消息表 outbox
- 本地事务提交后，再由后台任务或消息中继把消息投递到 MQ
- 这样能保证“业务状态变化”和“消息发出”最终一致

这比“先写 DB 再直接发 MQ”更稳，因为可以避免 DB 成功了但 MQ 没发出去的问题。

2. `消费端幂等 + 失败重试`
- 消费端处理失败时，MQ 自动重试
- 如果多次失败，进入死信队列
- 后续人工或任务补偿

3. `定时对账`
- 定时比对：
  - Redis 预扣流水
  - MySQL 实际订单和库存
  - MQ 消费状态
- 发现不一致时做修复：
  - 漏扣补扣
  - 漏补回补
  - 错单人工兜底

所以最终一致不是靠某一个点保证的，而是靠：

- `幂等`
- `重试`
- `死信`
- `补偿`
- `对账`

这一整套机制保证的。

#### 4. 消息重复消费或回补失败，怎么兜底

1. `消息重复消费`
- 每条消息必须有全局唯一 `requestId/orderId`
- 消费端处理前先查幂等表或业务状态表
- 已处理过就直接返回成功
- 这样重复消息不会造成重复下单或重复扣库

2. `回补失败`
- 回补动作本身也要幂等
- 回补失败后不能直接丢
- 应该：
  - 重试
  - 失败多次进入死信
  - 由补偿任务扫描并修复
  - 配合监控告警

3. `最终兜底`
- 建立定时对账任务，对比：
  - Redis 库存
  - DB 库存
  - 订单状态
  - 预扣流水
- 一旦发现差异，按规则自动修正或报警人工介入

#### 5. 一段可直接复述的回答

“这个方案的核心问题是 Redis `decr` 只保证单点原子，不保证 `Redis -> MQ -> MySQL` 全链路一致，所以高并发下会出现预扣成功但最终下单失败、重复消费、回补失败等问题，进而导致超卖和库存抖动。更稳妥的做法是：Redis 用 Lua 原子完成库存校验、预扣和流水记录，用 MQ 削峰异步下单，MySQL 用 `stock >= num` 的条件更新做最终不超卖硬兜底。订单创建、库存扣减、库存回补都要用 `requestId/orderId` 做幂等控制。最终一致性上，可以通过 Outbox 或事务消息保证业务状态变化和消息投递的一致性，再结合失败重试、死信队列、定时对账和补偿机制修复漏扣和错补。也就是说，Redis 负责前置拦截和削峰，MySQL 负责最终正确性，MQ 负责异步解耦，幂等和补偿机制负责最终一致。” 

#### 6. 追问扩展：正式场景如何兼顾“用户要快速返回”和“后台最终一致”？

这个问题非常关键，因为很多人一看到：

- Redis 预扣
- MQ 异步
- Outbox
- 补偿
- 对账

就会觉得链路太长，似乎用户下单会被卡很久。  
但真实线上并不是让用户线程等待整条链路完全跑完，而是把链路拆成：

- `同步快路径`
- `异步闭环路径`

也就是说，正式场景追求的是：

- `前台快速响应`
- `后台异步完成重活`
- `最终用状态机和补偿保证一致性`

##### 1. 不是所有场景都需要最重的方案

真实线上通常会按业务强度分层设计：

1. `普通下单场景`
- 并发不高
- 用户更关心立即拿到最终成功/失败结果
- 通常直接走同步事务和 MySQL 条件扣库
- Redis 更多是做缓存、限流或热点保护

这种场景往往可以直接在 MySQL 里做：

```sql
update stock
set available = available - 1
where product_id = ?
  and available >= 1
```

事务提交成功后直接返回“下单成功”。  
这才是很多普通电商场景的主流做法。

2. `秒杀 / 大促 / 热点抢购场景`
- 并发极高
- MySQL 顶不住所有请求同步落库
- 这时才会引入 Redis 预占、MQ 削峰、异步落单、补偿和对账

所以一定要记住：

`复杂方案主要用于高并发热点场景，不是所有下单场景的默认标配。`

##### 2. 用户为什么还能很快收到响应

因为用户请求线程通常只做“最短闭环”：

1. 参数和资格校验
2. Redis Lua 判断库存是否足够并完成预占
3. 生成 `requestId/orderId`
4. 写一条请求流水或可靠消息记录
5. 快速返回

这里返回的往往不是：

- “订单已经 100% 最终成功”

而是：

- `请求已受理`
- `抢购成功，订单处理中`
- `请稍后在订单列表查看结果`

所以前台快，并不是因为后端所有事都做完了，  
而是因为：

`同步阶段只做必要动作，重活都被异步化了。`

##### 3. 正式场景里最关键的是“同步返回什么”

这在产品层面非常重要。

常见有三种返回模式：

1. `直接返回最终成功`
- 适合普通下单
- 链路短，一般是同步事务

2. `返回已受理 / 排队中`
- 适合秒杀和高并发活动
- 前台只拿“受理结果”，最终订单异步生成

3. `先返回抢购资格成功，后续再生成订单`
- 适合极高并发秒杀
- 本质上是把“抢资格”和“落订单”拆成两步

这也是为什么很多秒杀页面不会承诺你点击后立刻拿到最终订单号，  
而是会提示：

- `抢购成功，正在生成订单`
- `请在订单列表查看结果`
- `请在 X 分钟内支付`

也就是说，技术方案本身就是和产品交互设计配套的。

##### 4. 一个典型的正式秒杀链路

真实可落地的秒杀链路通常是：

同步阶段，尽量控制在很短时间内完成：

- 活动资格校验
- Redis Lua 原子判断库存并预占
- 记录预占流水
- 写消息表 / 投递异步任务
- 返回“已受理”

异步阶段再完成：

- MQ 消费
- 创建订单
- MySQL 条件扣减兜底
- 更新业务状态
- 发送通知

超时补偿阶段负责：

- 扫描长时间未闭环的预占记录
- 超时未成功成单则回补 Redis
- 支付超时、订单取消也触发释放库存

所以你可以把它理解成：

- 前台负责“抢资格”和“快速受理”
- 后台负责“真正落订单和最终闭环”

##### 5. 为什么这套机制不会让用户体验变差

关键在于分业务类型：

1. `普通商品下单`
- 用户希望立即知道成败
- 所以一般同步返回最终结果

2. `秒杀、大促、热点场景`
- 用户本来就接受排队、抢购处理中、稍后查看
- 系统的首要目标是吞吐和稳定，而不是同步给出最终状态

所以不是说“所有场景都得异步化”，而是：

`普通场景更重同步确定性，热点场景更重快速受理和异步闭环。`

##### 6. 面试里怎么回答“正式场景到底怎么做”

可以直接这么说：

“正式场景不会把所有一致性动作都放在用户同步请求里完成，而是区分普通下单和秒杀场景。普通下单通常直接走数据库事务和条件扣库，同步返回最终结果；而秒杀这类高并发场景，会把同步链路压缩到 Redis 预占和请求受理，快速返回‘已受理/排队中’，再通过 MQ 异步创建订单，MySQL 负责最终不超卖兜底，补偿和对账负责最终一致。也就是说，复杂机制主要在后台闭环，不会直接让用户线程卡着等完整链路跑完。” 

##### 7. 一句话记忆

- `用户要快返回，不代表后台要立刻全做完`
- `普通下单追求同步确定性`
- `秒杀场景追求快速受理 + 异步闭环`

## Redis 的 RDB 和 AOF 有什么区别？线上如何选型？

### 标准面试回答

`RDB` 是按时间点生成的数据快照，记录的是某一时刻的最终数据结果；`AOF` 是按写命令追加日志，记录的是数据变更过程。

两者的核心差异可以从四个维度理解：

1. 数据记录方式不同

- RDB 记录“结果”，某一时刻把内存数据整体快照到磁盘。
- AOF 记录“过程”，把写命令按顺序追加下来，重启时通过命令重放恢复数据。

1. 数据可靠性不同

- RDB 会丢失两次快照之间的数据。
- AOF 的丢失窗口取决于 `appendfsync` 策略，通常比 RDB 更小。

1. 恢复速度不同

- RDB 通常恢复更快，因为直接加载快照文件即可。
- AOF 通常恢复更慢，因为需要重放日志中的写命令。

1. 工程代价不同

- RDB 文件通常更小，更适合全量备份和灾备传输。
- AOF 文件通常更大，需要关注日志膨胀、重写和磁盘吞吐。

线上一般会采用 `RDB + AOF(everysec)` 的组合方案：

- 用 RDB 保证恢复速度和备份效率。
- 用 AOF 控制数据丢失窗口。

如果 Redis 只是纯缓存，持久化要求可以适当放宽；如果 Redis 承担较重要的业务状态，通常会更重视 AOF 和高可用配置。

### 详细讲解

这题面试里不能只停留在“RDB 是快照，AOF 是日志”这句话上。真正有区分度的回答，需要把 `存储方式`、`恢复逻辑`、`性能影响` 和 `线上取舍` 都讲出来。

先说本质：

- `RDB` 更像“拍照片”，在某个时间点把当前 Redis 内存里的数据整体保存下来。
- `AOF` 更像“记流水账”，把每一条写命令都按顺序记下来，后续通过重放日志恢复状态。

所以更准确的说法是：

- RDB 记录的是某一时刻的“结果”。
- AOF 记录的是一段时间内的“过程”。

#### 1. RDB 的特点

RDB 的优势主要有三点：

- 文件通常更紧凑，适合做全量备份。
- 恢复速度更快，重启时直接加载快照即可。
- 适合做冷备、异地灾备和定时归档。

但它也有明显缺点：

- 两次快照之间如果 Redis 宕机，中间数据会丢失。
- 做快照时通常要 `fork` 子进程，大实例场景下可能带来延迟抖动。
- 如果业务写流量很高，`Copy-On-Write` 带来的额外内存页复制成本也会放大。

举个例子：

- 如果你每 5 分钟生成一次 RDB。
- 上一次快照在 `10:00`。
- Redis 在 `10:04:50` 宕机。

那 `10:00` 到 `10:04:50` 之间的新写入数据，理论上都可能丢失。

所以 RDB 的典型标签是：

- `恢复快`
- `文件小`
- `适合备份`
- `但数据丢失窗口更大`

#### 2. AOF 的特点

AOF 的核心优势是更高的数据完整性。

因为它持续记录写命令，所以只要日志落盘成功，Redis 重启后就可以通过重放这些命令，把数据恢复到接近故障前的状态。

它的主要优势是：

- 数据丢失窗口更小。
- 更适合对业务状态可靠性要求更高的场景。
- 相比 RDB，更能体现“持续记录变更”的能力。

它的主要代价是：

- 文件通常更大。
- 恢复通常更慢，因为需要逐条重放命令。
- 日志会持续膨胀，需要 `AOF rewrite` 做压缩重写。
- 对磁盘吞吐更敏感，磁盘抖动会影响整体表现。

所以 AOF 的典型标签是：

- `更可靠`
- `丢数据更少`
- `但文件更大、恢复更慢、依赖磁盘更多`

#### 3. appendfsync 三种策略怎么理解

AOF 是否真的“更安全”，要结合 `appendfsync` 看。

常见有三种配置：

1. `always`

- 每次写命令都尽量刷盘。
- 安全性最高。
- 性能开销最大。

1. `everysec`

- 每秒刷一次盘。
- 是线上最常见的折中方案。
- 理论上最多丢 1 秒数据。

1. `no`

- 不主动控制刷盘，交给操作系统决定。
- 吞吐更高。
- 数据风险也更大。

如果面试官问你线上最常用哪种，通常回答：

- `AOF everysec`

因为它在性能和可靠性之间通常是最合理的工程平衡点。

#### 4. 为什么线上通常不是二选一，而是组合使用

实际工程里，很多团队不会只开 RDB 或只开 AOF，而是直接采用组合策略：

- `RDB + AOF(everysec)`

原因是两者解决的问题不同：

- RDB 更擅长恢复速度和全量备份。
- AOF 更擅长缩小数据丢失窗口。

两者叠加后，往往能同时兼顾：

- 故障后的恢复效率
- 数据可靠性
- 备份便利性

这也是为什么面试里说“线上一般选 `RDB + AOF(everysec)`”通常是比较稳妥的回答。

#### 5. 什么时候偏向 RDB，什么时候更重视 AOF

如果 Redis 主要承担的是缓存职责，比如：

- 商品详情缓存
- 热点页面缓存
- 推荐结果缓存

这种场景下，即使 Redis 丢了一部分数据，也通常可以回源 DB 或重新预热，所以持久化要求没那么强，RDB 就已经比较够用，甚至可以适当降低持久化强度。

但如果 Redis 承担的是更重要的业务状态，比如：

- 扣减状态
- 排队状态
- 高频计数
- 某些不能轻易丢失的中间态

那一般就会更重视 AOF，至少希望通过 `everysec` 把数据丢失窗口控制在更小范围内。

当然，面试里还要补一句边界：

- Redis 即使开了 AOF，也不是强一致数据库。
- 真正关键的数据仍然要靠 MySQL、事务、幂等、对账和补偿机制兜底。

#### 6. 线上排查和工程关注点

这题如果要答得更像有实战经验，最好顺手补一下线上关注点：

- RDB 重点看 `fork` 耗时、持久化耗时、实例大小、主线程延迟抖动。
- AOF 重点看刷盘延迟、磁盘吞吐、日志重写频率、重启恢复耗时。
- 大实例下，无论是 RDB 还是 AOF，都可能带来明显抖动，所以控制单实例体量很重要。

面试里说到这里，通常会让人感觉你不是只会背概念，而是知道持久化方案会真实影响线上性能。

#### 7. 一段可直接复述的回答

“RDB 本质是时间点快照，记录的是某一时刻的数据结果；AOF 本质是写命令追加日志，记录的是数据变更过程。RDB 的优点是文件小、恢复快、适合全量备份，但会丢失快照间隔内的数据，而且大实例做快照时要关注 fork 抖动；AOF 的优点是数据完整性更高，尤其是 `everysec` 配置下通常最多丢 1 秒数据，但它文件更大、恢复更慢，还要关注 AOF rewrite 和磁盘吞吐。线上一般我会优先考虑 `RDB + AOF(everysec)`，用 RDB 保证恢复效率，用 AOF 缩小数据丢失窗口。如果 Redis 只是纯缓存，可以适当放宽持久化要求；如果 Redis 承担较重要的业务状态，就要更重视 AOF 和高可用方案。”

## Redis 为什么会出现缓存击穿、缓存穿透、缓存雪崩？如何治理？

### 标准面试回答

这三个问题本质上都是“缓存层失效后，请求继续打到下游系统”，但它们的来源和治理重点不同。

1. `缓存击穿`

- 指某个 `热点 key` 在失效瞬间，大量并发请求同时访问这个 key，导致请求集中回源数据库。
- 它的特点是：`热点 key`、`瞬时高并发`、`少量 key 冲击下游`。
- 常见治理方式：
  - 互斥重建
  - 逻辑过期
  - 热点 key 不过期
  - 多级缓存

1. `缓存穿透`

- 指请求访问的数据本身就不存在，所以 Redis miss，数据库也查不到，请求持续穿透缓存层。
- 它的特点是：`无效请求`、`数据库中也不存在`、`缓存层形同虚设`。
- 常见治理方式：
  - 布隆过滤器
  - 空值缓存
  - 参数校验
  - 限流和风控拦截

1. `缓存雪崩`

- 指大量 key 在同一时间集中失效，或者 Redis 整体不可用，导致海量请求直接打向数据库或下游服务。
- 它的特点是：`影响范围大`、`系统性流量冲击`、`容易引发连锁故障`。
- 常见治理方式：
  - 过期时间打散
  - 多级缓存
  - 限流降级
  - 熔断兜底
  - Redis 高可用

一句话总结：

- `击穿` 是热点 key 问题
- `穿透` 是无效请求问题
- `雪崩` 是系统性失效问题

### 详细讲解

这题的关键，不是把三个定义背出来，而是要真正区分：

- 流量从哪来
- 为什么缓存没有挡住
- 下游为什么会被压垮
- 应该怎么针对性治理

很多人面试时只会说：

- 击穿是热点 key 过期
- 穿透是查不到
- 雪崩是大量过期

这当然不算错，但层次太浅。更好的回答方式，是按“问题本质 + 实际场景 + 治理方案”来展开。

#### 1. 缓存击穿

缓存击穿的核心，不是普通的缓存 miss，而是：

- 某个热点 key
- 在过期的那个瞬间
- 大量并发请求同时进来
- 因为缓存 miss，一起去查数据库

所以击穿本质上是：`热点数据失效瞬间引发的尖峰流量问题。`

举个典型场景：

- 某个爆款商品详情页平时访问量极高。
- 商品详情缓存刚好在这个时间点过期。
- 几千上万请求同时过来查这个商品。
- Redis miss 后，大量请求直接回源 MySQL。

如果数据库本身余量不足，就会瞬间被打高，甚至出现超时、线程堆积、连接池耗尽。

击穿的治理重点，不是“让缓存命中率变高”这么泛，而是要解决“失效瞬间不能让大量请求一起回源”。

常见做法有：

1. `互斥重建`

- 缓存失效时，只允许一个线程去查 DB 并重建缓存。
- 其他线程等待、重试或者快速失败。

这是最经典的击穿治理方案，本质上是把“并发回源”变成“单线程回源”。

1. `逻辑过期`

- 缓存中的数据不直接物理删除，而是带一个逻辑过期时间。
- 当发现逻辑过期时，先返回旧值，再由后台异步刷新。

这个方案适合：

- 对短时间旧数据可容忍的场景
- 比如商品详情、推荐列表、配置信息

1. `热点 key 不过期`

- 对极热点数据直接设置成永不过期。
- 通过消息通知、主动刷新、双写等方式更新缓存。

但面试里要注意补一句：

- `不设置过期时间不等于不用治理一致性`
- 仍然要有主动刷新机制，否则只是把击穿问题换成数据陈旧问题。

1. `多级缓存`

- 本地缓存 + Redis + DB
- 即使 Redis miss，一部分流量也能先被本地缓存拦住

#### 2. 缓存穿透

缓存穿透和击穿最大的区别是：

- 击穿查的是“存在的数据，只是缓存刚好失效”
- 穿透查的是“根本不存在的数据”

也就是说：

- Redis 没有
- DB 也没有
- 但请求还在持续来

这样缓存层就没有起到任何保护作用，请求会不断穿过去打数据库。

典型场景有：

- 恶意请求传不存在的商品 ID
- 请求参数非法，例如负数 ID、错误格式 ID
- 扫描大量不存在的订单号、用户号

这些请求即使走完完整链路，也查不到任何数据，但却不断消耗 Redis、DB、应用线程池和网络资源。

穿透的治理重点，是在请求进入数据库之前就把“无效流量”挡掉。

常见方案：

1. `布隆过滤器`

- 先把系统中可能存在的数据 key 放进布隆过滤器。
- 请求进来时先查布隆过滤器：
  - 如果判定一定不存在，直接拦截
  - 如果判定可能存在，再走 Redis 和 DB

它的价值在于：

- 可以在缓存层之前挡掉大量一定不存在的数据请求

面试里可以顺手补一句：

- 布隆过滤器允许“误判存在”
- 但不会“误判不存在”

这里最好再把布隆过滤器本身讲清楚，因为面试官很可能继续追问：

- 为什么它能节省空间？
- 为什么会有误判？
- 为什么适合防穿透？

#### 布隆过滤器

本质上是：

- 一个很长的二进制位数组 `bitmap`
- 配合多个不同的哈希函数

写入一个 key 时，会经过多次 hash，得到多个位置，然后把这些位置都置为 `1`。\
查询一个 key 时，也走同样的多个 hash：

- 只要有任意一个位置不是 `1`，就说明这个 key `一定不存在`
- 如果这些位置全都是 `1`，只能说明这个 key `可能存在`

这就是布隆过滤器的核心特性：

- `可以高概率判断某个元素可能存在`
- `可以准确判断某个元素一定不存在`

也正因为多个不同 key 可能映射到同一批 bit 位，所以才会出现：

- `误判存在`
- 但不会 `误判不存在`

这也是它特别适合防缓存穿透的原因。因为防穿透最重要的是：

- 尽快挡掉那些“肯定不存在”的请求

至于少量“本来不存在，但被误判成可能存在”的请求，最多只是继续走一遍 Redis / DB，不会影响正确性。

工程上使用布隆过滤器时，要注意几个点：

1. `初始化数据来源`

- 可以在系统启动时，把历史有效 key 批量导入布隆过滤器。
- 也可以在新增数据写入成功后，同步把新 key 加入布隆过滤器。

1. `适用场景`

- 适合数据规模大、查询量大、允许少量误判的场景。
- 比如用户 ID、商品 ID、订单号这类主键存在性判断。

1. `不适合单独使用`

- 布隆过滤器不能删除元素，或者删除代价较高（普通布隆过滤器不支持准确删除）。
- 它只能做“存在性预判”，不能代替 Redis 和数据库本身。
- 所以线上通常是：`布隆过滤器 + 空值缓存 + 参数校验` 组合使用。

1. `容量和误判率`

- 如果预计数据量很大，但位图开得太小，误判率会明显升高。
- 所以布隆过滤器需要根据 `元素规模` 和 `可接受误判率` 提前规划容量。

一个面试里比较完整的表达可以是：

“布隆过滤器本质上是位图加多个哈希函数。写入元素时，把多个 hash 位置置为 1；查询时，如果任意一个位置不是 1，就说明这个 key 一定不存在，如果都为 1，只能说明它可能存在。它的特点是空间效率高、判断速度快，允许误判存在，但不会误判不存在，所以特别适合用来拦截缓存穿透里的无效请求。线上一般会把它和空值缓存、参数校验一起使用，而不会单独依赖它解决所有穿透问题。”

1. `空值缓存`

- 如果 DB 查不到，也把这个“空结果”缓存起来
- 比如缓存一个空对象或特殊标记，并设置较短 TTL

这样同一个不存在的 key 再次访问时，就不会再回源 DB。

1. `参数校验`

- 对明显非法的请求参数在入口就拦掉
- 比如：
  - ID 小于等于 0
  - 长度不合法
  - 格式错误
  - 非法枚举值

这类方案成本很低，但收益往往很高。

1. `限流和风控`

- 对短时间内大量访问不存在 key 的请求做限流、黑名单、验证码或网关拦截

所以穿透的本质治理思路是：

- `入口拦非法`
- `缓存层拦空值`
- `过滤器拦不存在`

#### 3. 缓存雪崩

缓存雪崩和前两个问题最大的不同，是它不是某一个 key 或某一类无效请求的问题，而是：

- 大量 key 同时过期
- 或者 Redis 整体不可用
- 导致海量请求一起压到下游系统

所以雪崩本质上是：`缓存层大面积失效导致的系统级流量洪峰问题。`

常见场景：

1. `大量 key 同时过期`

- 比如你做了一次统一缓存预热
- 所有 key 的 TTL 都是 30 分钟
- 30 分钟后大量 key 集中失效

1. `Redis 集群故障`

- Redis 节点宕机
- 哨兵切换期间短暂不可用
- 机房网络故障
- 整个缓存集群访问异常

这时候问题就不是某个热点 key 失效，而是整个缓存层的保护能力下降，导致大量请求同时打向 DB、ES、下游服务，最终可能引发：

- 数据库打满
- 应用线程池堆积
- RT 飙升
- 服务雪崩

雪崩的治理重点，是“不要让大面积缓存失效直接变成大面积下游故障”。

常见方案：

1. `过期时间打散`

- 给 key 的 TTL 增加随机值
- 避免大批 key 同一时间集中失效

例如：

- 不统一设置 30 分钟
- 而是设置成 `30 分钟 + 随机 0~5 分钟`

1. `多级缓存`

- 本地缓存、Redis、静态缓存、CDN 等一起兜底
- Redis 出现抖动时，本地缓存仍能兜住一部分热点请求

1. `限流降级`

- 当缓存层异常时，不能让所有流量都打向数据库
- 要通过限流、降级、默认值、旧值返回等方式保护下游

1. `熔断兜底`

- 当探测到 Redis 异常或 DB 已经高负载时，主动熔断部分请求
- 防止故障继续扩散

1. `高可用架构`

- Redis 主从
- 哨兵
- 集群
- 多机房部署

不过面试里一定要体现边界意识：

- 高可用只能降低 Redis 故障概率
- 不能替代限流降级和系统容量保护

#### 4. 三者最本质的区别

这题如果想说得更清楚，建议按两个维度区分：

1. `流量来源`

- 击穿：热点 key 失效后的真实业务流量
- 穿透：请求本身就是无效流量
- 雪崩：大面积失效或缓存系统故障后的全局流量

1. `影响范围`

- 击穿：通常是少量热点 key
- 穿透：通常是某类非法请求持续冲击
- 雪崩：系统级、大范围冲击

所以一句话记忆就是：

- `击穿 = 热点 key 过期`
- `穿透 = 查不存在的数据`
- `雪崩 = 大量 key 同时失效或 Redis 故障`

#### 5. 工程上要体现“技术措施 + 容量措施”

很多候选人只会讲技术点，比如：

- 布隆过滤器
- 空值缓存
- 热点 key 不过期
- TTL 打散

但真正线上要稳住系统，必须再补一层：

- 限流
- 降级
- 熔断
- 告警
- 下游容量预案
- 热点监控和命中率监控

也就是说，真正完整的回答应该是：

- 技术上减少 miss 和回源
- 系统上防止回源流量压垮下游

这也是面试中很能拉开差距的一点。

#### 6. 一段可直接复述的回答

“缓存击穿是某个热点 key 在失效瞬间，大量并发请求同时回源数据库；缓存穿透是请求的数据本身就不存在，Redis 和数据库都 miss，导致无效请求持续打穿缓存层；缓存雪崩是大量 key 同时过期或者 Redis 整体不可用，造成大面积请求直接压到数据库。三者的区别主要在流量来源和影响范围：击穿是热点 key 问题，穿透是无效请求问题，雪崩是系统性失效问题。治理上，击穿重点是互斥重建、逻辑过期和热点 key 不过期；穿透重点是布隆过滤器、空值缓存和参数校验；雪崩重点是 TTL 打散、多级缓存、限流降级、熔断和高可用设计。线上真正要稳住系统，还要把技术措施和容量保护一起做。”

## Redis 主从复制和哨兵故障切换的关键流程是什么？

### 标准面试回答

Redis 高可用链路可以拆成两部分来看：

1. `主从复制`

- 从节点通过 `PSYNC` 和主节点建立同步关系。
- 如果是首次同步，或者复制积压缓冲区无法覆盖断点，就会触发 `全量复制`。
- 全量复制流程通常是：
  - 主节点执行 `bgsave` 生成 RDB
  - 把 RDB 发送给从节点
  - 从节点加载 RDB
  - 主节点再补发生成快照期间积累的增量命令
- 如果断点还在复制积压缓冲区里，就可以走 `增量复制`，只同步缺失的命令流。

1. `哨兵故障切换`

- Sentinel 先通过心跳检测主节点状态。
- 单个 Sentinel 先判定 `主观下线`。
- 多个 Sentinel 达成一致后，判定 `客观下线`。
- Sentinel 之间选出一个 leader，负责本次故障转移。
- leader 按照 `replica-priority`、复制偏移量、延迟、节点健康状态等因素选择新主。
- 其他从节点切换为复制新主，客户端刷新主节点地址。

工程上要注意：

- 全量复制代价高，大实例容易引发抖动。
- 故障切换存在短暂不可用窗口。
- 主从复制默认是异步的，不保证强一致，主节点宕机前未同步的数据可能丢失。

### 详细讲解

这题面试官一般不是只想听“主从复制 + 哨兵”这几个词，而是想确认你是否真正理解：

- 从节点怎么追主节点
- 为什么会分全量复制和增量复制
- 哨兵如何避免误判
- 为什么还要选举 leader
- 新主节点为什么会选它

#### 1. 主从复制的目标

主从复制的本质，是让从节点尽量追上主节点的数据状态。\
这样主节点故障时，系统才能从从节点里挑一个“数据尽量新”的节点顶上去。

它不仅用于：

- 数据冗余
- 读写分离

还为故障切换提供候选新主。

#### 2. PSYNC 和复制建立

从节点启动后，会向主节点发起 `PSYNC` 请求。\
你可以把它理解成：“我需要从哪里开始同步数据？”

然后主节点根据从节点状态判断：

- 能不能在原有断点上继续追
- 还是必须重新发一份完整数据

这就引出了全量复制和增量复制。

#### 3. 全量复制

通常发生在：

- 从节点第一次连接主节点
- 从节点断开太久，主节点的复制积压缓冲区已经无法覆盖断点

全量复制的完整流程是：

1. 从节点发起 `PSYNC`
2. 主节点判断无法增量续传
3. 主节点执行 `bgsave` 生成 RDB 快照
4. 把 RDB 文件发送给从节点
5. 从节点加载 RDB
6. 主节点把生成快照期间积压的写命令补发给从节点
7. 后续进入持续命令流同步

所以全量复制不是“只传一个 RDB 就结束”，而是：

- `RDB` 负责把从节点拉到某个时间点
- `补发增量命令` 负责把快照之后的数据补齐

#### 4. 增量复制

如果从节点之前同步过，只是中途短暂断开，而主节点的复制积压缓冲区里还保留着断点之后的命令，那么就可以走增量复制。

增量复制的好处很明显：

- 不需要重新 `bgsave`
- 不需要重新传输整份 RDB
- 恢复快
- 对主节点压力更小

它依赖两个关键机制：

- `replication id`
- `复制积压缓冲区 repl backlog`

面试里你不一定要把底层细节背得很细，但要知道：

- backlog 太小，断线一会儿就可能续不上
- 续不上就只能重新触发全量复制

#### 5. 全量复制的风险

工程上全量复制是高风险操作，尤其在大实例场景下：

- 主节点要 `fork`
- 要生成 RDB
- 要占用网络带宽传输快照
- 从节点要加载快照
- 快照期间写流量还要缓存后续补发

这容易带来：

- 主节点延迟抖动
- CPU 和内存压力上升
- 复制延迟放大
- 网络带宽占满

所以面试里如果补一句“要控制单实例体量，减少全量复制频率”，会很加分。

#### 6. Sentinel 的主观下线和客观下线

如果只靠一个 Sentinel 判断主节点是否挂了，误判风险很高。\
因为可能只是：

- 网络瞬时抖动
- 某个 Sentinel 自己异常
- 机房间超时

所以 Sentinel 设计了两层判断：

1. `主观下线`

- 某个 Sentinel 在一段时间内没有收到主节点有效响应
- 它自己认为主节点不可用

1. `客观下线`

- 不止一个 Sentinel 认为主节点不可用
- 达到 `quorum` 票数后，大家才达成共识：主节点真的挂了

这个设计的目的，是降低误判率，避免因为单点网络问题就触发主从切换。

#### 7. 为什么还要选 leader Sentinel

主节点被判为客观下线后，不能所有 Sentinel 一起做故障转移。\
否则可能出现：

- 多个 Sentinel 同时选不同的新主
- 重复发切换命令
- 集群状态混乱

所以 Sentinel 会先选出一个 leader，由它统一执行本次 failover。\
其他 Sentinel 则配合它完成切换。

#### 8. 新主节点怎么选

新主节点不是随机挑一个从节点，而是综合考虑：

- `replica-priority` / `slave-priority`
- 复制偏移量
- 复制延迟
- 节点健康状态
- 与原主断开时间

一般原则是：

- 先排除不健康的
- 再优先选择数据更完整、延迟更低、优先级更高的从节点

因为 Redis 主从复制默认是异步的，选出的从节点越接近原主最新状态，切换后的数据丢失风险越小。

#### 9. 切换后还会做什么

leader 选出新主后，还会继续做这些事：

- 把新主正式提升为主节点
- 让其他从节点改为复制新主
- 通过 Sentinel 通知客户端新的主地址

所以完整的故障转移链路是：

- 发现故障
- 达成共识
- 选 leader
- 选新主
- 重建复制关系
- 通知客户端刷新拓扑

#### 10. 工程风险和边界

这题如果答得更像实战，最好补三点：

- `全量复制代价高`：大实例会带来明显抖动
- `切换窗口不可避免`：主挂到新主生效之间有短暂不可用期
- `异步复制可能丢数据`：主刚写成功但尚未同步到从就宕机，最近一小段数据可能丢失

一句话概括：

`主从复制解决数据同步，Sentinel 解决故障发现和主从切换，但它解决的是高可用，不保证强一致。`

## 什么是 Redis Sentinel 模式？它和普通主从复制是什么关系？

### 标准面试回答

Sentinel 模式是在 `主从复制` 的基础上，再增加一组专门的 Sentinel 进程，用来做：

- 主节点监控
- 故障发现
- 自动故障转移
- 通知客户端新主地址

所以你可以这样理解：

- `主从复制` 解决数据同步
- `Sentinel` 解决主节点故障后的自动切换

Sentinel 本身不存业务数据，它是 Redis 的高可用控制平面。\
它适合：

- 单机容量还够用
- 写流量还不需要分片
- 但不能接受单主故障无人接管的场景

### 详细讲解

很多人第一次学 Redis 高可用时，会把主从复制和 Sentinel 混成一个概念。\
其实它们是两层东西：

- 主从复制是数据层能力
- Sentinel 是运维和高可用能力

如果只有主从复制，没有 Sentinel，会发生什么？

- 主节点挂了
- 从节点虽然有数据
- 但系统不知道该选谁当新主
- 客户端也不知道该连谁

这时候就需要人工介入。

Sentinel 的价值，就是把这部分自动化。

#### 1. Sentinel 的职责

Sentinel 主要做三件事：

1. `监控`

- 持续检查主节点和从节点是否存活

1. `通知`

- 节点状态变化时，向管理员或客户端发布变更信息

1. `自动故障转移`

- 主节点挂了以后，自动选新主并调整主从关系

所以 Sentinel 本质上不是数据节点，而是“高可用控制节点”。

#### 2. 典型拓扑

一个典型的 Sentinel 架构通常是：

- 1 个主节点
- 多个从节点
- 3 个或更多 Sentinel 节点

为什么 Sentinel 常用奇数个，比如 3 个或 5 个？

- 因为要投票
- 奇数更容易形成多数派
- 可以降低平票和误判风险

#### 3. Sentinel 的适用边界

Sentinel 的优点是：

- 自动切主
- 方案相对简单
- 对中小规模业务比较友好

但它的边界也很明确：

- 不做分片
- 不能解决单机内存瓶颈
- 不能解决单主写性能瓶颈

所以一句话总结：

`Sentinel 是单主架构的高可用方案，不是分布式分片方案。`

## 什么是 Redis Cluster？为什么说它是“分片 + 高可用”？

### 标准面试回答

Redis Cluster 是 Redis 官方的分布式集群方案。\
它不是单纯在主从复制上加自动切主，而是同时解决两件事：

- `水平扩容`
- `高可用`

它的核心机制是：

- 把整个 key 空间拆成 `16384` 个槽位 `slot`
- 由多个主节点分别负责不同槽位
- 每个主节点再挂从节点做故障转移

所以更准确地说：

- `Sentinel` 是单主高可用
- `Cluster` 是多主分片高可用

### 详细讲解

如果业务继续增长，单主 Redis 会遇到两个问题：

- 单机内存放不下
- 单主写性能扛不住

这时候仅靠 Sentinel 已经不够了，因为 Sentinel 只解决“主挂了怎么办”，不解决“单机扛不动怎么办”。

Cluster 的设计目标，就是在高可用的基础上再做分片扩容。

#### 1. 16384 个槽位

Cluster 最重要的概念就是 `16384 slots`。

Redis Cluster 不会直接说：

- key 在哪台机器上

而是先做两步：

1. 对 key 计算 hash
2. 计算它落在哪个槽位

然后由某个主节点负责这个槽位。

例如：

- 主节点 A 负责 `0 ~ 5000`
- 主节点 B 负责 `5001 ~ 10000`
- 主节点 C 负责 `10001 ~ 16383`

所以 Cluster 的本质，是把数据按槽位分散到多个主节点。

#### 2. 请求路由

在 Cluster 里，不同 key 可能在不同主节点上：

- 写请求会发到 key 所在槽位对应的主节点
- 读请求通常也走该槽位主节点，或按配置走从节点

因此客户端必须理解集群路由，而不能像单机或 Sentinel 模式那样只知道“当前主是谁”就够了。

#### 3. Cluster 的高可用

Cluster 并不是只有分片，没有高可用。\
它的做法是：

- 每个主节点都可以挂一个或多个从节点
- 当某个主节点挂了，它的从节点里会有一个升为新主
- 接管原来主节点负责的那些槽位

所以你可以把 Cluster 理解成：

- 多个主从组
- 每个主从组负责一部分槽位
- 整个集群共同组成“分片 + 高可用”体系

#### 4. Cluster 的限制

Cluster 很强，但不是万能的。

它最典型的限制是：

- 跨多个 slot 的多 key 操作受限制
- 跨 slot 的事务和 Lua 脚本受限制

因为这些 key 可能在不同主节点上，Redis 没法像单机那样轻松做到跨节点原子操作。

所以 Cluster 更适合：

- 单 key 操作
- 或者可以设计成同 slot 的相关 key 操作

#### 5. hash tag

Redis Cluster 提供了 `hash tag` 机制，用来让多个 key 落到同一个 slot。

例如：

- `user:{123}:profile`
- `user:{123}:cart`

只要大括号里的内容一样，这两个 key 就会落到同一个 slot。\
这对事务、Lua、多 key 操作非常重要。

#### 6. Sentinel 和 Cluster 的区别

这题面试经常一起问，可以直接这么记：

- `Sentinel = 单主高可用`
- `Cluster = 多主分片高可用`

更展开一点：

1. `Sentinel`

- 不分片
- 所有数据仍然在一个主节点上
- 重点解决自动切主

1. `Cluster`

- 有分片
- 多个主节点各自负责一部分数据
- 同时解决扩容和高可用

#### 7. 选型场景

适合 `Sentinel` 的场景：

- 数据量还不大
- 单机内存够用
- 更关心高可用，不急着分片

适合 `Cluster` 的场景：

- 单机内存放不下
- 单主写性能成为瓶颈
- 需要水平扩容

一句话总结：

`Sentinel 适合中小规模单主高可用，Cluster 适合需要分片扩容的大规模场景。`

## Redis Sentinel 和 Cluster 有哪些关键参数？线上怎么理解它们？

### 标准面试回答

如果面试官问“你对 Sentinel / Cluster 的参数了解多少”，建议至少掌握下面这些：

1. `主从复制相关`

- `replicaof`
- `masterauth`
- `repl-backlog-size`
- `repl-backlog-ttl`
- `repl-timeout`
- `repl-diskless-sync`
- `min-replicas-to-write`
- `min-replicas-max-lag`

1. `Sentinel 相关`

- `sentinel monitor <name> <ip> <port> <quorum>`
- `sentinel down-after-milliseconds`
- `sentinel failover-timeout`
- `sentinel parallel-syncs`
- `replica-priority`

1. `Cluster 相关`

- `cluster-enabled`
- `cluster-config-file`
- `cluster-node-timeout`
- `cluster-require-full-coverage`
- `cluster-migration-barrier`
- `cluster-replica-validity-factor`
- `cluster-announce-ip / port / bus-port`

理解这些参数时，不能只背名字，而要知道它们分别控制：

- 复制是否稳定
- 故障探测是否敏感
- 故障转移是否平稳
- 槽位不完整时集群是否继续服务
- 云环境或容器环境下节点地址是否正确通告

### 详细讲解

#### 1. 主从复制相关参数

`replicaof <masterip> <masterport>`

- 指定当前节点去复制哪个主节点
- 这是建立主从复制关系最基础的配置

`masterauth`

- 从节点连接主节点时使用的认证密码
- 主有密码、从没配对，就建立不了复制关系

`repl-backlog-size`

- 复制积压缓冲区大小
- 它决定断线重连后，增量复制命中的概率
- 太小的话，断一会儿就可能续不上，只能重新全量复制

`repl-backlog-ttl`

- 没有从节点连接时，复制积压缓冲区保留多久
- 配太短会更省内存，但也更容易导致后续重连只能走全量复制

`repl-timeout`

- 复制链路超时时间
- 太短容易误判网络抖动，太长则故障感知变慢

`repl-diskless-sync`

- 是否开启无盘复制
- 开启后，主节点可以直接通过网络流把 RDB 发给从节点，减少磁盘 IO

`min-replicas-to-write` 和 `min-replicas-max-lag`

- 这两个参数用于主节点写保护
- 如果健康从节点数量不足，或复制延迟太大，主节点可以拒绝写入
- 它不是强一致，但能减少“主写成功却没有可靠副本”的风险

#### 2. Sentinel 相关参数

`sentinel monitor <name> <ip> <port> <quorum>`

- 用来声明一个被监控的主节点
- 其中 `quorum` 最关键，表示至少需要多少个 Sentinel 都认为主不可用，才会判为客观下线

`sentinel down-after-milliseconds`

- Sentinel 多久没收到节点有效响应，就会先把它判成主观下线
- 太小容易误判，太大故障发现慢

`sentinel failover-timeout`

- 一次故障转移流程的总超时控制参数
- 会影响 leader 选举、重试和 failover 的节奏

`sentinel parallel-syncs`

- 新主切换完成后，允许多少个从节点并发去同步新主
- 配小一点更稳，配大一点收敛更快但新主压力更大

`replica-priority`

- 控制从节点被选为新主的优先级
- 值越小优先级越高
- 如果设置为 `0`，表示该从节点不参与升主

#### 3. Cluster 相关参数

`cluster-enabled yes`

- 开启 Cluster 模式

`cluster-config-file`

- 集群元信息文件，保存节点、槽位、角色等状态
- 这是 Redis 自动维护的，不建议人工改动

`cluster-node-timeout`

- 节点间多久联系不上，就会被认为存在故障风险
- 它影响故障探测速度和误判率

`cluster-require-full-coverage`

- 当部分槽位无人负责时，集群是否仍继续对外提供服务
- `yes` 更保守，要求槽位完整
- `no` 更偏可用性，允许其他正常槽位继续工作

`cluster-migration-barrier`

- 控制从节点迁移时的门槛，避免某些主节点一个副本都没有

`cluster-replica-validity-factor`

- 控制从节点与主断开太久后，是否还有资格参与故障转移
- 本质上是在控制“过旧副本”是否允许升主

`cluster-announce-ip / cluster-announce-port / cluster-announce-bus-port`

- 在 NAT、容器、云环境下尤其重要
- 用来明确告诉其他节点和客户端，应该通过什么地址访问当前节点
- 这类配置不对，常见结果就是节点互相能看到但连不上，或者客户端拿到错误地址

#### 4. Cluster 里常见的关键机制

这些不完全是配置参数，但面试里经常和参数一起问：

`16384 slots`

- Cluster 把整个 key 空间分为 16384 个槽位

`MOVED`

- 表示某个 slot 已经正式迁移到其他节点
- 客户端应该更新路由表

`ASK`

- 表示 slot 正在迁移过程中
- 当前请求临时去另一个节点处理，但不一定立即刷新全量路由

面试里一句话记忆：

- `MOVED = 正式搬家`
- `ASK = 搬家途中`

#### 5. 线上怎么理解这些参数

这类参数不要死记硬背成配置项清单，更重要的是理解它们对应的工程问题：

- `复制参数` 决定你是不是频繁触发全量复制
- `Sentinel 参数` 决定你是否容易误判、切换是否平稳
- `Cluster 参数` 决定你故障探测灵敏度、槽位缺失时的服务策略、容器环境下的连通性

所以如果面试官追问参数，最好的思路是：

- 先说它控制什么
- 再说配小/配大分别有什么风险
- 最后补一句它在线上常见的影响

#### 6. 一组最值得优先记住的参数

如果要先抓最有价值的，可以优先记这 10 个：

主从复制：

- `repl-backlog-size`
- `repl-timeout`
- `min-replicas-to-write`
- `min-replicas-max-lag`

Sentinel：

- `quorum`
- `down-after-milliseconds`
- `failover-timeout`
- `parallel-syncs`

Cluster：

- `cluster-node-timeout`
- `cluster-require-full-coverage`

这 10 个参数已经足够支撑你把面试答到“不是只会背概念”的水平。

## Redis 分布式锁有哪些坑？如何设计一个更稳妥的方案？

### 标准面试回答

Redis 分布式锁可以用，但它本质上是一个 `高性能协调工具`，不是绝对强一致协议。  
所以面试里既要讲“怎么加锁”，也要讲“它有哪些坑，以及怎么兜底”。

最基础的正确加锁方式是：

```text
SET lock_key requestId NX PX 30000
```

其中：

- `NX` 表示只有锁不存在时才加锁成功
- `PX` 表示设置过期时间，避免死锁
- `requestId` 必须是唯一值，用来标识锁的持有者

常见坑主要有：

1. `setnx + expire` 分两步做，不是原子操作，可能导致死锁
2. 锁过期了但业务还没执行完，会导致并发进入
3. 解锁时直接 `DEL`，可能误删别人刚拿到的锁
4. 主从切换时锁还没同步到从节点，可能导致锁丢失
5. 业务异常、GC 停顿、网络抖动等都会放大锁超时问题

更稳妥的方案一般包括：

- 加锁用 `SET key value NX PX ttl`
- value 使用唯一 `requestId`
- 解锁用 Lua 做“比较 value 后删除”
- 长任务引入自动续期 `watch dog`
- 获取失败时做重试退避和超时控制
- 对强一致场景加数据库唯一约束、状态机、幂等机制做最终兜底

一句话总结：

`Redis 锁负责减少并发冲突，数据库约束负责保证最终正确性。`

### 详细讲解

这题最适合按“从基础到完善”的方式讲。  
因为 Redis 分布式锁不是会一个命令就算会了，而是要一层层补坑。

建议按下面这条主线来理解：

- 第 1 层：先做到“能锁住”
- 第 2 层：避免死锁
- 第 3 层：避免误删他人锁
- 第 4 层：避免业务未完成锁先过期
- 第 5 层：考虑主从切换和分布式故障
- 第 6 层：认识到 Redis 锁不是强一致协议，要加业务兜底

#### 1. 为什么需要分布式锁

分布式锁的本质是：

- 你的服务不是单机
- 有多个进程、多个节点、多个实例
- 它们可能同时处理同一份共享资源
- 你希望同一时刻只有一个执行者进入临界区

典型场景有：

- 库存扣减
- 用户重复下单控制
- 定时任务防重复执行
- 某些状态流转只能有一个执行者推进

如果只是单机应用，使用本地锁就够了；  
但一旦是多台机器，本地锁就失效了，所以才需要 Redis 这种外部协调组件。

#### 2. 第一版：只用 SETNX

最朴素的想法是：

- 谁先把某个 key 创建成功
- 谁就拿到了锁

例如：

```text
SETNX lock:order:123 1
```

如果成功：

- 说明当前线程拿到了锁

如果失败：

- 说明锁已经被别人持有

这个思路本身没问题，但立刻会遇到一个致命问题：

- 如果拿锁成功后，进程挂了，锁怎么办？

因为这个 key 会一直留在 Redis 里，别人再也拿不到锁，这就形成了死锁。

#### 3. 第二版：SETNX + EXPIRE

于是很多人会继续改：

1. 先 `SETNX lock_key 1`
2. 再 `EXPIRE lock_key 30`

含义是：

- 先抢锁
- 抢到后给锁加 TTL
- 即使进程挂了，过期后锁也会自动释放

这个思路比只用 `SETNX` 更进一步，但它仍然有一个经典问题：

- 这两个操作不是原子的

可能出现：

1. `SETNX` 成功
2. 还没执行 `EXPIRE`
3. 进程就挂了

结果就是：

- 锁拿到了
- 但没有过期时间
- 最终还是死锁

所以面试里一定要明确讲：

`setnx + expire 分两步做，是不安全的。`

#### 4. 第三版：SET NX PX，原子加锁

为了解决上面的问题，Redis 分布式锁最基础、也最标准的写法是：

```text
SET lock_key requestId NX PX 30000
```

这里每个部分都要讲清楚：

- `lock_key`：锁的名字
- `requestId`：唯一请求标识，标识这把锁是谁持有的
- `NX`：只有 key 不存在时才 set 成功
- `PX 30000`：设置过期时间 30 秒，避免死锁

这个命令把“抢锁”和“设置过期时间”合并成了一个原子操作，  
到这里，Redis 锁才算进入了“基础可用版”。

#### 5. 为什么 value 不能随便写 1

很多人会写成：

```text
SET lock_key 1 NX PX 30000
```

这依然不够稳妥。  
因为后面解锁时，你必须知道：

- 这把锁是不是我自己的

如果 value 只是固定的 `1`，那么任何线程都无法判断锁归属。  
所以 value 必须是唯一值，比如：

- UUID
- requestId
- 进程 ID + 线程 ID + 请求流水号

总之要能唯一标识：

- `锁是谁加的`

#### 6. 为什么解锁不能直接 DEL

很多人觉得业务执行完后直接：

```text
DEL lock_key
```

就行了。  
但在分布式场景里，这会带来非常典型的问题：

1. 线程 A 拿到锁，TTL 30 秒
2. A 的业务执行很慢，超过了 30 秒
3. 锁自动过期
4. 线程 B 又拿到了这把锁
5. 线程 A 终于执行完，直接 `DEL lock_key`

结果：

- A 把 B 的锁删掉了

这就叫：

- `误删他人锁`

这是 Redis 锁面试里最经典的坑之一。

#### 7. 正确解锁：先校验 value，再删除

所以正确做法应该是：

- 删除之前，先判断当前锁里的 value 是否还是自己最初写入的 `requestId`

伪代码是：

```text
if get(lock_key) == requestId:
    del lock_key
```

但这里依然有问题：

- `GET` 和 `DEL` 如果分两步执行，也不是原子的

可能发生：

1. 线程 A `GET` 到还是自己的锁
2. 锁刚好过期
3. 线程 B 重新拿到了锁
4. 线程 A 再执行 `DEL`
5. 删掉了 B 的锁

所以最终要用 Lua 来做原子解锁。

#### 8. 最终版解锁：Lua 脚本原子删除

正确的解锁方式是把“比较 value”和“删除 key”放进一段 Lua 脚本里原子执行：

```lua
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end
```

它的含义是：

- 只有当当前锁里的 value 仍然等于我的 `requestId`
- 才允许删除

否则：

- 什么都不做

到这里，一个相对正确的基础版本 Redis 锁已经具备了：

- 原子加锁
- 自动过期
- 唯一 owner 标识
- 原子解锁

#### 9. 还不够：业务没执行完，锁先过期

前面解决的是“死锁”和“误删锁”问题，  
但还有一个更常见的工程问题：

- 业务还没执行完，锁先过期了

比如：

- 锁 TTL 30 秒
- 平时业务执行 5 秒
- 但某次遇到慢 SQL、GC、网络抖动，执行了 50 秒

那就会出现：

1. 线程 A 拿到锁
2. A 业务还没结束，30 秒到了
3. 锁自动过期
4. 线程 B 又拿到锁
5. A 和 B 同时进入临界区

所以第三个经典坑就是：

- `锁 TTL 早于业务执行完成时间`

#### 10. 解决办法：自动续期

应对这个问题，最稳妥的做法不是简单把 TTL 设得无限长，而是引入：

- `自动续期`

思路是：

- 线程拿到锁后
- 启动一个后台续期机制
- 只要业务还在执行，并且锁还是自己的
- 就定时把 TTL 往后延

这也是很多框架里 `watch dog` 的核心思路。

例如：

- 初始 TTL 30 秒
- 每隔 10 秒检查一次
- 如果业务还没结束，且锁 owner 还是当前请求
- 再把 TTL 延回 30 秒

这样就能避免：

- 业务未完成，锁先过期

#### 11. 自动续期也不能乱续

续期机制也有边界。  
续期前仍然要校验：

- 当前锁是不是自己的

否则可能出现：

- 锁已经过期，被别人拿走
- 你还在继续给这把锁续期

那就相当于把别人的锁错误地延长了。

所以续期和解锁一样，核心原则都是：

- `先确认 owner`
- `再做操作`

#### 12. 没拿到锁怎么办

很多人只会讲“拿到锁以后怎么做”，却忽略了：

- 没拿到锁时怎么办

如果只是简单地 while 自旋：

```text
while not get_lock:
    retry
```

会带来：

- 高频打 Redis
- 无意义 CPU 消耗
- 在高竞争场景下形成放大效应

更稳妥的做法是：

- 设置最大等待时间
- 限制重试次数
- 每次重试之间加退避或随机抖动
- 超时后快速失败

所以一个完整的锁设计，不只是“怎么加锁”，还包括：

- `竞争失败时如何优雅退出`

#### 13. 进入分布式场景：主从切换导致锁丢失

前面讲的都是单 Redis 节点视角。  
但真实环境里，Redis 常常是：

- 主从
- 哨兵
- 集群

这时就会遇到第四个关键问题：

- `主从切换导致锁丢失`

典型时序如下：

1. 线程 A 在主节点加锁成功
2. 锁还没来得及同步到从节点
3. 主节点挂了
4. 从节点升为新主
5. 新主里没有这把锁
6. 线程 B 又拿到了锁

结果就是：

- A 和 B 都可能认为自己持有锁

这里暴露的本质是：

- Redis 主从复制默认是异步的

所以 Redis 分布式锁天然存在一致性边界。

#### 14. Redis 锁是不是不能用

结论不是“不能用”，而是：

- `Redis 锁能用，但要知道它适合什么，不适合什么`

它很适合：

- 控制重复执行
- 资源竞争协调
- 减少并发冲突
- 做高性能分布式协调

但它不适合单独承担：

- 资金扣减绝对一致
- 库存绝对不超卖
- 跨系统不可逆状态流转的最终正确性

所以面试里最好明确一句：

`Redis 锁是协调工具，不是最终真相来源。`

#### 15. RedLock 要不要知道

面试官有时会追问 `RedLock`。

简单理解：

- RedLock 试图在多个独立 Redis 实例上取多数锁
- 希望降低单主故障和复制延迟带来的锁丢失风险

但工程界对它一直存在争议，原因在于：

- 实现复杂
- 对时钟、网络延迟、暂停、网络分区依赖比较敏感
- 它也不是绝对强一致协议

所以更成熟的说法不是盲目吹 RedLock，而是：

- `RedLock 是提升可靠性的一种思路，但不是终极强一致方案`

#### 16. 更稳妥的工程方案应该长什么样

如果要从基础走到完善，一个更完整的 Redis 锁方案通常会包含：

1. `原子加锁`

```text
SET key requestId NX PX ttl
```

2. `唯一 owner`
- value 用唯一 `requestId`

3. `原子解锁`
- 用 Lua 做“比较 value 后删除”

4. `自动续期`
- 对长任务加 watch dog

5. `重试和退避`
- 没拿到锁时不要无脑自旋

6. `监控和告警`
- 关注持锁时间、获取失败率、续期异常、释放失败率

7. `业务兜底`
- MySQL 唯一约束
- 乐观锁
- 状态机
- 幂等表
- 对账与补偿机制

真正稳妥的方案不是：

- `只靠 Redis 命令`

而是：

- `Redis 锁 + 业务幂等 + DB 约束 + 补偿机制`

#### 17. 为什么业务兜底这么重要

这是这题最有深度的地方。

即使 Redis 锁做得已经很规范，依然可能遇到：

- 主从切换
- 网络分区
- GC 停顿
- 进程卡死
- 续期失败
- 客户端重试乱序

所以最终必须有一层“最后防线”。

例如库存扣减场景里，数据库最终仍然要做：

```sql
update stock
set count = count - 1
where product_id = ?
  and count >= 1
```

或者配合：

- 唯一索引
- 幂等流水
- 状态机 CAS

这样即使 Redis 锁偶发失效，也不会导致不可逆错误。

一句话理解：

- `Redis 锁负责减少冲突`
- `数据库约束负责兜住底线`

#### 18. 不同场景要用不同强度的方案

这题如果想答得更像实战，最好体现出“不同业务不同策略”：

1. `定时任务防重复执行`
- Redis 锁通常已经足够
- 因为偶尔重复一次通常成本可控

2. `库存扣减`
- Redis 锁可以用
- 但必须配数据库条件更新、幂等、消息补偿

3. `资金扣减`
- 不能把 Redis 锁当核心一致性手段
- 应该以数据库事务、账务系统、状态机为核心

所以面试里最好明确：

`不是所有场景都应该依赖同一强度的 Redis 锁设计。`

#### 19. 一段可直接复述的完整回答

“Redis 分布式锁最基础的正确做法，是用 `SET key value NX PX ttl` 原子加锁，其中 value 必须是唯一请求标识，不能简单用 `setnx + expire` 分两步做，否则会有死锁风险。拿到锁以后，解锁不能直接 `DEL`，因为锁过期重入后可能误删别人的锁，所以要用 Lua 脚本做‘比较 value 后删除’的原子解锁。如果业务执行时间可能超过 TTL，还要加自动续期，也就是 watch dog，避免业务没执行完锁先过期。再进一步，获取失败时要做重试退避和超时控制。需要注意的是，Redis 锁本身并不是强一致协议，在主从切换、网络分区等场景下仍可能失效，所以对于库存、订单、资金这类关键场景，不能只依赖 Redis 锁，还要结合数据库唯一约束、乐观锁、状态机或幂等机制做最终兜底。也就是说，Redis 锁负责减少并发冲突，数据库约束负责保证最终正确性。” 

#### 20. 一句话记忆

- `能加锁，不代表锁正确`
- `能正确解锁，不代表业务绝对安全`
- `Redis 锁负责减少并发冲突，不单独保证强一致`

## 如果让你设计一个高并发限流器，你会怎么用 Redis 实现？

### 标准面试回答

如果让我用 Redis 设计一个高并发限流器，我通常会按“精度、成本、流量形态”来选择方案。  
常见方案有：

1. `固定窗口`
- 按固定时间段统计请求数
- Redis 常见实现是 `INCR + EXPIRE`
- 优点是简单、性能高
- 缺点是存在窗口边界突刺问题

2. `滑动窗口`
- 如果要求严格按“最近 N 秒”统计，就不按自然时间段分桶，而是直接统计真实时间范围内的请求数
- Redis 最常见实现是 `ZSET + 时间戳`
- 每次请求到来时：
  - 先删除窗口外的旧请求
  - 再统计当前窗口内请求数
  - 未超过阈值才允许放行
- 优点是精度高，能很好解决固定窗口边界问题
- 缺点是内存和操作成本更高

3. `漏桶`
- 请求先进桶，再按固定速率流出
- 特点是输出速率稳定，适合保护下游
- 缺点是高峰流量可能排队甚至丢弃

4. `令牌桶`
- 系统按固定速率生成令牌
- 请求来了先拿令牌，有令牌才放行
- 特点是既能限制平均速率，又允许一定突发流量
- 在工程上比漏桶更常见

5. `分布式场景下怎么保证原子性`
- 限流的核心是把：
  - 判断是否还能通过
  - 更新计数或扣减令牌
  做成一个原子操作
- 简单固定窗口可以用 Redis 自增命令配合过期时间
- 更复杂场景通常用 `Lua` 脚本，把“校验 + 更新 + 设置过期”合成一个原子流程

一句话总结：

- `固定窗口` 适合简单高性能场景
- `ZSET 滑动窗口` 适合要求更严格的限流
- `令牌桶` 最适合常见接口限流
- `原子性` 是分布式限流能不能真正生效的关键

### 详细讲解

这题面试里通常会考两层：

- 你知不知道几种常见限流算法
- 你能不能把它们和 Redis 的具体实现方式结合起来

真正回答时，建议按下面这条主线展开：

- 为什么要限流
- 固定窗口怎么做
- 滑动窗口为什么比固定窗口更合理
- 漏桶和令牌桶分别适合什么场景
- 分布式限流为什么一定要强调原子性

#### 1. 为什么要做限流

限流的本质是：

- 不让系统在短时间内被超过承载能力的流量打垮

常见场景有：

- 接口防刷
- 用户维度防滥用
- 网关限流
- 秒杀入口保护
- 保护下游数据库、缓存或第三方服务

如果没有限流，系统在瞬时高并发下很容易出现：

- RT 飙高
- 线程池堆积
- 连接池打满
- 下游级联雪崩

所以限流本质上是系统保护手段，而不是单纯“拒绝请求”。

#### 2. 固定窗口限流

固定窗口是最容易理解的一种方案。

比如规则是：

- `1 秒内最多 100 次请求`

实现方式通常是：

- 用一个 key 表示这个时间窗口的计数
- 每来一个请求就 `INCR`
- 第一次创建这个 key 时设置过期时间 `EXPIRE 1s`

例如 key 可以是：

- `rate_limit:user:123:20260418120001`

这种方案的优点非常明显：

- 实现简单
- Redis 压力小
- 吞吐高

但它有一个经典问题：

- `窗口边界突刺`

举个例子：

- 限制 10 秒内最多 100 次
- 在 `9.9s` 来了 100 个请求
- 在 `10.1s` 又来了 100 个请求

从固定窗口视角看，它们分属两个不同窗口，所以都合法。  
但现实里在极短时间内放过了 200 个请求，限流就显得不够平滑。

所以固定窗口的结论是：

- 优点：简单、高性能
- 缺点：精度不高，边界突刺明显

#### 3. 精确滑动窗口

如果业务要求严格按照“最近 N 秒”的真实请求数来限流，  
更合理的做法是用：

- `ZSET + 时间戳`

这也是 Redis 里最常见的精确滑动窗口实现。

假设规则是：

- `最近 10 秒最多 100 次请求`

做法是：

1. 每个请求到来时，把当前时间戳写入 `ZSET`
2. `score` 使用请求时间戳
3. 先删除 `当前时间 - 10 秒` 之前的旧请求
4. 再统计当前 `ZSET` 里剩余元素个数
5. 如果数量超过阈值，就限流

这样在 `10.1s` 这一刻统计时，真正看的不是：

- 固定的 `0~10s`
- 或 `10~20s`

而是：

- `0.1s ~ 10.1s` 这 10 秒里的真实请求数

这就不会被固定窗口的边界问题骗过去。

它的优点是：

- 精度高
- 语义最符合“最近 N 秒”的真实定义
- 很适合对限流平滑性要求高的场景

它的缺点是：

- 每个请求都要写一条时间记录
- 高 QPS 场景下内存和操作成本更高
- 比固定窗口实现复杂

所以精确滑动窗口的关键词是：

- `语义最准确`
- `成本比固定窗口更高`

#### 4. 漏桶

漏桶的思路是：

- 请求先进桶
- 桶按固定速率往外漏水

它的特点是：

- 无论瞬时进来多少请求，输出速率尽量恒定

这对保护下游很有价值。  
例如下游系统最多只能稳定处理每秒 200 个请求，那漏桶就能把请求流量平滑成一个固定速率。

它的优点是：

- 流量输出稳定
- 对下游保护效果好

缺点是：

- 高峰期请求会排队
- 排队过长时就得丢弃
- 不太适合必须快速响应的场景

所以漏桶更强调：

- `平滑输出`

#### 5. 令牌桶

令牌桶是工程上非常常用的一种限流思路。

它的核心逻辑是：

- 系统按固定速率往桶里放令牌
- 请求来了先取令牌
- 取到令牌才允许通过

它和漏桶最大的不同是：

- 令牌桶允许桶里积累一定量令牌
- 所以能够承受一定程度的突发流量

比如：

- 平均每秒生成 100 个令牌
- 桶容量是 200

那如果前面一段时间流量较低，桶里攒了一些令牌，  
短时间内就可以放行一波突发请求。

所以令牌桶的优点是：

- 控制平均速率
- 允许一定突发流量
- 工程适配性很强

这也是为什么很多网关、接口限流、秒杀入口保护都会更偏向令牌桶。

#### 6. Redis 里怎么实现这些方案

从 Redis 视角看，常见实现对应关系大致是：

1. `固定窗口`
- `INCR + EXPIRE`

2. `精确滑动窗口`
- `ZADD + ZREMRANGEBYSCORE + ZCARD`

3. `令牌桶`
- Redis 里保存：
  - 上次发放令牌时间
  - 当前剩余令牌数
- 每次请求到来时，先根据时间差计算应该补多少令牌，再判断是否允许通过

4. `漏桶`
- 可以维护：
  - 上次漏水时间
  - 当前桶中积压量
- 请求到来时根据时间推进桶状态，再判断是否还能接收

其中：

- 固定窗口实现最简单
- 滑动窗口语义最准
- 令牌桶工程上最实用

#### 7. 分布式场景下为什么一定要保证原子性

分布式限流最怕的情况是：

- 多个请求几乎同时进来
- 每个请求都先查“当前计数还没满”
- 然后又都更新成功

结果就是：

- 实际放行数量超过阈值

所以限流的关键，不只是存储在 Redis 里，而是：

- `判断是否放行`
- `更新状态`

必须在一个原子操作里完成。

例如：

- 固定窗口里，要把“读计数 + 增加计数 + 设置过期”处理好
- 滑动窗口里，要把“删旧数据 + 加新时间戳 + 统计数量 + 判断是否通过”放在一个完整流程里
- 令牌桶里，要把“补令牌 + 扣令牌 + 更新状态”做成一个原子过程

工程上最常见的做法就是：

- `Lua 脚本`

因为 Lua 可以把多步 Redis 操作合并成一次原子执行。

#### 8. 怎么选型

这题最后最好补一句“我会怎么选”。

一般可以这样回答：

1. `只要求简单和吞吐`
- 选固定窗口

2. `要求严格按最近 N 秒限流`
- 选 `ZSET` 精确滑动窗口

3. `既要控平均速率，又允许一定突发`
- 选令牌桶

4. `要把流量平滑到稳定处理速率`
- 选漏桶

如果你想更贴近线上经验，可以补一句：

- 大多数业务接口限流，工程上更常见的是 `固定窗口` 或 `令牌桶`
- 如果业务对限流精度特别敏感，再考虑 `ZSET` 滑动窗口

#### 9. 一段可直接复述的回答

“Redis 做高并发限流，常见方案有固定窗口、滑动窗口、漏桶和令牌桶。固定窗口通常用 `INCR + EXPIRE` 实现，优点是简单高性能，但会有窗口边界突刺问题；如果要求严格按最近 N 秒统计，我会用 `ZSET + 时间戳` 做精确滑动窗口，每次请求先删除窗口外数据，再统计当前窗口内请求数，这种方式精度更高，但成本也更大；漏桶适合把请求以稳定速率输出，保护下游；令牌桶更适合常见接口限流，因为它既能控制平均速率，又允许一定突发流量。分布式场景下的关键是把‘判断是否放行’和‘更新计数或扣减令牌’做成原子操作，复杂场景一般会用 Lua 脚本实现。” 

#### 10. 一句话记忆

- `固定窗口：简单，但有边界突刺`
- `ZSET 滑动窗口：语义最准，但成本更高`
- `漏桶：强调平滑输出`
- `令牌桶：控制平均速率，同时允许一定突发`
- `Lua：保证分布式限流原子性`

## Redis 的 ZSet 为什么适合做排行榜？底层跳表是怎么工作的？

### 标准面试回答

Redis 的 `ZSet` 特别适合做排行榜，核心原因是：

- 它既保证 `member` 唯一
- 又给每个 `member` 绑定一个 `score`
- 并且支持按 `score` 排序、查排名、查范围、动态更新分数

这正好和排行榜场景的需求高度匹配。

如果和其他结构比较：

1. `为什么不用 List`
- List 只有顺序，没有 score 概念
- 某个用户分数变化后，很难高效插入到新的正确位置
- 查排名、查 topN、查分数区间都不方便

2. `为什么不用 Set`
- Set 只有唯一性，没有排序能力
- 适合去重，不适合做排行榜

3. `ZSet 的核心能力`
- member 唯一
- score 可排序
- 支持：
  - topN
  - 排名查询
  - score 区间查询
  - 动态更新分数

4. `底层为什么能又快又适合范围查询`
- Redis 的 ZSet 底层通常结合：
  - `dict`
  - `skiplist`
- `dict` 负责通过 member 快速定位 score
- `skiplist` 负责按 score 有序组织数据，方便查排名和范围查询

5. `超大排行榜线上注意点`
- 内存占用
- 高频更新带来的写放大
- 是否只保留 TopN
- 是否按日榜/周榜/月榜拆分
- 是否做冷热分离，避免超大 ZSet 变成大 key

一句话总结：

`ZSet 适合排行榜，是因为它同时具备唯一性、分值排序、排名查询和范围查询能力，而跳表正好擅长这种“有序 + 动态更新 + 区间扫描”的场景。`

### 详细讲解

这题其实分两层：

- 第一层：为什么排行榜要选 ZSet
- 第二层：为什么 Redis 在底层用跳表来支撑这些能力

#### 1. 为什么排行榜天然需要“唯一 + 排序”

排行榜场景通常有这些需求：

- 每个人只能有一条记录
- 每个人都有一个分数
- 分数会动态变化
- 要查前 N 名
- 要查某个人当前排名
- 要查某个分数区间的人

所以这个场景天然要求：

- `唯一性`
- `有序性`
- `支持动态更新`
- `支持范围查询`

这也是为什么：

- List 不合适
- Set 也不合适
- ZSet 刚好最匹配

#### 2. 为什么 List 不适合排行榜

List 的特点是：

- 有顺序
- 支持两端插入和弹出

但它的问题在于：

- 没有 score 语义
- 不适合按分数重新调整位置
- 查某个成员当前排名效率差
- 查 score 区间几乎不自然

所以 List 更适合：

- 队列
- 栈
- 简单消息流

而不是动态排行榜。

#### 3. 为什么 Set 不适合排行榜

Set 的特点是：

- 元素唯一
- 无序
- 支持交并差

这很适合：

- 去重
- 标签集合
- 共同好友 / 共同关注

但排行榜最关键的是“谁在前、谁在后”，  
Set 做不到这一点，所以不适合。

#### 4. ZSet 的核心能力

ZSet 可以理解成：

- 一个唯一集合
- 里面每个元素都有一个 score

所以它天然支持：

- `ZADD`：新增或更新分数
- `ZREVRANGE`：查 topN
- `ZRANK / ZREVRANK`：查某个成员排名
- `ZRANGEBYSCORE`：查某个分数区间

这些操作和排行榜需求几乎是一一对应的。

#### 5. 为什么底层不是只用一个结构

如果只考虑“按分数排序”，那么单独一个有序结构就够了。  
但排行榜不只需要排序，还需要：

- 根据 `member` 快速找到它当前的分数
- 更新这个 member 的分数后快速调整位置

所以 Redis 没有只用一个结构，而是通常结合：

- `dict`
- `skiplist`

这是一种典型的“用两套结构分别优化不同访问路径”的设计。

#### 6. dict 在 ZSet 里负责什么

`dict` 的作用是：

- 通过 member 快速找到它的 score

比如：

- 你要更新用户 `u123` 的积分
- 如果没有 dict，你得先在有序结构里全局查这个用户
- 成本会很高

有了 dict 后，可以直接：

- `member -> score`

这让按 member 定位和更新变得很快。

#### 7. skiplist 在 ZSet 里负责什么

`skiplist` 的作用是：

- 按 score 有序组织所有成员
- 支持范围查询
- 支持排名查询
- 支持动态插入删除

这正好服务排行榜最常见的几个需求：

- topN
- 某个成员排名
- 分数段区间查询

所以你可以理解成：

- `dict` 解决“找到这个人”
- `skiplist` 解决“这个人应该排哪、前 N 名是谁、这个分数范围里有哪些人”

#### 8. 什么是跳表

跳表本质上是：

- `多层有序链表`

最底层存放完整有序数据，  
上层则是下层的稀疏索引。

你可以把它理解成：

- 最底层是一条完整有序链表
- 上面抽一部分节点出来，做第 2 层索引
- 再从第 2 层抽一部分节点出来，做第 3 层索引

查找时：

- 先从高层快速跳
- 快接近目标时再往下层细找

所以它叫“跳表”，因为不是一个一个往后挪，而是可以“跳着走”。

#### 9. 跳表长什么样

你可以把跳表想成这样：

```text
Level 3:  1 ----------------------> 17
Level 2:  1 --------> 9 ---------> 17
Level 1:  1 --> 5 --> 9 --> 13 --> 17
Level 0:  1 -> 3 -> 5 -> 7 -> 9 -> 11 -> 13 -> 15 -> 17
```

特点是：

- `Level 0` 最底层包含所有节点
- 越往上节点越少
- 上层是下层的快捷索引

查找时就是：

- 先走最高层
- 发现再往右会超过目标，就下降一层
- 继续向右
- 一直走到最底层找到目标

#### 10. 为什么跳表查找是 O(log n)

跳表的核心优势在于：

- 每往上一层，节点数量会明显减少

如果上一层节点数大约是下一层的一半，那么层数大约就是：

- `log n`

查找时：

- 每层只走有限几步
- 总层数又不高

所以平均查找复杂度是：

- `O(log n)`

这里强调的是：

- `平均 O(log n)`

不是绝对最坏严格保证，但在工程上已经足够高效。

#### 11. 跳表为什么插入和删除也快

插入和删除的本质是：

1. 先按查找过程定位位置
2. 再局部修改多层链表指针

所以它们的平均复杂度也可以做到：

- `O(log n)`

这对排行榜非常重要，因为排行榜不是只读结构，而是要频繁更新分数、调整位置。

#### 12. 跳表最关键的设计：随机层高

跳表并不是手工每隔几个节点就拉一层，而是：

- 插入新节点时，通过随机算法决定这个节点有几层

常见思想是：

- 每一层以一定概率继续向上提升

于是会形成这样的分布：

- 大多数节点只在底层
- 少数节点会出现在更高层
- 越高层节点越稀疏

这让跳表天然形成“稀疏索引”。

所以跳表的核心不是复杂旋转，而是：

- `用随机化维持近似平衡`

#### 13. 为什么跳表适合范围查询

这正是 Redis 选它的一个重要原因。

如果你已经通过高层索引定位到了某个起点，比如：

- 第一个 `score >= 1000` 的节点

那接下来只要沿底层链表一路往后扫，直到：

- `score > 2000`

就能完成区间查询。

所以跳表非常适合这种模式：

- `先快速定位`
- `再顺序遍历一段连续区间`

而排行榜、排名区间、本周积分榜这类场景恰好大量需要这种能力。

#### 14. Redis 里 ZSet 的排序规则

在 ZSet 里，排序通常先按：

- `score`

如果 score 相同，再按：

- `member` 的字典序

这样做的好处是：

- 排序稳定
- 同分时顺序可确定

这也是为什么排行榜里即使分数相同，Redis 仍然能给出确定顺序。

#### 15. 跳表和平衡树相比有什么特点

面试官有时会追问：

- 为什么不用红黑树？

你可以这样回答：

跳表的优点：

- 实现更简单
- 插入删除不需要像红黑树那样频繁旋转
- 范围查询很自然
- 工程可读性更好

跳表的缺点：

- 指针较多，内存开销不小
- 依赖随机性
- 理论最坏情况不如严格平衡树稳定

所以它不是绝对碾压平衡树，而是一个工程折中：

- `用概率换实现简洁`

#### 16. 超大排行榜线上要注意什么

如果排行榜规模很大，线上要重点关注：

1. `内存占用`
- 全量排行榜放在 Redis 里会很吃内存

2. `更新频率`
- 分数更新过于频繁会造成写压力

3. `只保留 TopN`
- 很多业务并不需要全量榜单，只保留前几千、前几万即可

4. `按时间维度拆榜`
- 日榜、周榜、月榜拆开，避免一个无限增长的大榜

5. `冷热分离`
- 热门榜单放 Redis
- 长尾或历史数据放 DB / ES / 离线系统

6. `大 key 风险`
- 超大 ZSet 本身也可能成为大 key，影响迁移、持久化、删除和故障恢复

#### 17. 一段可直接复述的回答

“Redis 的 ZSet 适合做排行榜，是因为它同时具备唯一性、分值排序、排名查询和范围查询能力。List 只有顺序，没有按分数动态排序能力；Set 只有唯一性，没有排序能力；而 ZSet 给每个 member 绑定 score，并支持按 score 排序、查 topN、查排名、查分数区间。底层上，ZSet 通常结合 dict 和 skiplist，dict 负责按 member 快速定位，skiplist 负责按 score 有序组织。跳表本质上是多层有序链表，通过上层稀疏索引让查找、插入、删除平均复杂度达到 `O(log n)`，而且范围查询非常自然，所以特别适合排行榜、排名查询和分数区间查询场景。线上如果排行榜规模很大，还要关注内存占用、更新频率、大 key 风险，以及是否只保留 TopN、按时间维度拆榜和做冷热分离。” 

#### 18. 一句话记忆

- `ZSet = 唯一集合 + score 排序`
- `dict 负责按 member 找人`
- `skiplist 负责按 score 排序和范围查询`
- `跳表 = 多层有序链表 + 稀疏索引`
- `适合排行榜、排名、区间查询`
