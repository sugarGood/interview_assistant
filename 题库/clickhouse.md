# ClickHouse
## ClickHouse 为什么适合做 OLAP，不适合高并发 OLTP？
### 标准面试回答
ClickHouse 是列式存储 + 向量化执行 + 大规模并行扫描引擎，天然适合聚合分析（OLAP）。  
但它不擅长高频小事务更新、行级锁竞争和强事务场景，因此不适合典型 OLTP。

### 详细讲解
OLAP 关心的是“批量读 + 聚合速度”，ClickHouse 在这方面优势明显：只读取查询涉及的列，CPU SIMD 计算效率高，压缩率高降低 IO。  
而 OLTP 关心“频繁点查/更新 + 事务一致性”，ClickHouse 的数据写入和合并机制更偏批处理，不适合频繁行级更新。

## MergeTree 系列引擎的核心机制是什么？
### 标准面试回答
MergeTree 的核心是“分区 + 主键排序 + 后台合并（merge）”。  
数据先写入 part，随后后台线程做有序合并和压缩，从而兼顾写入吞吐与查询性能。

### 详细讲解
写入不是直接覆盖旧数据，而是追加新 part，后台再合并。  
主键在 ClickHouse 中主要用于“数据跳过和范围过滤”，不是 MySQL 那种唯一约束主键。  
调优重点通常是分区键和排序键设计，不合理会导致查询扫描量暴增。

## ClickHouse 的分区键和排序键该怎么设计？
### 标准面试回答
分区键决定数据生命周期管理和粗粒度裁剪，排序键决定查询过滤效率。  
一般建议：分区键按时间（如天/月），排序键按高频过滤维度 + 时间组合。

### 详细讲解
分区不要过细，否则会产生大量小分区和元数据开销。  
排序键要贴合最常见 where 条件，才能让稀疏索引有效跳过数据块。  
面试可补一句：如果查询条件经常变，需结合物化视图或二级索引能力做补偿。

## ClickHouse 去重与“更新”通常怎么做？
### 标准面试回答
ClickHouse 不擅长传统行级 update/delete，常用替代方案有：
1. ReplacingMergeTree 做最终去重。
2. Collapsing/VersionedCollapsing 处理版本或撤销语义。
3. 业务侧改为“追加写 + 查询时取最新版本”。

### 详细讲解
这些方案大多是“最终一致”而不是“立即可见一致”。  
因此业务需要接受短时间重复或旧值可见，并通过查询条件（如 `FINAL`）或离线归并兜底。  
面试中要强调：是否能接受最终一致，是 ClickHouse 选型边界之一。

## ClickHouse 查询慢，常见排查思路是什么？
### 标准面试回答
排查顺序通常是：
1. 看查询日志与 ProfileEvents，确认瓶颈在扫描、聚合还是网络。
2. 看分区/排序键是否命中，是否出现全表扫描。
3. 看并发和资源（CPU、内存、磁盘 IO）是否饱和。
4. 看是否需要预聚合（物化视图）或数据模型调整。

### 详细讲解
很多慢查询不是 SQL 写法问题，而是模型与查询模式不匹配。  
例如查询维度和排序键无关时，跳数索引几乎无效。  
优化手段通常包括：调整 ORDER BY、增加物化视图、分层明细与汇总表、控制单次查询返回量。
