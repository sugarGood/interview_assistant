## MyBatis 中 `#{}` 和 `${}` 有什么区别？分别适用于什么场景？
`#{}` 是预编译参数占位符，MyBatis 会先把它转换成 JDBC 的 `?`，再通过 `PreparedStatement` 完成参数绑定，所以普通查询条件基本都应该优先使用 `#{}`。它不会直接改变 SQL 结构，能够有效防止 SQL 注入，同时还会根据参数类型完成对应的类型处理。

`${}` 是字符串直接替换，在 SQL 执行前就会把变量值原样拼接到 SQL 中，不会走预编译参数绑定。因此它能够参与 SQL 结构的拼接，但也会带来明显的 SQL 注入风险。

两者的典型使用场景不同：

+ `#{}` 适合普通参数传递，例如 `id`、`name`、`status` 这类查询条件
+ `${}` 适合不能使用 `?` 占位的位置，例如动态表名、动态列名、`order by` 后面的排序字段

如果要做动态排序字段，通常会使用 `${column}`，因为 `order by #{column}` 会把字段名当成普通参数值处理，最终语义不对，无法实现真正按列排序。但生产环境里不能直接信任前端传入的字段，必须先在服务端做白名单校验，比如只允许 `id`、`create_time`、`update_time` 这些合法字段，再将校验后的结果传给 `${}`。

面试里可以这样总结：`#{}` 的本质是预编译占位符，安全、通用，适合绝大多数参数场景；`${}` 的本质是字符串拼接，适合动态 SQL 结构位置，但必须严格配合白名单校验使用，否则很容易产生 SQL 注入问题。

## MyBatis 的一级缓存和二级缓存分别是什么？为什么很多公司会谨慎使用二级缓存？
MyBatis 有两级缓存。一级缓存是 `SqlSession` 级别的本地缓存，默认开启。在同一个 `SqlSession` 中，如果执行相同 SQL、参数也一致，MyBatis 可以直接复用查询结果，减少数据库访问。它的生命周期跟当前会话绑定，`SqlSession` 关闭后缓存就消失。

一级缓存通常会在以下场景失效或被清空：执行了 `insert`、`update`、`delete`，手动调用 `clearCache()`，事务执行 `commit` 或 `rollback`，以及 `SqlSession` 关闭。

二级缓存是 `Mapper`，也就是 `namespace` 级别的缓存，多个 `SqlSession` 可以共享。它不是默认直接可用，一般需要全局允许二级缓存，并在具体 mapper 上配置 `<cache/>` 才会生效。查询语句需要允许使用缓存，结果对象通常也需要满足序列化等要求。

二级缓存的优势是能跨会话复用查询结果，但它的一致性风险也更高。通常在对应 `namespace` 发生增删改时会清空该命名空间下的缓存；但如果业务存在跨表关联、跨 mapper 更新，或者多实例分布式部署，二级缓存很容易出现脏数据、旧数据无法及时失效的问题。

很多公司线上会谨慎使用甚至关闭二级缓存，原因主要有三点：一是二级缓存是 `namespace` 维度，不是全局强一致缓存，复杂业务下失效边界难以控制；二是分布式场景下多节点缓存同步复杂；三是很多系统已经统一使用 Redis 或业务层缓存，再叠加 MyBatis 二级缓存会增加排查成本和一致性风险。

面试里可以这样总结：一级缓存是 `SqlSession` 级别，默认开启，适合一次会话内复用查询结果；二级缓存是 `namespace` 级别，需要额外配置，多个会话可共享，但因为一致性问题在生产环境通常会谨慎使用。

## MyBatis 为什么只写一个 Mapper 接口就能执行 SQL？核心执行流程是什么？
MyBatis 之所以只写一个 `Mapper` 接口就能执行 SQL，本质上是因为框架会在运行时为该接口生成动态代理对象，而不是要求开发者手写实现类。开发者调用的其实不是接口本身，而是 MyBatis 创建出来的代理对象。

项目启动时，MyBatis 会解析配置文件和 `Mapper.xml`，把每条 SQL 封装为一个 `MappedStatement`。`MappedStatement` 里保存了 SQL 内容、参数映射、结果映射等核心元信息。之后通过 `SqlSession.getMapper()` 获取 `Mapper` 时，拿到的就是代理对象。

当调用某个 `Mapper` 方法时，代理逻辑会根据“接口全限定名 + 方法名”定位到对应的 `MappedStatement`，然后把这次调用交给 `SqlSession`。`SqlSession` 是面向开发者的核心入口，负责提供查询、更新、事务管理和获取 Mapper 等能力；而真正负责执行 SQL、维护缓存并调用底层 JDBC 的核心组件是 `Executor`。

一次典型调用流程可以概括为：`Mapper` 方法调用进入动态代理，代理定位对应的 `MappedStatement`，再由 `SqlSession` 发起执行，`Executor` 负责组织 SQL 执行流程，完成参数绑定、语句执行、结果集处理，最后将结果映射成 Java 对象返回。

面试里可以这样总结：`Mapper` 接口没有手写实现类，是因为 MyBatis 运行时通过动态代理接管了方法调用；方法调用后会根据接口名和方法名找到对应的 `MappedStatement`，再由 `SqlSession` 和 `Executor` 完成 SQL 执行与结果映射。

## MyBatis 插件机制是怎么实现的？分页插件为什么通常拦截 `StatementHandler`？
### 详细讲解整理
MyBatis 插件机制的本质是 `Interceptor` 加动态代理。开发者实现 `Interceptor` 接口后，通过 `@Intercepts` 和 `@Signature` 声明要拦截的目标类型和方法。MyBatis 在创建核心执行对象时，会判断这些对象是否命中拦截声明；如果命中，就使用代理对象进行包装。后续方法执行时，会先进入插件的 `intercept()` 方法，再通过 `invocation.proceed()` 放行到下一个插件或原始对象，因此多个插件叠加时本质上就是一条拦截器链。

MyBatis 官方允许拦截的核心对象只有 4 类：`Executor`、`StatementHandler`、`ParameterHandler`、`ResultSetHandler`。它不是任意方法级别的通用切面，而是围绕这几个底层执行对象做增强。

`StatementHandler` 一般最适合做分页插件，因为这一层最接近最终要执行的 SQL，便于拿到 `BoundSql` 并改写 SQL，例如追加 `limit`、`offset`，或者生成 `count` SQL 做总数查询，也更方便适配不同数据库方言。

4 个拦截点的常见使用场景可以这样区分：

+ `Executor`：适合从更高层统一控制一次查询或更新的执行流程，比如统计执行次数、做统一审计、控制读写路由、做通用执行开关或轻量缓存增强
+ `StatementHandler`：适合直接处理最终 SQL，比如分页、租户条件注入、数据权限 SQL 改写、慢 SQL 监控
+ `ParameterHandler`：适合在参数绑定到 `PreparedStatement` 之前做增强，比如统一补字段、参数脱敏、参数加密、自定义参数修正或特殊类型处理
+ `ResultSetHandler`：适合在结果集映射为 Java 对象时做增强，比如结果脱敏、字典值翻译、返回结果格式转换、补充派生字段

如果面试官问“为什么分页通常不拦 `Executor`、`ParameterHandler`、`ResultSetHandler`”，可以这样回答：`Executor` 更偏整体流程控制，离具体 SQL 改写稍远；`ParameterHandler` 负责参数设置，不适合改 SQL 结构；`ResultSetHandler` 已经处于结果处理阶段，SQL 执行已经完成，因此分页这类“改 SQL + 控制执行”的需求通常优先放在 `StatementHandler`。

面试里可以这样总结：MyBatis 插件机制本质上是拦截器链和动态代理，只能拦截 `Executor`、`StatementHandler`、`ParameterHandler`、`ResultSetHandler` 这 4 类核心对象；而分页插件通常选 `StatementHandler`，因为这一层最接近最终执行 SQL，最适合做分页语句改写和方言适配。

## MyBatis 中参数绑定和结果映射分别由谁负责？`TypeHandler` 是干什么的？什么时候会自定义它？
在 MyBatis 中，参数绑定和结果映射是两条不同的处理链路。参数绑定主要由 `ParameterHandler` 负责，它会把 SQL 中的 `?` 占位符和 Java 参数对象对应起来，并在执行前把参数设置到 `PreparedStatement`。结果映射主要由 `ResultSetHandler` 负责，它会读取查询返回的 `ResultSet`，再把列值封装成目标 Java 对象。

`TypeHandler` 的职责是完成 Java 类型和 JDBC 类型之间的双向转换。写入数据库时，它负责把 Java 参数按正确方式设置到 `PreparedStatement`；读取查询结果时，它负责把 `ResultSet` 中的列值转换成对应的 Java 类型。因此 `TypeHandler` 不只参与结果读取，也参与参数写入。

MyBatis 已经内置了大量常见类型的默认 `TypeHandler`，例如 `String`、`Integer`、`Long`、`Date` 等。但当业务里出现默认映射不够用的类型时，就适合自定义 `TypeHandler`，把转换逻辑统一封装起来。

常见需要自定义 `TypeHandler` 的场景包括：

+ 数据库 `JSON` 字段映射为 Java 对象或集合
+ 枚举类型和数据库编码值互转
+ 加密字段在读写数据库时自动加解密
+ 特殊日期时间类型转换
+ 数据库存储逗号分隔字符串，而 Java 中希望映射为 `List`

面试里可以这样总结：参数绑定主要由 `ParameterHandler` 负责，结果映射主要由 `ResultSetHandler` 负责，而 `TypeHandler` 负责 Java 类型和 JDBC 类型之间的转换，既参与参数设置，也参与结果读取；像 JSON、枚举、加密字段、特殊时间类型等场景，通常会考虑自定义 `TypeHandler`。

## `RowBounds` 是怎么用的？它既然提供了 `offset` 和 `limit`，为什么很多时候仍然会查出更多数据？
### 详细讲解整理
`RowBounds` 的用法很直接，通常是在调用 MyBatis 查询方法时额外传入一个分页对象，例如 `new RowBounds(offset, limit)`。它的语义是告诉 MyBatis：从第几条开始，最多返回多少条结果。

但关键点在于，`RowBounds` 默认只是 MyBatis 层的分页参数，并不等于数据库一定会执行物理分页。也就是说，传了 `offset` 和 `limit`，不代表最终 SQL 一定会自动改写成 `limit ?, ?` 这样的数据库分页语句。

很多默认场景下，数据库执行的仍然是原始 SQL，而 MyBatis 会在结果集处理阶段根据 `RowBounds` 的 `offset` 和 `limit` 做跳过和截取。也可以理解为：数据库和 JDBC 层可能已经处理了比目标页更多的数据，MyBatis 只是从返回结果里再按分页参数裁剪出需要的那一段。

这就是为什么说 `RowBounds` 更接近逻辑分页，而不是可靠的物理分页。它的问题不一定是“一次性把所有数据都塞进 JVM 内存”，而是数据库侧通常没有真正只返回目标页，导致数据库扫描、网络传输、驱动缓冲和应用层结果处理都可能比实际需要更多。

例如你传入 `new RowBounds(100, 10)`，很多人以为数据库执行的是 `select ... limit 100,10`，但默认情况下更可能仍然执行原始查询，再由 MyBatis 在结果处理时跳过前 100 条、截取后 10 条。因此如果数据量很大，这种方式性能通常不如真正的物理分页。

生产环境里更常见的做法是：手写 `limit/offset`，或者使用分页插件在 `StatementHandler` 等层面改写 SQL，把分页逻辑真正下推到数据库执行。这样数据库只返回目标页数据，性能和可控性通常更好。

面试里可以这样总结：`RowBounds` 虽然提供了 `offset` 和 `limit`，但默认只是 MyBatis 层的分页参数，不保证自动改写成数据库物理分页 SQL；很多情况下它更像在结果集处理阶段做截取，因此数据库和网络层面仍可能处理比目标页更多的数据，所以线上通常更推荐手写物理分页或使用分页插件。

## MyBatis 的延迟加载是什么？为什么很多团队会谨慎使用？请举一个具体场景
### 详细讲解整理
MyBatis 的延迟加载，指的是关联对象或关联集合不是在主查询时一次性查出，而是在第一次真正访问属性时再触发查询。它常用于一对一或一对多关联映射场景，例如查询订单时，先查订单主表信息，等真正调用 `order.getUser()` 或 `order.getItems()` 时，再查询对应的用户信息或订单明细。

一个很典型的场景是“订单列表页”。页面通常只展示订单号、金额、状态、下单时间等基础字段，但 `Order` 对象里可能还关联了 `user` 和 `items`。如果使用立即加载，那么查订单列表时，用户信息和订单明细也可能一起查出来；但这些数据在列表页里往往根本用不到。此时开启延迟加载，就可以先只查订单主数据，等真正访问 `order.getUser()` 或 `order.getItems()` 时再查关联数据，从而减少首轮查询开销。

它的好处是按需加载，避免把暂时用不到的关联数据一开始就全部查出来，对“主数据常用、子数据偶尔用”的场景比较友好。

但很多团队会谨慎使用，核心原因是它容易引发 `N+1` 查询问题。比如先查出 100 条订单后，在循环里依次访问每个订单的 `getUser()`，就可能出现“1 条订单主查询 + 100 条用户子查询”。此外，延迟加载让 SQL 的触发时机变得不直观，一个普通的 getter 调用背后可能就会访问数据库，排查慢 SQL、性能抖动和会话边界问题都会更麻烦。

面试里可以这样总结：延迟加载就是关联属性在真正访问时才触发查询，适合主数据常用、关联数据偶尔用的场景；但它容易带来 `N+1` 查询，而且 SQL 触发时机不直观，所以很多团队会谨慎使用。

## 如果线上一个 MyBatis 查询突然变慢了，一般应该怎么系统排查？
如果线上一个 MyBatis 查询突然变慢，我不会一上来就判断是索引问题，而是先分层定位：到底是某条 SQL 真慢，还是整个请求里数据库访问次数变多；到底是数据库执行变慢，还是 MyBatis、插件、结果映射等框架层开销变大。

第一步要先结合接口监控、链路日志、慢 SQL 日志确认慢点位置，判断是单条 SQL 变慢，还是因为循环查库、延迟加载、`N+1`、多次子查询把整体耗时拉高。

第二步是拿到最终执行 SQL 和真实参数，而不是只看 mapper XML 模板。因为 MyBatis 常常会通过动态 SQL、插件、分页组件对原始语句做改写，必须确认最终 SQL、真实绑定参数、排序字段、模糊查询条件、`in` 列表长度等信息。

第三步是把最终 SQL 带参数去数据库侧执行 `explain`，重点看是否命中预期索引、是否发生全表扫描、联表顺序是否合理、排序和分组是否代价过高、是否存在深分页和大量回表问题。如果是近期突然变慢，还要考虑数据量增长、统计信息变化、执行计划漂移等因素。

第四步是排查分页和 SQL 写法本身，例如是否用了大 `offset` 深分页、是否错误使用 `RowBounds`、是否联表过多、是否 `select *`、是否对索引列做函数处理、是否存在不合理子查询或 `or` 条件导致索引失效。

第五步是排查 MyBatis 框架层开销，例如分页插件、租户插件、数据权限插件是否把原始 SQL 改写得更复杂；动态 SQL 是否拼出了冗余条件；延迟加载和关联映射是否导致 `N+1`；复杂 `resultMap`、嵌套对象和集合映射是否增加了结果集处理耗时。

最后还要结合缓存和连接层一起看，例如一级缓存、二级缓存、Redis 缓存命中率是否变化，数据库连接池是否耗尽，数据库实例本身的 CPU、IO、锁等待是否异常。

面试里可以这样总结：MyBatis 查询变慢时，我会按“定位慢点 -> 拿最终 SQL 和参数 -> explain 看执行计划和索引 -> 排查分页/联表/动态 SQL -> 排查插件、延迟加载、`N+1`、结果映射 -> 再看缓存、连接池和数据库负载”的顺序系统排查，而不是一上来只说加索引。

## 如果要设计一套“安全、可维护、性能可控”的 MyBatis 查询规范，应该怎么定？
如果让我设计 MyBatis 查询规范，我会先明确目标是：`安全优先、默认可维护、性能可控、问题可排查`。规范不只是代码风格约束，而是为了避免 SQL 注入、深分页、动态 SQL 失控、插件乱改写、慢 SQL 难定位等线上问题。

在参数使用上，普通查询条件一律优先 `#{}`，禁止在 where 条件里直接使用 `${}`。`${}` 只允许用于动态表名、动态列名、`order by` 字段等确实不能使用占位符的位置，而且必须配合服务端白名单映射，不能直接信任前端入参。

在分页上，默认只允许物理分页，优先手写 `limit/offset` 或使用统一分页插件；禁止在大数据量场景直接依赖 `RowBounds`；对深分页要有页码上限、游标分页或主键范围翻页等约束。

在动态 SQL 上，只允许用 `if`、`where`、`set`、`foreach` 解决条件拼接问题，不鼓励把 XML 写成过度复杂的逻辑脚本；`in` 列表长度要限制，复杂查询优先拆分，避免动态 SQL 拼出冗余条件和难维护语句。

在结果映射上，简单单表查询优先使用 `resultType`，复杂联表、嵌套对象、一对多关系再使用 `resultMap`；`association`、`collection` 要谨慎，避免不必要的嵌套映射和延迟加载引发 `N+1`。

在插件治理上，分页、租户、数据权限等插件必须统一选型，禁止多个插件重复改写同一类 SQL；插件要可开关、可追踪、可审计，避免“看起来简单的 SQL 在运行时被静默改复杂”。

在缓存上，一级缓存保留默认行为即可，但不把它当业务缓存设计；二级缓存默认谨慎使用，复杂更新或分布式场景通常关闭，真正的业务缓存统一交给 Redis 或业务层处理。

在日志和排障上，开发测试环境要能看到最终执行 SQL 和参数；生产环境不能无脑打印全量 SQL，但必须具备慢 SQL 日志、链路追踪、采样输出和 `explain` 排查能力；同时禁止 `select *`，限制返回字段、返回行数和大查询直接上线。

面试里可以这样总结：我会围绕“安全、可维护、性能、可观测”四个目标来制定 MyBatis 规范，例如普通参数默认只用 `#{}`、`${}` 必须白名单控制；分页默认走物理分页并限制深分页；动态 SQL 只做条件拼接；简单查询优先 `resultType`，复杂关联再用 `resultMap`；插件统一选型并可追踪；二级缓存默认谨慎使用；生产环境保留慢 SQL 和最终 SQL 的排查能力。这样规范既能控风险，也方便后期性能治理和线上排障。

## 如果对比 `JPA/Hibernate` 和 `MyBatis`，应该怎么选？
如果让我对比 `JPA/Hibernate` 和 `MyBatis`，我会先强调这不是简单的“一个自动、一个手写”区别，而是两种不同的开发模式。`JPA/Hibernate` 更偏对象关系映射和领域建模，强调抽象和开发效率；`MyBatis` 更偏 SQL 显式控制，强调复杂查询落地能力和性能可控性。

从 SQL 可控性看，我会更偏向 `MyBatis`。因为 SQL 是自己写的，最终执行什么语句、怎么联表、怎么分页、怎么走索引都更清晰，也更方便做性能调优和问题排查。

从开发效率看，简单 CRUD、标准单表操作、基础实体管理上，`JPA/Hibernate` 通常更高效，因为它对实体状态管理、级联、仓储抽象支持更完整，能减少重复 SQL 和样板代码。

从复杂查询场景看，我通常会更倾向 `MyBatis`。像复杂联表、动态筛选、报表、分页排序、数据库方言优化这类需求，本质上就是和 SQL 打交道，用 `MyBatis` 通常更直接；如果用 `JPA/Hibernate` 强行承接过于复杂的查询，JPQL、Criteria 或自动生成 SQL 往往会增加理解和维护成本。

从性能调优角度看，`MyBatis` 往往更适合性能敏感系统，因为最终 SQL 完全可见、可控，调优路径更短。`JPA/Hibernate` 并不是不能调优，但由于多了一层 ORM 抽象，排查时往往还要先搞清楚框架生成了什么 SQL、是否出现延迟加载和 `N+1` 等问题。

从团队协作看，如果团队更偏领域建模、查询复杂度不高、希望统一仓储风格，`JPA/Hibernate` 会更规范一些；如果团队 SQL 能力较强，系统里复杂查询多、性能要求高、报表多，则 `MyBatis` 更适合。很多项目里也会分场景组合使用，而不是绝对二选一。

面试里可以这样总结：如果是后台管理、领域对象稳定、查询相对标准，我会考虑 `JPA/Hibernate` 提升开发效率；如果是查询复杂、性能要求高、需要强 SQL 控制的互联网业务，我通常更倾向 `MyBatis`。也就是说，前者更强调抽象和效率，后者更强调控制和落地，选型要看业务复杂度和团队能力。

## MyBatis-Plus 为什么只让 `Mapper` 继承 `BaseMapper<T>` 就能拥有大量单表 CRUD？为什么复杂查询还是常回到 XML 或手写 SQL？
MyBatis-Plus 的定位是 MyBatis 的增强工具，而不是替代 MyBatis 的另一套框架。它底层执行 SQL、参数绑定和结果映射的能力仍然来自原生 MyBatis，只是在此基础上补充了通用 CRUD、条件构造器、分页、逻辑删除、乐观锁等增强能力。

当 `Mapper` 继承 `BaseMapper<T>` 后，MyBatis-Plus 会根据实体类的表信息，例如表名、主键、字段映射、主键策略等，自动注入一批通用 CRUD 方法对应的 SQL 映射，例如 `insert`、`deleteById`、`updateById`、`selectById`、`selectList`、`selectPage` 等，因此开发者不需要再为单表操作重复编写 XML。

它主要解决的是重复单表 CRUD、条件构造器、分页和常见通用能力的样板代码问题，也就是常说的“覆盖 80% 的标准持久层重复劳动”。

但一旦进入复杂联表、报表查询、动态筛选很多、聚合分析、特殊排序、数据库方言优化、性能精细调优等场景，本质上还是在和 SQL 细节打交道。这时自定义 XML 或手写 SQL 更直接、更清晰，也更方便精确控制最终执行语句，所以很多项目的常见模式是：简单 CRUD 用 MyBatis-Plus，复杂查询和核心链路用 XML 或手写 SQL。

## `LambdaQueryWrapper` 和 `QueryWrapper` 有什么区别？为什么很多团队更推荐前者？
`QueryWrapper` 和 `LambdaQueryWrapper` 本质上都是 MyBatis-Plus 的条件构造器，能力上相近，主要区别在于字段表达方式不同。

`QueryWrapper` 直接写字符串字段名，例如 `eq("name", "Tom")`；`LambdaQueryWrapper` 使用方法引用，例如 `eq(User::getName, "Tom")`。

很多团队更推荐 `LambdaQueryWrapper`，核心原因不是性能，而是安全性和可维护性。因为它通过方法引用绑定实体属性，字段改名时 IDE 能辅助重构，能减少字符串字段名写错、字段重构遗漏修改等问题。

它主要解决的是“字符串字段名不安全、不好维护、重构不友好”的问题。代价是底层需要把方法引用解析成字段名，存在一定解析成本；另外在复杂联表、聚合字段、动态排序字段等场景下，也不一定能完全只靠 Lambda 表达，往往还需要结合字符串或白名单映射处理。
