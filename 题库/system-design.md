# 系统设计

## 设计一个高并发下单系统，你会怎么做分层和拆模块？

### 标准面试回答

高并发下单系统一般拆为三层：

1. 接入层：鉴权、限流、防刷、幂等校验。
2. 交易编排层：下单、库存、优惠、支付状态流转。
3. 数据与异步层：订单库、库存库、缓存、MQ、搜索索引。

设计原则是"核心链路短、异步边界清晰、一致性责任明确"。

### 详细讲解

下单链路里要区分"必须同步成功"的步骤和"可异步补齐"的步骤。
例如订单主记录、库存扣减通常在核心链路；通知、积分、推荐画像更新可以异步。

这类问题面试加分点在于你能说明：
- 为什么这样拆层。
- 每层的失败如何处理。
- 是否有降级路径。

## 秒杀系统如何同时保证高可用和不超卖？

### 标准面试回答

秒杀治理要同时做"流量控制 + 库存一致性"：

1. 前置限流：网关限流、令牌桶、验证码防刷。
2. 库存控制：Redis 预扣 + DB 条件更新兜底。
3. 异步削峰：下单请求入 MQ，消费者幂等落库。

防超卖关键是 DB 层必须有原子条件（如 `stock > 0`）与幂等保护。

### 详细讲解

秒杀失败通常不是算法问题，而是容量与一致性边界没划清。
Redis 预扣解决的是高并发性能，DB 条件更新解决的是最终正确性，两者缺一不可。

此外要准备"库存回补"机制：下单超时未支付、消息消费失败，都要能把预扣库存安全归还。

## 一个接口 RT 突增，你的系统化排查路径是什么？

### 标准面试回答

我会按"现象 -> 定位 -> 止血 -> 复盘"四步排查：

1. 先看指标：RT、QPS、错误率、线程池、GC、连接池。
2. 再分层定位：应用层、数据库层、下游 RPC 层。
3. 先止血：限流、降级、熔断、扩容或回滚。
4. 最后固化：根因分析、监控补齐、容量调整。

### 详细讲解

排查顺序必须可执行，避免"想到哪查哪"。
常见建议是先判断是"整体变慢"还是"局部接口变慢"，再判定是资源瓶颈还是依赖抖动。

对外沟通也很关键：要同步影响范围、临时措施、预计恢复时间，避免业务侧信息真空。

#### 3-5 年高级开发可复述模板

先止血，再定位，再修复，最后复盘。

1. 止血（0-10 分钟）：降级非核心功能、限流、熔断慢依赖，必要时回滚，优先保核心交易链路可用。
2. 定位（10-30 分钟）：按网关 -> 应用 -> 缓存 -> DB -> 下游分层排查，先看错误类型、RT 分位数、线程池/连接池、依赖健康度。
3. 修复：给出短期修复（配置与策略）和长期治理（容量与架构）两层方案。
4. 复盘：沉淀时间线、根因证据链、告警阈值和预案动作。

#### 为什么要降级非核心，怎么降级

降级本质是用"功能完整性"换"主流程可用性"。故障时非核心功能也会占用 CPU、线程、连接和下游 QPS，不让渡资源会拖垮核心链路。

- 影响：推荐不准、页面信息减少、非关键写延迟，但下单/支付/核心查询保持可用。
- 做法：先分级 `P0 核心`、`P1 重要`、`P2 非核心`，优先关闭或弱化 P2。
- 手段：收紧超时、熔断并返回兜底值、读接口走缓存快照、非关键写异步化、配置开关一键关闭高成本模块。
- 副作用控制：灰度降级（10% -> 50% -> 100%），全程盯核心成功率、核心 RT、投诉量，恢复后逐步回开。

#### 如果确认是 DB 瓶颈，临时如何处理

目标是先把 DB 压力打下来并保护核心链路：

1. 限流与降级：限制非核心查询和报表流量。
2. 慢 SQL 止血：优先处理 Top N 慢 SQL，减少全表扫、大分页、`select *`。
3. 缓存兜底：热点读加缓存与互斥回源，降低 DB 回源量。
4. 资源保护：收紧连接池等待与重试次数，避免重试风暴放大故障。
5. 读写分担：有条件时把读流量切到只读副本，主库优先保障写事务。

一句话收口：DB 瓶颈场景先控流降级，再做慢 SQL 和缓存双向降压，同时收紧连接与重试策略，最后通过读写分担把主库拉回安全区。

#### 展开版总结（3 分钟可复述）

这类故障题我会按"可用性优先"处理：先保核心交易，再谈体验完整性和性能最优。流程是先止血、再定位、再修复、最后复盘。

- 止血阶段（0-10 分钟）：入口限流、下游熔断、非核心降级、必要回滚，目标是先把错误率和 P99 拉回可控区间。
- 定位阶段（10-30 分钟）：按网关 -> 应用 -> 缓存 -> DB -> 下游分层排查，用指标 + 日志 + Trace 做时间线证据链，不靠经验拍脑袋。
- 修复阶段（30 分钟后）：短期做慢 SQL 止血、缓存兜底、重试收敛、连接与线程保护；长期做容量治理、资源隔离、多级缓存和自动化预案。
- 复盘阶段（事后）：沉淀影响范围、根因证据、处置动作、恢复时间、告警缺口和改进责任人，确保问题不重复发生。

面试收口可以这样说：我会先保主流程可用，通过控流、熔断和降级快速止损；再用分层定位和证据链确认根因；最后短期修复 + 长期治理双轨推进，把这次故障转化为系统能力提升。

#### 真实生产案例版

场景：`order-service` 的订单预览接口在发布后 5 分钟出现故障，`P99 RT` 从 `180ms` 升到 `3.2s`，超时率从 `0.3%` 升到 `11%`，CPU 只有 `35%`，业务线程池队列从 `0` 堆到 `1800+`。

第一步我不会先改线程池，而是先确认时间线和变更关系：

- `14:05` 发布完成
- `14:08` RT 开始抬升
- `14:10` 超时率上升
- `14:12` 线程池队列明显堆积

这个时间关系说明问题与发布强相关，优先怀疑代码逻辑、SQL、配置或依赖调用链变化。

然后我会快速看四组监控：

- 应用层：CPU、内存、GC
- 线程池：`active`、`queue`、`reject`
- 下游依赖：RPC RT、超时率、失败率
- 存储层：DB/Redis RT、连接池等待、慢 SQL

这次我看到：

- CPU 和 GC 都正常，说明不是计算打满或内存抖动
- 业务线程池 `active=64`、`queue=1834`
- Redis 和 DB 指标没有先明显异常
- 但 `promotion-service.queryAvailableCoupons` 的 RPC `P99` 从 `50ms` 飙到 `2.4s`

接着我抓线程 dump，发现大量业务线程状态是 `WAITING/TIMED_WAITING`，主要卡在 `CompletableFuture.get()` 和 RPC 调用上，而不是大量 `BLOCKED` 在同一把锁上，所以基本排除了锁竞争。

再结合 Trace 看调用链，发现订单预览接口总耗时 `3.1s`，其中本地计算只有 `40ms`，Redis `12ms`，DB `18ms`，而优惠券 RPC 占了 `2.7s`。说明线程池堆积不是根因，而是线程被下游慢调用长时间占住后的结果。

继续查 `promotion-service`，最终定位到新版本一条 SQL 没命中联合索引，导致查询退化。到这里证据链闭环：

1. 发布后开始异常
2. 上游线程主要在等待下游
3. 下游慢点集中在单个 RPC
4. RPC 变慢根因是新 SQL 退化

所以最终结论不是线程池本身，也不是锁竞争，而是下游 SQL 退化引发的慢调用，再传导为上游线程池堆积。

#### 三类问题怎么区分

##### 1. 线程池参数问题

常见特征：

- `active` 很快打满
- `queue` 持续堆积
- `reject` 可能开始出现
- 线程不一定都在等待下游，也可能是本地任务执行不过来

注意：线程池堆积不等于线程池就是根因，它很多时候只是最先暴露出来的承压点。

##### 2. 锁竞争

常见特征：

- CPU 不一定高
- 吞吐下降、RT 变长
- 线程 dump 中大量线程为 `BLOCKED`
- 调用栈集中卡在同一个 `synchronized` 或锁对象上

典型根因：

- 全局锁
- 粗粒度锁
- 热点 key 竞争
- 把耗时逻辑放进锁里

##### 3. 下游慢调用

常见特征：

- CPU 不高，但线程池队列堆积
- 线程状态更多是 `WAITING` / `TIMED_WAITING`
- 调用栈中出现 RPC、socket read、Future.get、数据库查询等待
- Trace 中耗时集中在某个下游依赖

这类场景下，线程池通常是受害者，不是根因。

#### 止血动作为什么这么做

一旦确认问题在慢依赖，我会先做下面几件事：

1. 降级非核心能力：例如关闭优惠券推荐，只保留订单基础价格预览。
2. 收紧超时：避免业务线程长时间阻塞在下游上。
3. 熔断异常依赖：持续高超时率时直接返回兜底值。
4. 必要时回滚：如果与发布强相关且回滚成本低，优先回滚。

为什么不先扩线程池？因为线程当前是在等待下游，扩线程池只会制造更多并发请求去打下游，反而把故障放大。

#### 5 个高频追问

##### 为什么没有第一时间扩线程池？

因为线程池队列堆积只是现象，不一定是根因。我要先判断线程是在执行、等锁还是等下游。如果线程都卡在 RPC/DB 上，扩线程池只会把下游打得更惨。

##### 怎么快速区分锁竞争和下游慢调用？

最低成本的手段就是线程 dump + Trace：

- 锁竞争：大量 `BLOCKED`
- 下游慢调用：大量 `WAITING/TIMED_WAITING`，卡在 RPC/Future/socket read

再结合依赖 RT 和日志交叉验证。

##### 如果下游一时修不好，怎么保主链路？

原则是非核心依赖不能拖垮核心链路。做法是降级非核心能力、收紧超时、熔断异常依赖、必要时返回兜底值。

##### 怎么证明不是数据库整体性能问题？

如果是数据库整体问题，通常多个调用方都会受影响，DB CPU、IO、连接、整体慢查询都会上升。若只是单个服务、单条 SQL 的 `P99` 异常，并且和新版本变更吻合，更像局部 SQL 退化。

##### 如果没有 APM 怎么排查？

仍然可以用监控、线程 dump、超时日志、慢日志、依赖服务日志串出时间线。APM 提高效率，但不是唯一手段，核心还是分层排查能力。

#### 专业名词解释

- `RT`：响应时间，请求从进入系统到返回结果的耗时。
- `P99`：99 分位响应时间，表示 99% 的请求都不超过这个耗时，能更好反映长尾问题。
- `长尾问题`：少量极慢请求把整体体验拖差，平均值往往掩盖不了它。
- `线程 dump`：JVM 当前所有线程的状态和调用栈快照，用来排查锁竞争、死锁、慢调用、线程阻塞。
- `WAITING / TIMED_WAITING`：线程在等待某个结果、锁条件或超时返回，不是在忙计算。
- `BLOCKED`：线程正在等待进入被其他线程持有的同步块，常见于锁竞争。
- `Trace`：一次请求在多个服务之间流转的完整调用链。
- `APM`：应用性能监控系统，用于观测接口耗时、调用链、异常和依赖慢点。
- `慢 SQL`：执行时间显著超过预期阈值的 SQL。
- `SQL 退化`：原本较快的 SQL 因索引不匹配、数据量增长或执行计划变化而变慢。
- `联合索引`：多个字段组成的索引，适合多条件过滤和排序。
- `熔断`：当依赖异常时暂时不再真实调用，直接返回兜底结果，避免故障传染。
- `降级`：关闭或弱化非核心功能，把资源优先让给核心主链路。
- `回滚`：恢复到上一个稳定版本，是线上故障中最常见的低风险止血动作。
- `SLO`：服务等级目标，例如 99% 请求在 200ms 内完成。

#### 面试最终收口模板

你可以这样回答：

“这类故障我会先判断线程是在算、在等锁还是在等下游。CPU 不高但线程池队列堆积时，我会优先怀疑等待型问题，而不是计算型瓶颈。然后通过监控、线程 dump、Trace 和慢日志形成证据链，区分到底是线程池参数问题、锁竞争还是下游慢调用。处理顺序上先止血，再定位，再治理，避免把线程池堆积这种现象误判成根因。”

## 如何做容量评估和压测，避免上线后被流量打垮？

### 标准面试回答

容量评估要先给出流量模型，再做压测验证：

1. 估算峰值 QPS、并发、读写比、依赖调用次数。
2. 计算单请求资源开销（CPU/内存/IO）。
3. 用基准压测 + 混合压测验证，并预留安全余量。

### 详细讲解

压测不能只看单接口极限，还要看"混合流量 + 真实依赖"。
很多系统单测表现很好，上线仍崩，是因为压测未覆盖下游限流、数据库锁竞争、消息堆积等真实场景。

上线前至少明确三条线：
- 告警线（提前预警）。
- 降级线（自动触发保护）。
- 崩溃线（必须人工干预）。

## 可观测性怎么设计，才能支撑线上快速定位问题？

### 标准面试回答

可观测性核心是 Metrics、Logs、Tracing 三位一体：

1. Metrics 看系统健康（QPS、RT、错误率、资源指标）。
2. Logs 看业务细节（结构化日志 + TraceId）。
3. Tracing 看跨服务链路（定位慢点和失败点）。

### 详细讲解

好的可观测性不是"采集很多数据"，而是"问题来时能快速定位"。
所以告警必须可行动，指标必须有维度（接口、实例、机房、租户），日志必须能按 TraceId 串联。

面试里可以补一句：观测体系要和应急预案绑定，否则看到了问题也不一定能快速止损。

## 大促期间峰值 QPS 涨到平时的 5 倍，如何设计限流、熔断与降级策略？

### 标准面试回答

限流、熔断、降级是分层防守，不是一选一：

- 限流：控入口流量，按令牌桶/滑动窗口控速率。
- 熔断：保护下游，当失败率超阈值切断调用链。
- 降级：兜底可用，熔断后返回缓存/默认值/友好提示。

优先级：先限流稳住入口，再熔断保护下游，最后降级保核心功能可用。

### 详细讲解

#### 一、为什么是分层而不是三选一

```
流量进来
   |
   v
[限流] -> 超过阈值直接拒绝
   |
   v
[熔断] -> 下游失败率超阈值，切断调用
   |
   v
[降级] -> 切断后返回兜底结果
   |
   v
用户拿到响应（不是报错）
```

- 限流解决"我能扛多少"
- 熔断解决"下游坏了怎么办"
- 降级解决"熔断后用户看到什么"

#### 二、限流算法对比

##### 计数器算法

原理：固定时间窗口内统计请求数，超过就拒绝。

优点：简单好实现。

缺点：临界抖动问题。比如请求 A 在 0.99s，请求 B 在 1.01s，看似在两个窗口但实际只隔 20ms。

##### 滑动窗口算法

原理：把时间窗口切分成更细的小窗口，统计最近 N 个小窗口的总量。

优点：比计数器精确，抖动少。

缺点：实现稍复杂，仍然是时间窗口统计而非真正速率。

##### 令牌桶算法（最常用）

原理：有一个桶，按固定速率往里放令牌，每个请求必须拿到令牌才能处理，桶有容量上限。

```
桶容量: 100
放令牌速率: 每秒放 50 个

T=0: 桶里有 50 个
T=1: 桶里有 100 个（上限）
T=1.5: 来了 80 个请求，只能处理 80（桶里还有 20）
T=2.5: 又来 80 个，只能再处理 20（剩下的 60 被拒绝）
```

优点：允许一定突发流量，长期速率稳定，是互联网最常用方案。

缺点：实现稍复杂，需要考虑并发安全。

##### 漏桶算法

原理：请求像水一样流进桶里，桶按固定速率往外漏，桶满就拒绝。

优点：输出速率完全平稳，对下游保护最强。

缺点：不允许任何突发，即使下游空闲也只能慢慢处理。

##### 一句话总结对比

```
算法          突发支持   精度   复杂度   适用场景
-------------------------------------------------
计数器        差         低      低       简单粗暴
滑动窗口      中         中      中       稍微精确限流
令牌桶        强         高      中       大多数互联网场景
漏桶          无         高      中       下游需要绝对平稳
```

#### 三、分布式限流

##### 单机限流的致命问题

10 台机器，每台限流 100 QPS，总设计限流 1000 QPS。
但入口实际流量是 5000 QPS：

- 每台机器都觉得自己的 100 QPS 没超
- 5000 QPS 全部打进去
- 下游直接被打爆

这就是单机限流的局限性：各自为政，不知道全局状态。

##### Redis + Lua 方案（最主流）

思路：限流计数器存在 Redis 里，多台机器共用同一个 Redis，用 Lua 脚本保证原子性。

Redis + Lua 令牌桶（伪代码）：

```lua
local key = KEYS[1]
local capacity = tonumber(ARGV[1])
local rate = tonumber(ARGV[2])
local now = tonumber(ARGV[3])
local requested = tonumber(ARGV[4])

local bucket = redis.call('HMGET', key, 'tokens', 'last_time')
local tokens = tonumber(bucket[1]) or capacity
local last_time = tonumber(bucket[2]) or now

local elapsed = now - last_time
local to_add = elapsed * rate
tokens = math.min(capacity, tokens + to_add)

if tokens >= requested then
    tokens = tokens - requested
    redis.call('HMSET', key, 'tokens', tokens, 'last_time', now)
    redis.call('EXPIRE', key, 60)
    return 1
else
    return 0
end
```

这样多台机器共用同一个 Redis，全局不会超限。

#### 四、熔断 vs 限流

| | 限流 | 熔断 |
|--|------|------|
| 目的 | 控入口流量，不让太多请求进来 | 保护下游，下游坏了就切断 |
| 触发条件 | 请求速率超过阈值 | 下游失败率/超时率超过阈值 |
| 行为 | 超过阈值的请求直接拒绝 | 超过阈值的调用直接返回，不发请求到下游 |
| 解决什么问题 | 我自己能扛多少 | 下游能不能扛得住 |

---

#### 五、熔断器三状态（Hystrix / Sentinel）

```
         失败率超阈值
     <--------------------
     |                     |
[Closed]            [Open]
     |                     |
     |  探测成功            |  熔断计时器到期
     |                     |
     v                     v
[Half-Open] --------> [Open]
```

- **Closed（正常状态）**：统计请求成功/失败/超时，超过失败率阈值切 Open。
- **Open（熔断状态）**：所有请求直接返回降级结果，不发请求到下游。
- **Half-Open（探测状态）**：放少量请求试探下游，成功率高则切回 Closed，成功率低则继续 Open。

##### Hystrix 熔断器核心逻辑（伪代码）

```java
public void execute() {
    if (state.get() == CircuitState.OPEN) {
        return fallback();
    }

    try {
        T result = run();
        successCount.incrementAndGet();
        return result;
    } catch (Exception e) {
        failureCount.incrementAndGet();
        if (shouldOpen()) {
            state.set(CircuitState.OPEN);
        }
        return fallback();
    }
}

private boolean shouldOpen() {
    long total = successCount.get() + failureCount.get();
    if (total < circuitBreakerRequestVolumeThreshold) {
        return false;
    }
    double errorRate = (double) failureCount.get() / total;
    return errorRate > circuitBreakerErrorThresholdPercentage / 100.0;
}
```

##### Sentinel vs Hystrix

- Hystrix：用请求计数器统计失败率。
- Sentinel：用滑动时间窗口计数器统计，更精确。

#### 六、降级和熔断的配合

##### 为什么熔断后需要降级

没有降级：熔断打开 → 用户拿到报错。
有了降级：熔断打开 → 用户拿到兜底结果（缓存、默认值、友好提示）。

##### 降级分层策略

- **第一级**：返回缓存数据（用户感知内容旧了但不报错）。
- **第二级**：返回静态兜底（功能基本可用）。
- **第三级**：返回友好提示（"系统繁忙，请稍后再试"）。

##### 熔断恢复要逐步放量

- 不突然放全部流量，否则会把刚恢复的下游再次打挂。
- 先放 3~5 个请求试探，成功率高则逐步放量。

#### 七、常见坑

1. **限流只在本机生效**：每台机器限流 100 QPS，但入口 5000 QPS 全部打进去。解决方案：分布式限流，Redis + Lua。

2. **熔断阈值设得太低**：下游偶尔抖动就开熔断。解决方案：合理设置最小请求量和失败率阈值。

3. **降级逻辑没有提前准备**：熔断后才临时写降级代码，容易出错。解决方案：降级逻辑纳入架构设计，提前准备。

4. **限流后没有监控**：限流了多少请求不知道。解决方案：限流/熔断指标进监控，告警阈值明确。

---

#### 八、面试一句话总结

"限流是入口保护，按令牌桶/滑动窗口控速率；熔断是下游保护，当失败率超阈值切断调用链；降级是兜底可用，熔断后返回缓存/默认值。三者分层配合，限流先进、熔断次之、降级最后。分布式限流用 Redis + Lua 保证全局原子性，熔断器用 Closed/Open/Half-Open 三状态循环，Sentinel 用滑动窗口统计失败率更精确。"
