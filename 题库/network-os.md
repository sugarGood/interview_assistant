# 计算机网络与操作系统
## TCP 三次握手和四次挥手为什么这么设计？
### 标准面试回答
三次握手用于确认双方收发能力和初始序列号，避免历史报文建立伪连接。  
四次挥手是因为 TCP 全双工，双方关闭发送方向通常是独立动作。  
`TIME_WAIT` 的核心作用是：防止旧报文干扰新连接，并确保最后 ACK 可重传。

### 详细讲解
面试里建议从“可靠性目标”来解释握手挥手：
- 建连要确认双方状态一致。
- 断连要确保双方都完成数据发送。

`TIME_WAIT` 虽会占用端口，但它是协议可靠性的必要代价。  
如果简单粗暴缩短，可能在高并发下引入脏连接问题。

## 为什么会出现大量 TIME_WAIT？线上怎么优化？
### 标准面试回答
大量 `TIME_WAIT` 常见于短连接高频创建/关闭，尤其当本机是主动关闭方。  
优化优先级：
1. 连接复用优先（长连接、连接池）。
2. 减少无效短请求（批量化、合并请求）。
3. 最后才考虑内核参数调优，并做风险评估。

### 详细讲解
`TIME_WAIT` 多不一定是故障，关键看是否已导致端口耗尽或连接失败。  
排查时建议结合 `ss -s`、应用连接池指标、网关访问模式一起看，不要只盯内核参数。

很多线上事故是“先调参数，后发现根因是调用方疯狂建短连接”。

## 从用户态到内核态发生了什么？为什么频繁切换会影响性能？
### 标准面试回答
用户态访问受限资源（网络、磁盘、设备）时，会通过系统调用进入内核态。  
频繁切换性能下降，是因为每次切换都要做上下文保存/恢复和权限边界切换。

### 详细讲解
高并发服务里，性能瓶颈常在“调用次数”而非“单次调用耗时”。  
因此优化方向通常是减少系统调用频率：
- 批量读写代替小包频发。
- 采用零拷贝减少内核/用户空间数据复制。
- 用异步 IO 提升线程利用率。

## 进程和线程的本质区别？为什么线程切换通常更轻？
### 标准面试回答
进程是资源隔离单位，线程是 CPU 调度单位。  
同进程线程共享地址空间，因此线程切换通常比进程切换轻，但并非零成本。

### 详细讲解
线程切换虽然不需要完整切换地址空间，但仍有寄存器保存、调度决策、缓存失效等开销。  
当线程数远超 CPU 核数时，会出现明显的调度抖动和上下文切换开销。

这也是线程池要控上限、避免“线程越多越快”误区的原因。

## 常见 Linux 性能排查命令你怎么组合使用？
### 标准面试回答
我会先做资源维度定位，再做进程线程下钻：
- CPU：`top`、`pidstat`、`perf`
- 内存：`free`、`vmstat`、`pmap`
- 磁盘与网络：`iostat`、`sar -n DEV`、`ss -s`

核心方法是先判定瓶颈类型，再深入调用栈，不盲目全量排查。

### 详细讲解
排查效率来自“顺序”：
1. 先看系统整体（是否 CPU/内存/IO 饱和）。
2. 再看目标进程（是否线程池阻塞、GC 频繁）。
3. 最后看热点函数（火焰图或 perf 采样）。

这样能把分钟级止血和小时级根因分析衔接起来，避免定位漂移。

## Socket 到底是什么？它和 TCP 三次握手是什么关系？
### 标准面试回答
`socket` 不是协议，也不是一串代码，更准确地说，它是操作系统提供给应用程序的网络通信对象 / 接口。  
程序通过 `socket()`、`connect()`、`listen()`、`accept()`、`send()`、`recv()` 这些系统调用去操作它。

和 TCP 的关系可以这样理解：
- `TCP` 负责定义可靠传输规则。
- `socket` 是程序使用 TCP 能力的入口。
- `TCP 三次握手` 是内核基于 socket 按 TCP 协议完成连接建立的过程。

服务端通常有两类 socket：
- `监听 socket`：负责监听端口、等待连接。
- `连接 socket`：三次握手完成后，由 `accept()` 返回，用来和某个客户端单独通信。

一句话总结：
- `socket 是通信对象`
- `TCP 是通信规则`
- `三次握手是 socket 建立 TCP 连接时由内核完成的建连流程`

### 详细讲解
很多人卡住 socket，核心原因是把这几个概念混在一起了：
- `IP`
- `端口`
- `TCP`
- `HTTP`
- `socket`

建议这样分层理解：

- `IP`：找到哪台机器
- `端口`：找到机器上的哪个服务
- `TCP`：双方如何可靠传输数据
- `HTTP`：传输内容长什么样
- `socket`：程序拿来使用网络能力的对象

#### 1. socket 不是“一串代码”
代码里我们写的是：

```c
int fd = socket(AF_INET, SOCK_STREAM, 0);
```

这里：
- `socket(...)` 是函数调用
- `fd` 是返回给程序的文件描述符
- 真正的 socket 对象是在操作系统内核里

这个内核对象里会记录：
- 本地 IP 和端口
- 对端 IP 和端口
- 使用的是 TCP 还是 UDP
- 发送缓冲区和接收缓冲区
- 当前连接状态，如 `LISTEN`、`ESTABLISHED`
- TCP 相关控制信息，如序列号、窗口等

所以更准确地说：
- 程序写代码“创建 socket”
- 操作系统真正“生成并维护 socket 对象”

#### 2. 服务端为什么不只一个 socket
服务端一般至少有两类 socket：

1. `监听 socket`
- 通过 `bind()` 绑定端口
- 通过 `listen()` 进入监听状态
- 作用是“守在门口等人来连接”

2. `连接 socket`
- 某个客户端握手完成后，服务端调用 `accept()`
- 内核返回一个新的 socket
- 这个新 socket 专门负责和该客户端收发数据

所以服务端不是拿监听 socket 直接和所有客户端对话，而是：
- 一个监听 socket 负责接客
- 多个连接 socket 负责具体通信

#### 3. socket 和 TCP 三次握手怎么连起来
客户端调用：

```text
socket() -> connect()
```

服务端通常先做：

```text
socket() -> bind() -> listen()
```

当客户端执行 `connect()` 时，内核开始发起 TCP 三次握手。

三次握手过程：

1. 第一次：客户端发送 `SYN`
- 表示“我想建立连接”
- 客户端 socket 进入连接中的状态

2. 第二次：服务端返回 `SYN + ACK`
- 表示“我收到了你的请求，我也愿意建立连接”
- 此时服务端内核已经感知到有新连接到来

3. 第三次：客户端返回 `ACK`
- 表示“我也确认收到了你的响应”
- 至此连接正式建立

握手完成后：
- 客户端的 socket 进入 `ESTABLISHED`
- 服务端内核把这个连接放入可 `accept` 的队列
- 应用程序调用 `accept()` 后拿到一个新的连接 socket

所以不要把三次握手理解成“应用代码自己手搓包”，更准确地说是：
- 应用程序调用 `connect()` / `listen()` / `accept()`
- 操作系统内核根据这些调用去执行 TCP 协议流程

#### 4. 监听 socket、accept 和连接 socket 的关系
这是面试里最容易说不清的地方。

你可以这样描述：

- `listen()`：告诉内核，这个 socket 现在开始监听连接
- `accept()`：从已经完成握手的连接里取出一个，返回新的连接 socket
- 返回的新 socket 才是真正用于 `read/write` 或 `send/recv` 的通信对象

也就是说：
- 监听 socket 负责“等连接”
- accept 出来的 socket 负责“传数据”

#### 5. Redis 里怎么理解 socket
Redis 启动时会：
- 创建监听 socket
- 绑定 `6379`
- 调用 `listen()`

客户端连接 Redis 时：
- 创建客户端 socket
- 调用 `connect()`
- 内核完成 TCP 三次握手

握手完成后：
- Redis 通过 `accept()` 获得一个新的连接 socket
- 后续 Redis 读取命令、返回结果，本质上都是对这个连接 socket 的读写

这也是为什么 Redis 要配合 `I/O 多路复用`：
- 它本质上是在高效管理大量 socket
- 不是在并行执行大量命令

#### 6. 面试里容易混淆的点
- `socket` 不是协议，协议是 `TCP/UDP`
- `socket` 不是 HTTP，HTTP 数据也是通过 socket 传输
- `socket` 不是端口，端口只是服务入口编号
- `socket` 不是“一串代码”，代码只是操作 socket 的方式
- `监听 socket` 和 `连接 socket` 不是一个东西

#### 7. 一段可直接复述的回答
“socket 本质上是操作系统提供给应用程序的网络通信对象，程序不是直接操作网线，而是通过 socket 来建立连接、发送和接收数据。TCP 是传输协议，定义了如何可靠传输；socket 是程序使用 TCP 的入口。服务端通常先创建监听 socket，执行 bind 和 listen，客户端创建 socket 后调用 connect，内核就会基于这个 socket 发起 TCP 三次握手。握手完成后，服务端通过 accept 得到一个新的连接 socket，后续双方就是通过这个连接 socket 进行数据收发。所以 socket 是通信对象，TCP 是通信规则，三次握手是内核借助 socket 完成建连的过程。”
