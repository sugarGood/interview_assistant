# RocketMQ

## RocketMQ 基础入门

### 1. RocketMQ 是什么

RocketMQ 是一个分布式消息中间件，核心作用是让系统之间通过消息异步通信。常见价值有 4 个：

- `异步解耦`：下单后不用同步等积分、通知、发券全部完成
- `削峰填谷`：把瞬时高流量先写入 MQ，再慢慢消费
- `流量缓冲`：保护下游系统，避免数据库或服务被突发流量打垮
- `最终一致`：通过消息把多个系统的动作串起来

一句话理解：

```text
RocketMQ 的本质，是把“同步调用链路”改造成“事件驱动链路”。
```

### 2. 核心角色

- `Producer`：消息生产者，负责发消息
- `Consumer`：消息消费者，负责订阅并处理消息
- `Broker`：真正存储消息、投递消息的服务节点
- `NameServer`：路由注册中心，保存 Topic 和 Broker 的路由信息
- `Topic`：消息分类，相当于逻辑上的消息通道
- `MessageQueue`：Topic 下的物理队列，消息最终落在这里

可以这样理解：

```text
Producer -> Topic -> MessageQueue -> Consumer
                 |
               Broker
```

核心角色速查表：

| 角色             | 作用       | 你可以怎么理解       |
| -------------- | -------- | ------------- |
| `Producer`     | 发送消息     | 上游业务系统        |
| `Consumer`     | 消费消息     | 下游处理系统        |
| `Broker`       | 存消息、投递消息 | MQ 的核心服务器     |
| `NameServer`   | 提供路由信息   | 告诉客户端“消息该往哪发” |
| `Topic`        | 逻辑分类     | 消息主题          |
| `MessageQueue` | 物理队列     | 真正并发消费和落盘的单元  |

### 3. 一条消息的基本链路

一条消息从发送到消费，通常会经历下面几个步骤：

1. Producer 先从 NameServer 获取 Topic 的路由信息
2. Producer 选择一个 Broker 和对应的 MessageQueue 发送消息
3. Broker 持久化消息
4. Consumer 订阅 Topic，从对应队列拉取消息
5. Consumer 处理成功后返回消费结果

面试里最简洁的表达可以是：

```text
RocketMQ 的消息链路就是：生产者拿路由、发消息到 Broker、Broker 落盘、消费者订阅并消费。
```

消息流转图：

```text
+-----------+       query route        +--------------+
| Producer  | ----------------------> | NameServer   |
+-----------+                         +--------------+
      |                                        ^
      | send message                           |
      v                                        | register route
+-------------------+                          |
| Broker            | -------------------------+
|  - CommitLog      |
|  - ConsumeQueue   |
+-------------------+
      |
      | pull message / ack
      v
+-----------+
| Consumer  |
+-----------+
```

### 4. RocketMQ 常见消息类型

- `普通消息`：最常见，适合大多数异步场景
- `顺序消息`：保证同一业务键局部有序
- `延时消息`：消息过一段时间后才能被消费
- `事务消息`：保证本地事务和发消息结果最终一致

常见消息类型对比表：

| 消息类型   | 解决什么问题     | 典型场景       | 代价 / 注意点    |
| ------ | ---------- | ---------- | ----------- |
| `普通消息` | 异步解耦       | 下单后发通知、发积分 | 需要自己处理幂等    |
| `顺序消息` | 保证局部有序     | 订单状态流转     | 吞吐更低，容易局部阻塞 |
| `延时消息` | 延后执行       | 订单超时关单     | 注意时效精度和积压   |
| `事务消息` | 本地事务和发消息一致 | 下单成功后异步扣库存 | 不等于完整分布式事务  |

新手学习顺序建议：

1. 先理解普通消息
2. 再理解重复消费和幂等
3. 再看顺序消息
4. 最后看事务消息、堆积排查和线上治理

### 5. 为什么说 Topic 不是队列

- `Topic` 是逻辑概念，用来做消息分类
- `MessageQueue` 是物理存储和并发消费的最小单元
- 一个 Topic 底下通常会有多个 MessageQueue
- 是否能并行消费、是否会产生热点、顺序性怎么做，很多时候都和队列数有关

这题很容易被问到，标准一句话：

```text
Topic 是逻辑分类，MessageQueue 才是消息真正落盘和消费并发的物理单元。
```

### 6. 为什么 MQ 不能天然保证业务正确

RocketMQ 能保证的是“消息尽量可靠地送达和重试”，但它不直接保证你的业务结果一定正确，所以业务侧必须补这些能力：

- 幂等：防止重复消费导致重复扣款、重复发券
- 重试：临时失败后重新处理
- 死信队列：多次失败后隔离问题消息
- 对账补偿：修复最终一致场景下的长尾异常

一句话总结：

```text
MQ 解决的是消息传递问题，业务一致性仍然要靠业务系统自己设计。
```

### 7. 新手最容易混淆的 5 个点

- 不是所有消息都要顺序消费，顺序越强，吞吐越差
- 不是消息发送成功就等于业务成功，消费端也可能失败
- 不是用了 MQ 就自动实现分布式事务，很多时候只是最终一致
- 不是 `msgId` 一定适合做幂等键，很多场景更应该用业务主键
- 不是堆积了就一定是 Broker 问题，多数场景先查消费端

### 8. RocketMQ 面试最小知识框架

如果你是新手，至少先把这 6 个点记住：

1. RocketMQ 用来做异步解耦、削峰填谷、流量缓冲、最终一致
2. 核心角色有 Producer、Consumer、Broker、NameServer、Topic、MessageQueue
3. Topic 是逻辑分类，Queue 是实际存储和消费并发单元
4. 消息语义通常是“至少一次”，所以消费端要做幂等
5. 顺序消息一般是局部有序，不是全局有序
6. 事务消息解决的是本地事务和发消息的一致性，不等于完整分布式事务

### 9. 新手开口版

如果面试官让你先简单介绍 RocketMQ，你可以直接这样说：

```text
RocketMQ 是一个分布式消息中间件，主要用来做异步解耦、削峰填谷和系统间最终一致。它的核心角色包括 Producer、Consumer、Broker 和 NameServer。Producer 把消息发到 Broker，Consumer 再从 Topic 对应的队列里消费消息。RocketMQ 常见的能力包括普通消息、顺序消息、延时消息和事务消息。它一般提供至少一次投递语义，所以消费端通常还要自己做幂等、重试和补偿。
```

## RocketMQ 架构原理

### 1. NameServer 为什么看起来很简单

- `NameServer` 主要做路由发现，不负责真正存储消息
- Producer 和 Consumer 会定期从 `NameServer` 拉取 Topic 路由信息
- Broker 启动后会把自己的 Topic、队列、地址信息注册到 `NameServer`
- `NameServer` 之间通常不做强一致同步，而是各自独立、无状态部署

这样设计的好处是：

- 结构简单，容易水平扩展
- 某一个 `NameServer` 挂掉不会直接影响整个集群
- 路由查询压力和消息存储压力被拆开

一句话记忆：

```text
NameServer 更像“轻量路由中心”，不是 ZooKeeper 那种强一致协调组件。
```

架构关系图：

```text
                 +------------------+
                 |   NameServer A   |
                 +------------------+
                         ^
                         |
                 +------------------+
                 |   NameServer B   |
                 +------------------+
                    ^            ^
                    | register   | query
                    |            |
      +-------------+------------+-------------+
      |                                           |
+-------------+                           +-------------+
| Broker Master| <--- replicate -------> | Broker Slave |
+-------------+                           +-------------+
      ^                                           ^
      | send                                      | pull
      |                                           |
+-----------+                               +-----------+
| Producer  |                               | Consumer  |
+-----------+                               +-----------+
```

### 2. Broker 在集群里干什么

- 接收 Producer 发来的消息
- 把消息持久化到磁盘
- 维护 Topic 下的多个 `MessageQueue`
- 给 Consumer 提供拉取能力
- 处理重试、死信、事务回查等逻辑

如果把 `NameServer` 理解成“告诉你去哪”，那 `Broker` 就是“真正存消息和发消息的人”。

### 3. RocketMQ 为什么吞吐高

RocketMQ 的高吞吐主要来自这几个点：

- 顺序写磁盘：磁盘顺序写比随机写快很多
- 文件分段存储：存储结构针对消息场景做了优化
- 读写分离：路由、存储、消费职责拆分
- 队列模型：天然支持并发消费和水平扩展
- 批量拉取与批量发送：减少网络交互开销

面试里可以这样答：

```text
RocketMQ 吞吐高，本质上是因为它把消息写入做成了顺序写磁盘，同时通过多队列并发、批量收发和专门的存储结构降低了 IO 开销。
```

### 4. RocketMQ 的存储结构要知道什么

新手不需要一开始把所有源码细节背下来，但至少要知道这几个关键词：

- `CommitLog`：消息真正落盘的主日志，所有消息先顺序写到这里
- `ConsumeQueue`：消费队列索引，帮助 Consumer 按 Topic 和 Queue 快速定位消息
- `IndexFile`：可选索引，方便按 `key` 查消息

最简理解：

```text
CommitLog 存消息本体，ConsumeQueue 像目录索引，IndexFile 像按 key 查询的辅助索引。
```

存储结构图：

```text
Message
  |
  v
+----------------------+
| CommitLog            |  <- 消息本体顺序写入
+----------------------+
           |
           v
+----------------------+
| ConsumeQueue         |  <- 记录 offset、size、tag 等索引
+----------------------+
           |
           v
+----------------------+
| IndexFile            |  <- 按 key 辅助查询
+----------------------+
```

存储结构速查表：

| 组件             | 存什么    | 主要用途               |
| -------------- | ------ | ------------------ |
| `CommitLog`    | 消息正文   | 顺序写入，承担主存储         |
| `ConsumeQueue` | 位置索引   | 按 Topic/Queue 快速消费 |
| `IndexFile`    | Key 索引 | 辅助按业务 key 查消息      |

### 5. 刷盘机制怎么理解

RocketMQ 常见会考刷盘策略，核心是两类：

- `同步刷盘`：消息写入磁盘后再返回成功，可靠性更高，但延迟更高
- `异步刷盘`：先返回成功，再异步落盘，性能更好，但机器宕机时丢消息风险更高

这个问题本质是性能和可靠性的权衡。

刷盘策略对比表：

| 策略     | 成功返回时机  | 优点   | 风险 / 代价    |
| ------ | ------- | ---- | ---------- |
| `同步刷盘` | 真正落盘后返回 | 更可靠  | 延迟更高       |
| `异步刷盘` | 先返回，后落盘 | 性能更好 | 宕机时丢消息风险更高 |

### 6. 主从和高可用怎么理解

- 主节点负责处理读写
- 从节点负责同步主节点数据
- 主节点故障时，需要有主从切换或副本接管能力
- 如果追求更强一致和更高可用，通常会结合 `DLedger` 或其他一致性复制方案

新手先记一句：

```text
RocketMQ 的高可用，不只是多台机器，而是消息副本、主从同步和故障切换能力的组合。
```

## RocketMQ 消费与重试机制

### 1. 推模式和拉模式到底怎么理解

- RocketMQ 常说的“推模式”，本质上底层仍然是 Consumer 主动去拉消息
- 所谓“推”的体验，是客户端框架帮你持续拉取并回调消费逻辑

所以更准确的说法是：

```text
RocketMQ 本质上是拉模型，只是客户端封装成了接近推的使用体验。
```

消费模型理解图：

```text
应用代码视角：
Broker ----推送给你----> ConsumerListener

真实底层视角：
ConsumerClient ----主动拉取----> Broker
       |
       +---- 拿到消息后回调业务代码
```

### 2. 集群消费和广播消费

- `集群消费`：同一个消费组里的多台消费者分摊消息，适合大多数业务场景
- `广播消费`：每台消费者都会收到同一份消息，适合配置下发、通知同步等场景

面试常见一句话：

```text
集群消费解决“分摊处理”，广播消费解决“每个节点都要收到”。
```

消费模式对比表：

| 模式     | 消息会给谁    | 适合什么场景    | 风险点                |
| ------ | -------- | --------- | ------------------ |
| `集群消费` | 同组内分摊    | 大多数业务消费   | rebalance 会影响局部稳定性 |
| `广播消费` | 每个实例都收一份 | 配置同步、缓存刷新 | 实例越多，总消费量越大        |

### 3. 为什么会重试

消息会进入重试，通常是因为：

- 消费返回失败
- 消费超时
- 消费进程异常退出
- Broker 没收到成功确认

所以重试不是 bug，而是 MQ 保障“至少一次”的正常机制。

### 4. 普通消息的重试和死信

- 普通消息消费失败后，Broker 会按规则重新投递
- 如果多次重试仍失败，消息会进入 `DLQ`，也就是死信队列
- 死信的意义是把持续失败的问题消息隔离出来，避免一直拖垮正常消费链路

一句话记忆：

```text
重试是为了恢复临时失败，死信是为了隔离长期失败。
```

重试流程图：

```text
收到消息
   |
   v
消费成功? ---- yes ----> ACK 成功，结束
   |
   no
   v
进入重试队列
   |
   v
达到最大重试次数? ---- no ----> 再次投递
   |
   yes
   v
进入 DLQ 死信队列
```

### 5. 顺序消息为什么更容易卡住

- 普通并发消费里，一条消息失败不一定影响其他队列继续跑
- 但顺序消息要求同一队列内有序
- 所以前面一条消息失败时，后面同队列消息通常不能越过它继续执行
- 这就是为什么顺序消息更容易出现“局部阻塞”

普通消息 vs 顺序消息：

| 对比项    | 普通消息    | 顺序消息         |
| ------ | ------- | ------------ |
| 消费并发   | 更高      | 更低           |
| 失败影响范围 | 常常局部可控  | 更容易阻塞同队列后续消息 |
| 适合场景   | 大多数异步通知 | 订单、账户等状态流转   |
| 核心代价   | 需要幂等    | 需要在顺序和吞吐之间权衡 |

### 6. 消费端必须补哪些工程能力

消费端除了“能消费”，还必须补齐这几件事：

- 幂等：防止重复执行
- 超时控制：防止单条消息长期卡住线程
- 降级隔离：下游慢时避免拖死整个消费组
- 监控告警：及时发现积压、失败率、重试升高
- 补偿能力：对长尾异常做修复

### 7. 毒消息怎么处理

所谓“毒消息”，就是怎么重试都处理不了的消息，比如：

- 数据格式本身错误
- 关键字段缺失
- 业务状态已经不可恢复
- 代码逻辑对某类数据永远报错

处理思路通常是：

1. 快速识别是否为毒消息
2. 不要让它无限重试
3. 进入死信或隔离队列
4. 结合人工修复、补偿任务或数据纠正后再重放

### 8. 面试收口版

```text
RocketMQ 消费本质是拉模型，常见模式有集群消费和广播消费。为了保证至少一次投递，消息消费失败后通常会进入重试，多次失败后进入死信队列。普通消息和顺序消息在重试上的最大区别，是顺序消息容易因为前一条失败而阻塞后续同队列消息，所以消费端必须具备幂等、超时控制、隔离降级和补偿能力。
```

## RocketMQ 部署与线上治理

### 1. 生产环境最小部署思路

如果是学习环境，单机就够了；但生产环境一般至少要考虑：

- 多个 `NameServer`
- 多个 `Broker`
- 主从副本
- 消费者多实例部署

目标不是“能跑”，而是：

- 某台机器挂了不要全线不可用
- 消息不要轻易丢
- 消费能力能横向扩展

生产环境简化拓扑图：

```text
                 +------------------+
                 |   NameServer A   |
                 +------------------+
                 +------------------+
                 |   NameServer B   |
                 +------------------+

      +-------------------+    +-------------------+
      | Broker Master 1   |    | Broker Master 2   |
      | Broker Slave 1    |    | Broker Slave 2    |
      +-------------------+    +-------------------+

         ^          ^              ^           ^
         |          |              |           |
     Producer A  Producer B    Consumer A  Consumer B
```

### 2. Topic 和队列数怎么规划

- 队列数决定并发消费上限
- 队列太少，消费扩不起来
- 队列太多，会增加管理成本和 rebalance 成本
- 规划时要结合：
  - 峰值 TPS
  - 消费耗时
  - 消费者实例数
  - 是否有顺序消费和热点业务键

常见经验是：不要一上来只建 1 个队列，也不要无脑开特别多队列。

### 3. 线上最该盯哪些指标

至少要盯下面这些核心指标：

- 生产 TPS
- 消费 TPS
- 消息堆积量
- 消费延迟
- 重试次数
- 死信数量
- Broker 磁盘使用率
- Broker 磁盘 IO 和网络带宽
- 消费失败率
- 单条消费耗时

这些指标里，最关键的是“产消差”和“积压趋势”。

监控指标表：

| 指标       | 看它是为了判断什么 | 异常时常见含义     |
| -------- | --------- | ----------- |
| `生产 TPS` | 生产压力大小    | 活动流量、补偿任务放量 |
| `消费 TPS` | 消费能力是否跟上  | 消费逻辑慢、实例不足  |
| `消息堆积量`  | 是否已经积压    | 消费落后于生产     |
| `消费延迟`   | 用户感知是否变差  | 下游处理慢或消息堵塞  |
| `重试次数`   | 是否频繁失败    | 业务异常、依赖抖动   |
| `死信数量`   | 是否有长期失败消息 | 毒消息或代码缺陷    |
| `磁盘使用率`  | 存储是否吃紧    | 积压严重或保留周期过长 |
| `网络带宽`   | 网络是否成瓶颈   | 批量流量、同步压力大  |

### 4. 常见线上问题有哪些

- 消息堆积
- 重复消费
- 顺序消息阻塞
- Broker 磁盘打满
- 消费者 rebalance 频繁
- 死信持续增长
- 下游数据库或接口拖慢消费

所以真正的线上治理，不只是会发消息和收消息，而是要能看懂这些异常信号。

### 5. 容量规划怎么做

容量规划不是拍脑袋估机器数，至少要回答这几个问题：

- 峰值每秒多少消息
- 单条消息大小多大
- 消息保存多久
- 消费高峰有多高
- 是否有大促、补偿、批量任务这种突刺流量

因为这些因素会直接影响：

- 磁盘容量
- 网络带宽
- Broker 数量
- 队列数
- 消费者实例数

容量规划简表：

| 规划项      | 你要先知道什么 | 会影响什么          |
| -------- | ------- | -------------- |
| 消息峰值 TPS | 每秒最高写入量 | Broker 数量、网络带宽 |
| 单条消息大小   | 平均和峰值大小 | 磁盘、网络、刷盘压力     |
| 保留时长     | 消息保存多久  | 磁盘容量           |
| 消费耗时     | 单条处理快慢  | 消费者实例数、线程数     |
| 高峰类型     | 平峰还是突刺  | 是否要预留冗余能力      |

### 6. 线上治理的核心原则

- 先保证消息不丢，再追求性能
- 先控制故障扩散，再恢复吞吐
- 先定位瓶颈层，再决定扩容方向
- 先做监控告警，再谈稳定治理

### 7. 新手常见部署误区

- 误区一：开发环境能跑，就以为生产环境也够用
- 误区二：只关注 Broker，不关注消费端扩展能力
- 误区三：只建很少的队列，后面扩容很被动
- 误区四：没有监控和死信治理，出了问题只能人工猜
- 误区五：把 MQ 当黑盒，只会“重启试试”

### 8. 部署与治理开口版

```text
RocketMQ 在生产环境里至少要考虑多 NameServer、多 Broker、主从副本和消费者多实例部署。队列数决定消费并发上限，Broker 数量和副本策略决定容量与可用性。线上治理重点要看生产 TPS、消费 TPS、消息堆积、消费延迟、重试、死信以及 Broker 的磁盘和网络指标。真正稳定运行 MQ，关键不是会用 API，而是能做好容量规划、监控告警和异常治理。
```

## RocketMQ 里如果要求同一个订单的消息严格按顺序消费，你会怎么设计？

### 标准面试回答

RocketMQ 的顺序消息一般做的是同一业务键的局部有序，不是整个 Topic 全局有序。实现上，生产者要以 `orderId` 作为分片键，通过哈希把同一订单的消息发送到同一个 `MessageQueue`；消费者使用顺序消费模式，保证同一队列内的消息串行处理。这样同一个订单的创建、支付、发货就能按顺序执行。

如果其中一条消息消费失败，后续同队列消息会被阻塞并进入重试，所以顺序消费的代价是吞吐量下降。实际项目里通常通过增加队列数、按业务键分片，做到同一订单有序、不同订单并行，在顺序性和性能之间做平衡。

### 详细讲解

#### 1. 原理

- 顺序消息通常不是全局顺序，而是“同一业务键有序”，比如同一个订单号、用户号、账户号。
- 要做到这一点，本质上是两步：
  - 生产端保证同一个订单的消息总是发到同一个 `MessageQueue`
  - 消费端对这个队列内的消息按顺序串行消费
- 所以顺序性的最小单位其实是“某个业务分片”，不是整个 `Topic`。

#### 2. 实现

- 生产者侧：
  - 以订单号作为分片键，比如 `orderId`
  - 发送消息时根据 `orderId` 选择固定队列，常见做法是 `queueIndex = hash(orderId) % queueNum`
  - 这样同一个订单的创建、支付、发货消息就会进入同一个队列
- 消费者侧：
  - 使用顺序消费模式，而不是普通并发消费模式
  - 顺序消费的语义是“同一队列同一时刻只被一个消费线程按顺序处理”
  - 这样可以保证这个订单相关消息按进入队列的先后顺序被执行
- 如果业务规模大，不会只靠一个队列，而是拆成多个队列：
  - 不同订单可以落到不同队列，并行处理
  - 同一订单仍然只落一个队列，从而保持局部有序

#### 3. 消费失败与排查

- 如果顺序消费里某条消息失败，后面的同队列消息一般不能直接越过它继续消费，否则顺序就乱了。
- 结果就是：
  - 当前这条消息会重试
  - 同一个队列后续消息会被阻塞
  - 如果某个热点订单或脏数据一直失败，会拖住这一队列的吞吐
- 排查时我会看：
  - 是业务代码异常、参数脏数据、幂等冲突，还是下游依赖超时
  - 重试次数是否过多，是否出现“毒消息”
  - 是否因为某个分片键特别热点，导致单队列积压严重

#### 4. 顺序性与吞吐量权衡

- 顺序性越强，并发能力通常越差，因为同一分片只能串行处理。
- 实际设计里一般不会追求全局有序，而是追求“业务可接受的局部有序”：
  - 订单维度有序
  - 账户维度有序
  - 用户维度有序
- 提升吞吐的办法主要有：
  - 增加队列数，让不同业务键分散到更多队列
  - 选择合适的分片键，避免热点分片
  - 非必要场景不要强制顺序，能并发就并发
- 一句话总结就是：
  - “同一订单内保证顺序，不同订单之间并行处理”，这是顺序和性能的常见平衡点

#### 5. 常见误区

- 误区一：以为顺序消息等于整个 `Topic` 全局有序，这通常做不到，也没必要
- 误区二：生产者没有固定路由，同一订单消息被发到不同队列，那消费者再顺序消费也没用
- 误区三：忽略失败阻塞问题，导致顺序消息一旦异常就造成队列积压

## RocketMQ 为什么会出现重复消费？如果要保证消息不被重复处理，你会怎么设计消费端幂等方案？

### 标准面试回答

RocketMQ 一般只能保证至少一次投递，所以重复消费是常见现象。出现重复消费的原因通常包括消费者处理成功但没来得及确认、网络抖动、消费失败重试、消费者重平衡等。要解决这个问题，重点不是要求 MQ 绝不重复，而是让消费端具备幂等能力。

常见做法包括业务唯一键配合数据库唯一索引、基于业务状态机限制重复状态迁移，以及单独建设消费记录表。像订单类业务更适合用订单号加状态机，扣库存类要结合唯一业务单号、扣减流水和原子更新，发优惠券类则要用用户和发券动作的唯一标识做幂等控制。核心原则是：消息可以重复投递，但业务结果只能生效一次。

### 详细讲解

#### 1. 原理

- RocketMQ 通常只能保证“至少一次”，也就是消息至少会被投递成功一次，但在某些场景下可能会被重复投递。
- 出现重复消费的常见原因有：
  - 消费者业务执行成功了，但还没来得及提交消费确认，进程就挂了
  - 网络抖动导致 Broker 没收到确认，于是触发重投
  - 消费超时、消费失败进入重试
  - 集群重平衡时，某些消息可能再次被拉取
- 所以面试里要先讲明白：MQ 本身很难天然保证“绝对只消费一次”，真正要落地的是消费端幂等。

#### 2. 实现

- 消费端幂等的核心目标是：同一条业务消息重复执行多次，最终结果只生效一次。
- 常见做法有 3 类：

第一类：业务唯一键 + 唯一索引

- 给消息绑定稳定的业务唯一标识，比如 `orderNo`、`couponRecordId`、`bizId`
- 消费时先落库，依赖数据库唯一索引防止重复插入
- 这是最通用、最稳的一种方式

第二类：幂等状态机

- 适合订单、流程流转这类业务
- 例如订单状态只能从“待支付”变“已支付”，如果已经是“已支付”，重复消息直接忽略
- 本质上是用状态迁移约束重复执行

第三类：去重表 / 消费记录表

- 单独建一张消费记录表，记录 `msgKey` 或 `bizKey`
- 每次消费前先判断是否处理过
- 常用于跨库、跨服务、逻辑较复杂的场景

#### 3. 不同业务怎么设计

- 订单类：
  - 更适合“业务唯一号 + 状态机”
  - 比如同一个订单支付成功消息，第一次把状态从“待支付”改成“已支付”，后面再来相同消息直接返回成功
- 扣库存类：
  - 不能只靠“查一下”，因为并发下会有竞态
  - 更稳的是“业务单号唯一约束 + 原子扣减 SQL + 扣减流水”
  - 要保证同一个订单不会重复扣库存
- 发优惠券类：
  - 更适合“用户 ID + 券模板 ID + 业务单号”做唯一约束
  - 防止同一业务事件重复发券
  - 如果是“允许一人多张”的场景，就要把幂等键设计成“这次发券动作”的唯一标识，而不是简单按用户去重

#### 4. 排查

- 如果线上怀疑重复消费，我会优先看：
  - 消费日志里同一个 `msgId` 或 `bizKey` 是否多次处理
  - 消费确认是否超时，是否有重试
  - 幂等表、唯一索引、状态机约束有没有真正生效
  - 是否发生过消费者重启、网络抖动、消费组重平衡
- 重点不是“有没有重复消息”，而是“重复消息来了以后，业务有没有被重复执行”。

#### 5. 常见误区

- 误区一：以为 MQ 能天然保证严格一次消费，实际工程里成本很高，通常做不到
- 误区二：用 `msgId` 当唯一幂等键，但业务真正稳定的幂等键应该优先选业务主键
- 误区三：先查再处理但没有原子性，在高并发下仍然可能重复执行
- 误区四：只说“加分布式锁”，锁不是首选，很多场景唯一约束和状态机更简单、更稳

## RocketMQ 的事务消息是怎么实现的？为什么它不能直接等价替代分布式事务？

### 标准面试回答

RocketMQ 事务消息的目标，是保证生产者本地事务结果和消息投递结果最终一致。它的流程是：生产者先发送一条 `half message` 到 `Broker`，这条消息会先持久化但暂时不投递；然后生产者执行本地事务，比如写订单表；执行完后再向 `Broker` 返回 `commit` 或 `rollback`。如果 `Broker` 长时间没收到结果，就会发起事务回查，生产者根据本地事务日志或业务状态返回最终结果。

事务消息适合下单、支付、发券这类“本地事务成功后通知下游异步处理”的场景。但它不能等价替代完整分布式事务，因为它只解决了本地事务和发消息的一致性，不能保证所有下游服务统一提交或统一回滚。和本地消息表相比，它把消息一致性能力内置在 MQ 中；和 `TCC` 相比，它更轻量，但一致性边界更弱，适合最终一致场景。

### 详细讲解

#### 1. 原理

- RocketMQ 事务消息本质上是为了解决一个经典问题：
- 本地业务事务执行成功了，但消息没发出去；或者消息发出去了，但本地事务回滚了。
- 它的目标不是做“全链路分布式事务”，而是保证“本地事务结果”和“消息是否对下游可见”这两件事最终一致。

#### 2. 基本流程

- 第一步，生产者先发送一条事务消息到 `Broker`
- 这时候 `Broker` 不会立刻把这条消息投递给消费者，而是先保存成一条 `half message`
- 第二步，生产者收到 `Broker` 的确认后，开始执行本地事务，比如写订单表、更新状态
- 第三步，本地事务执行完以后，生产者再告诉 `Broker` 本地事务结果是：
- `commit`：消息可以正式投递给消费者
- `rollback`：消息作废，不投递
- `unknown`：当前没法确定结果，稍后由 `Broker` 发起事务回查

#### 3. half message 是什么

- `half message` 可以理解成“半消息”或者“暂时不可投递的预备消息”
- 它已经被 `Broker` 持久化了，但还不能被消费者看到
- 只有等生产者确认本地事务成功并提交后，它才会变成真正可消费的消息
- 这么设计的意义是：
- 避免“消息先被下游消费了，但本地事务其实失败了”的不一致问题

#### 4. 回查机制是干什么的

- 如果生产者执行完本地事务后，因为网络抖动、宕机、重启等原因，没有把第二次确认结果告诉 `Broker`
- 这时 `Broker` 就不知道这条 `half message` 应该提交还是回滚
- 所以 `Broker` 会在一段时间后主动向生产者发起事务状态回查
- 生产者收到回查请求后，会根据本地事务日志或业务表状态判断：
- 这笔本地事务到底成功了、失败了，还是仍然未知
- 然后再返回 `commit` 或 `rollback`
- 这就是事务消息能做最终一致的关键补偿机制

#### 5. 适用场景

- 事务消息适合“本地事务成功后，需要可靠通知下游异步处理”的场景，例如：
- 下单成功后通知库存系统扣减库存
- 支付成功后通知积分系统加积分
- 订单完成后通知营销系统发券
- 这类场景的特点是：
- 当前服务先完成本地写库
- 然后通过消息驱动下游异步协同
- 要求的是最终一致，不是所有服务同一时刻强一致提交

#### 6. 为什么不能直接等价替代分布式事务

- 因为事务消息只解决了“生产者本地事务”和“消息发送”之间的一致性
- 它并不能保证消费者那边的本地事务一定成功
- 也不能自动帮你解决多个下游服务之间的统一提交和统一回滚
- 所以它不是 `2PC/XA`，也不是完整的 `TCC`
- 下游消费失败时，仍然要靠：
- 消费重试
- 幂等设计
- 死信队列
- 对账补偿
- 人工兜底
- 面试里一句话可以概括为：
- `RocketMQ` 事务消息解决的是“消息出库一致性”，不是“整个分布式链路的强一致事务”

#### 7. 和本地消息表、TCC 的区别

- 和本地消息表比：
- 本地消息表是“业务表 + 消息表”一起落本地库，再靠定时任务或投递器把消息发出去
- 事务消息则把“半消息 + 回查”这部分能力交给 `RocketMQ`
- 两者本质目标类似，都是解决“本地事务和发消息一致性”
- 区别在于：
- 本地消息表更通用、可控性更高，但要自己维护投递器、扫描、重试
- 事务消息接入更直接，但依赖 `RocketMQ` 的事务能力和回查逻辑
- 和 `TCC` 比：
- `TCC` 是对多个服务资源做显式预留、确认、取消
- 一致性更强，但业务侵入大、实现复杂
- 事务消息更轻量，适合事件驱动的最终一致场景
- 但它做不到 `TCC` 那种跨多服务资源预留和显式补偿控制

#### 8. 常见误区

- 误区一：把事务消息理解成“用了之后就没有分布式事务问题了”
- 误区二：只讲 `half message`，不讲回查机制
- 误区三：以为生产者本地事务成功就万事大吉，忽略消费端失败、重试、幂等和补偿
- 误区四：把事务消息和本地消息表、`TCC` 混成同一层次，不讲适用边界

## RocketMQ 里如果消息出现大量堆积，你会怎么排查？

### 标准面试回答

排查 RocketMQ 消息堆积时，我通常先看消费端，再看 Broker，最后回看生产端，因为大多数堆积问题本质上是消费能力不足、消费异常重试或者下游依赖变慢。常见原因包括消费线程不足、单条消息处理耗时过长、毒消息反复重试、顺序消费阻塞、热点队列倾斜，以及 Broker 磁盘或网络瓶颈。

区分瞬时高峰和真实消费不足，关键是看堆积曲线、生产 TPS 与消费 TPS 的关系，以及消费耗时和异常率。如果线上已经积压很多消息，我会先按业务优先级止血，快速恢复消费能力，比如扩容消费者、增加线程、优化慢依赖、处理毒消息，必要时对生产端限流；恢复时再分批释放积压，避免二次打爆下游。最终要通过扩容、拆热点、优化消费逻辑和补齐告警来做长期治理。

### 详细讲解

#### 1. 原理

- 我排查消息堆积时，通常不会一上来就拍脑袋说是 Broker 问题，而是先看产消关系：到底是生产太快，还是消费太慢，还是消费其实卡死了。
- 所以排查顺序一般是：
- 先看消费端
- 再看 Broker
- 最后回看生产端
- 原因是大多数堆积问题本质上都出在“消费速度跟不上”或者“消费逻辑异常阻塞”，真正是 Broker 打满的情况反而没那么高频。

#### 2. 常见原因

- 常见堆积原因大概有 5 类：

第一类：消费端能力不足

- 消费线程数太少
- 单条消息处理耗时太长
- 消费逻辑里有大量同步 IO，比如查库、调远程接口、写文件
- 下游服务变慢，导致消费 TPS 掉下去

第二类：消费异常重试

- 消费一直报错，消息不断重试
- 某些“毒消息”始终处理失败
- 顺序消费场景下，一条消息失败会卡住后面的消息

第三类：分区或热点不均

- 某个 `queue` 特别热，其他队列很闲
- 同一个业务键过于集中，导致局部堆积
- 顺序消息按业务键路由时，很容易出现热点分片

第四类：Broker 或网络瓶颈

- `Broker` 磁盘 IO 高、刷盘慢
- 网络带宽打满
- 主从同步压力大
- 消费者拉取消息不顺畅

第五类：生产端突发洪峰

- 某个活动、定时任务、补偿任务瞬时放量
- 短时间生产速率暴涨，但消费端来不及跟上
- 这种不一定是故障，也可能只是“削峰系统正常工作”

#### 3. 怎么区分瞬时高峰还是消费能力不足

- 我会重点看 3 个信号：

第一，看堆积曲线

- 如果是短时间快速上涨，随后逐渐回落，通常是瞬时流量高峰
- 如果持续上涨且长时间不回落，说明消费能力长期低于生产能力

第二，看生产 TPS 和消费 TPS

- 如果生产 TPS 短时高于消费 TPS，但很快恢复，问题不大
- 如果长期都是生产 TPS > 消费 TPS，说明系统真的吃不住

第三，看消费耗时和异常率

- 如果消费耗时明显变长，或者异常重试上升，说明消费端出了性能或稳定性问题
- 如果消费逻辑没变慢，只是短时间消息量暴增，更像突发高峰

#### 4. 止血方案

- 如果线上已经堆积很多消息，我会先止血，再恢复：

第一步，先判断业务优先级

- 这批积压消息是核心交易消息，还是通知、积分、埋点这类可延迟消息
- 核心链路优先保正确性，非核心链路优先保吞吐

第二步，快速恢复消费能力

- 扩消费者实例
- 增加消费线程数
- 优化慢 SQL、慢接口
- 对下游依赖做降级、限流、隔离
- 如果有重试风暴，先定位毒消息，避免无效重试把系统拖死

第三步，必要时做流量控制

- 临时限流生产端
- 暂停非核心消息生产
- 对补偿任务、批处理任务错峰执行
- 避免积压继续扩大

第四步，分批清理积压

- 不要一次性把积压全部压回下游
- 要按消费能力逐步放量，防止“恢复时二次打爆下游”
- 如果消息有时效性，还要区分哪些必须补消费，哪些可以丢弃或降级处理

#### 5. 恢复后的复盘

- 恢复之后要补 4 件事：
- 看是否需要扩容 `queue` 数量或消费者规模
- 看是否要拆分热点业务键
- 看是否要优化消费逻辑，减少同步阻塞
- 看是否要建立堆积告警、消费耗时告警、重试告警

#### 6. 常见误区

- 误区一：一看到堆积就先怪 `Broker`，实际上多数问题在消费端
- 误区二：只会说“扩容消费者”，但不看是不是毒消息、顺序阻塞或下游慢
- 误区三：积压恢复时无脑全量重放，结果把数据库和下游服务再次打挂
- 误区四：只看消息总量，不看按 `queue`、按业务键的局部热点

## RocketMQ 里消息丢失可能发生在哪些阶段，分别怎么防？
### 标准面试回答

消息丢失要按链路拆三段：生产端、Broker 端、消费端。生产端常见问题是发送超时、异常未重试、发送结果未校验；Broker 端常见风险是异步刷盘宕机、主从未同步完成即故障；消费端很多是“看起来丢”，比如业务失败却误返回消费成功。

端到端防丢方案是：生产端做发送确认与失败重试，Broker 端做高可靠刷盘和副本高可用，消费端做先处理后确认、失败重试和幂等保护，再配合消息轨迹、对账补偿和告警体系形成闭环。

### 详细讲解

#### 1. 总体框架

- 消息丢失要按链路拆三段：
- 生产端发出去之前/过程中
- Broker 存储与复制阶段
- 消费端处理与确认阶段
- 端到端防丢目标是：`能发送`、`能落盘`、`能被消费`、`能追踪补偿`。

#### 2. 生产端为什么会丢

- 发送超时或网络抖动，Producer 没拿到发送成功结果
- 异步发送回调未正确处理，失败被静默吞掉
- 只发不校验返回状态，业务误以为已成功
- 发送重试配置不合理（重试次数太小或异常未重试）

防法：

- 发送后严格检查 `SendResult`，只把成功状态当成功
- 失败重试（含指数退避），并记录失败日志/告警
- 核心链路用同步发送或事务消息
- 给消息加业务唯一键，便于重试与对账

#### 3. Broker 端为什么会丢

- 异步刷盘下，Broker 异常宕机，内存数据未来得及落盘
- 主从复制未完成就主节点故障，切换后新主没有那条消息
- 磁盘故障或配置不当导致保留策略误删（不是实时“丢”，但结果等价）

防法：

- 关键链路使用更高可靠配置（如同步刷盘、可靠副本策略）
- 合理主从与高可用方案，避免单点
- 监控磁盘、复制延迟、Broker 异常并及时告警
- 对关键 Topic 做消息轨迹与抽样核对

#### 4. 消费端为什么“看起来丢消息”

- 业务处理失败但代码误返回成功，消息被确认后不再重投
- 消费成功了但业务写库失败，表现成“消息没到业务系统”
- 广播/集群模式理解错误，预期每个实例都收到但实际是集群分摊
- 过滤条件（Tag/SQL）配置错误，把消息过滤掉了

防法：

- 先执行业务再确认消费成功，失败要明确返回重试
- 消费逻辑幂等，避免重试导致副作用
- 严格区分消费模式与订阅过滤规则
- 用消费日志+业务流水+对账任务识别“已投递未生效”

#### 5. 端到端防丢方案（面试可直接背）

- 生产端：发送确认 + 重试 + 失败告警 + 业务唯一键
- Broker 端：高可靠刷盘/副本策略 + 高可用部署 + 关键指标监控
- 消费端：先处理后确认 + 幂等 + 重试/死信 + 毒消息隔离
- 治理层：消息轨迹、全链路日志、定时对账、补偿任务、人工兜底

一句话收口：

- `RocketMQ` 很难保证“绝不丢、绝不重”，工程上是通过“防丢 + 幂等 + 对账补偿”把风险收敛到可控。

### 防丢速查表

| 阶段 | 丢失/异常表现 | 核心防护手段 |
| --- | --- | --- |
| 生产端 | 发送超时、回调失败、状态未校验 | 发送确认、失败重试、事务消息、告警 |
| Broker 端 | 宕机前未落盘、主从未同步完成 | 同步刷盘、可靠副本、HA、磁盘与复制监控 |
| 消费端 | 误确认成功、过滤错误、业务落库失败 | 先处理后确认、幂等、重试死信、对账补偿 |

## RocketMQ 消费端处理很慢时，如何给出可落地的提速方案？
### 标准面试回答

RocketMQ 消费慢时我会先分层定位瓶颈：先看 MQ 指标，再看业务处理耗时、数据库慢查询和下游 RPC 超时。短期先止血，包括扩容消费者、提高并发、隔离慢依赖、处理毒消息、分批回放积压；中长期再做结构优化，比如优化消费逻辑、拆热点队列、优化 SQL 和容量规划。

提速过程中要注意一致性，要求消费幂等、先处理后确认，并通过对账补偿兜底，避免出现重复消费和数据不一致。

### 详细讲解

#### 1. 先定位瓶颈在哪一层

- 我会先看四组指标：
- `MQ 层`：消费 TPS、堆积量、拉取延迟、重试/死信
- `业务层`：单条消息处理耗时、失败率、线程池排队
- `数据库层`：慢 SQL、锁等待、连接池耗尽
- `RPC 层`：下游 RT、超时率、熔断/限流触发率
- 判断方法：
- 如果消费 TPS 上不去且业务耗时高，多半是业务或下游问题
- 如果业务耗时正常但拉取慢，排查 Broker 或网络
- 如果失败率高、重试猛增，先查异常和毒消息

#### 2. 短期止血动作

- 扩消费者实例、调高消费并发线程
- 快速隔离慢依赖：降级、超时收紧、熔断
- 对非核心消息限流或错峰，先保核心链路
- 识别毒消息，避免无效重试拖垮队列
- 恢复积压时分批放量，避免二次打爆 DB/RPC

#### 3. 中长期优化动作

- 优化消费逻辑：减少同步阻塞，能异步就异步
- 拆分热点 Topic/Queue，优化分片键，降低局部热点
- SQL 与索引优化，必要时引入批量写或缓存
- 建立分层监控与告警：TPS、RT、失败率、重试、死信、堆积趋势
- 做容量规划：峰值消息量、队列数、消费者实例数、下游承载上限

#### 4. 提速后避免不一致

- 消费端必须幂等（业务唯一键 + 唯一约束/状态机）
- 先处理业务再确认消费成功，失败要返回重试
- 顺序消息场景不能盲目提并发，避免乱序
- 建立对账与补偿任务，修复“已消费未生效”长尾问题

#### 5. 一句收口

- 提速不是单纯“加机器”，而是“先定位瓶颈 -> 快速止血 -> 结构优化 -> 一致性兜底”。

### 提速动作速查表

| 维度 | 快速动作 | 风险控制 |
| --- | --- | --- |
| 并发能力 | 扩实例、提线程 | 防止打爆 DB 与下游 RPC |
| 业务处理 | 优化慢逻辑、异步化 | 保证事务边界与幂等 |
| 数据访问 | 优化索引、批处理 | 避免锁冲突和热点行 |
| 异常治理 | 毒消息隔离、重试优化 | 防止重试风暴 |
| 稳定性 | 告警、对账、补偿 | 防止“提速后数据错” |
