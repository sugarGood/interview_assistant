# Interview Assistant Save Interview

用于把当前对话中的题目和答案保存到本地 `题库/*.md`，并尽量避免重复写入。

## Mission

- 把对话里的题目与答案沉淀为可复用的本地题库资产
- 保持题库结构清晰，不因重复写入而变乱
- 让后续 `review-quiz` 和 `weak-analysis` 能稳定复用这些内容

## When To Invoke

- 用户输入 `/save-interview`
- 用户说“保存这题”“把这道题存到题库”“把详细讲解整理后保存”

## Inputs

- 当前题目
- 当前题目的标准答案或详细讲解整理稿
- 详细讲解模式中的连续追问与回答内容
- 现有 `题库/*.md` 文件列表与文件内容

## Output Contract

1. 必须先识别题目和答案是否完整
2. 必须先判断归档目录
3. 必须先检查同名题和相似题
4. 必须在 `融合更新 / 追加新题 / 保持不变` 三者中选择处理动作
5. 保存后必须输出结构化回执

## Core Principles

- 保存目标是 `题库/*.md`
- 题库目录随技术方向动态扩展，缺失方向直接创建新文件
- 发现同名或高度相似题时，不要无脑追加
- 若无法自动判断归档目录或处理策略，必须先询问用户
- 优先更新已有相似题，而不是一律追加到文件末尾

## Quality Bar

- 不保存空题目或无意义答案
- 不因为保存而破坏现有题库结构
- 不在归类不明确时擅自乱放
- 不在相似题明显更优时用较差内容覆盖旧内容

## Save Formats

标准格式：

```md
## {题目}
{答案内容}
```

详细讲解整理格式：

```md
## {题目}
### 详细讲解整理
{整理后的讲解内容}
```

## Question Bank File Spec

题库文件建议遵循以下规范：

- 一个文件代表一个方向，例如 `redis.md`、`mysql.md`、`elasticsearch.md`
- 每道题以 `## 题目` 开头
- 题目下方直接跟答案，或使用 `### 详细讲解整理`
- 不建议把多道题写成无标题大段文本
- 若一次保存中识别出多个独立知识点，允许拆成多道题连续写入同一文件

示例：

```md
## 什么是缓存穿透，常见解决方案有哪些？
缓存穿透指请求的数据既不在缓存，也不在数据库中，导致请求持续打到数据库。
常见方案包括布隆过滤器、空值缓存和参数校验。

## refresh 和 flush 有什么区别？
### 详细讲解整理
refresh 解决可搜索性，flush 更偏持久化与 translog 清理。
```

## Filing Rules

- 先扫描当前 `题库/*.md` 的实际文件列表
- 优先归档到语义最接近的已有分类文件
- 若出现新方向且没有对应文件，直接创建 `{方向}.md`
- 若题目跨多个方向且无法唯一判定，先问用户选主目录
- 归类时综合参考题目标题、答案内容、文件名和文件内已有上下文

## Execution Workflow

1. 校验题目与答案完整性
2. 扫描当前题库目录和分类文件
3. 判定目标归档目录
4. 检查同名和相似题
5. 选择处理动作
6. 写入或更新目标文件
7. 输出保存回执

## Similarity Handling

保存前做最小相似题判断：

1. 是否同名题
2. 是否同一知识点但问法略有不同
3. 新答案是否比旧答案更完整
4. 是否只是旧题的补充问法或补充答案要点

处理动作只有三种：

- 融合更新
- 追加新题
- 保持不变

## Interaction Rules

- 任一时刻只问 1 个确认问题
- 目录确认、覆盖策略、继续保存等问题必须串行
- 需要用户做选择时，优先明确列出简洁选项
- 若当前界面支持可点击选项，可使用可点击方式
- 如果判断为相似题但无法确定该融合还是新建，必须先询问用户

## Execution Flow

1. 检查 `题库` 目录是否存在，不存在则创建
2. 识别要保存的题目与答案
3. 校验题目非空、答案至少有关键点
4. 判定归档目录
5. 检查同名题和相似题
6. 选择动作：融合更新 / 追加新题 / 保持不变
7. 写入对应文件
8. 输出保存回执

## Batch Save Rules

- 若用户要求保存本轮内容，可批量保存多题
- 批量保存时按目录分组写入
- 保持同一目录内题目连续，避免写乱
- 若内容来自 `mock-interview` 的详细讲解对话模式，必须先判断这些追问是“原题补充”还是“可独立成题”

## Detailed-Dialogue Save Rules

- 当保存源来自详细讲解对话模式时，不能只保存最初那一道题，必须同时审视后续追问
- 若后续追问只是对原题的补充澄清、例子展开或同一知识点细化，优先融合到原题
- 若后续追问已经形成独立的面试题，例如围绕不同概念、机制、排查思路或设计取舍展开，必须拆为多题保存
- 拆题后，每道题都必须满足“题目明确 + 答案可复用”的最低标准
- 若没有额外追问，仍按单题保存，不要为了拆题而拆题
- 若用户明确要求“结合详细讲解对话保存”，默认允许拆题，不必额外询问是否一题或多题，除非归类冲突或相似题处理策略不明确

## Write Strategy

- 同名且新答案更完整：融合更新原题，补充缺失关键点并去掉重复表达
- 同知识点但只是问法相近：优先融合到原题，必要时追加“补充问法 / 补充答案要点”
- 同方向但考察角度明显不同：作为新题追加
- 来自详细讲解对话的追问，若已达到独立面试题标准：拆为新题或分别融合到对应旧题
- 若新内容没有新增价值：保持不变

## Success Criteria

- 写入后题库结构仍然清晰
- 相似题不会无限重复堆积
- 用户能明确知道这次保存到底做了什么动作

## Do

- 先判定目录，再判定相似题，再决定写入动作
- 优先融合已有优质内容
- 保存后明确告诉用户目标文件和处理动作
- 保存详细讲解对话时，先做“是否拆题”的判断，再做相似题判断

## Don't

- 不要在归类不明确时直接乱写
- 不要发现相似题后无脑追加
- 不要用质量更差的新答案覆盖更好的旧答案
- 不要把详细讲解中的连续追问无脑糅成一题
- 不要把已经具备独立面试题价值的追问遗漏掉

## Save Receipt

保存后输出：

- 保存结果：成功 / 部分成功 / 失败
- 处理动作：融合更新 / 新增题目 / 跳过不变
- 新增题目数
- 跳过题目数
- 归档目录
- 目标文件

## Error Handling

- 若用户只给题目未给答案：先引导补全关键答案
- 若用户要求“仅保存题目”：明确标记 `答案待补充`
- 若已有内容明显更好：保持原内容不变并说明原因
- 若用户指定的新方向文件不存在：直接创建对应分类文件再写入
