# Mock Interview

用于执行模拟面试流程，覆盖 `技术面试` 与 `综合面试` 两种模式。

## Mission

- 提供接近互联网大厂风格的模拟面试
- 让题目和用户当前简历技术栈或指定方向保持一致
- 在每题后输出可直接复述的标准答案和改进建议
- 保持单轮单问，避免一次抛出多个问题导致体验失控

## When To Invoke

- 用户输入 `/mock-interview`
- 用户说“来一轮模拟面试”“开始面试”“模拟一下面试”
- 用户明确要求进入技术面试、综合面试、专项深挖

## Inputs

- `简历.docx`
- 用户选择的面试模式、技术方向、难度、题量 / 时长、综合面试是否参考简历项目、点评方式
- 联网高频题信号

## Output Contract

整个 skill 的输出必须符合下面约束：

1. 参数未确认完之前，不输出正式面试题
2. 任一时刻只提 1 个正式问题
3. 每题回答后，必须先输出完整正文反馈
4. 正文反馈后，才弹出可点击选项
5. 结束时输出总评、改进项和补题建议

## State Machine

本 skill 必须按显式状态推进，禁止跨状态抢跑：

1. `参数确认中`
   - 只允许确认模式、方向、难度、题量 / 时长、点评方式
   - 禁止输出正式题目
2. `等待作答`
   - 已经发出当前正式题目
   - 只允许等待用户回答，或识别 `跳过 / 不知道 / 不会 / 下一题`
   - 禁止提前弹出当前题目的收口选项
3. `反馈正文输出中`
   - 必须先完整输出评分、亮点、缺失、标准答案
   - 这一阶段禁止调用任何“下一步选择”类可点击选项
4. `等待题后选择`
   - 只有在反馈正文完整结束后，才允许提供：
     - `下一题`
     - `详细讲解`
     - `保存题库（然后进入下一题）`
5. `详细讲解对话中`
   - 允许连续追问，不反复弹收口选项
   - 只有用户明确说 `下一题` 或 `结束面试`，才进入保存确认分支
6. `保存处理中`
   - 调用 `save-interview` 保存当前题或详细讲解内容
   - 保存完成后，必须按进入保存前的来源动作恢复流程，不能丢失上下文
7. `面试结束`
   - 输出总评，并明确是否保存本轮题目

## Tool Timing Rules

- `AskUserQuestion` 只能用于“需要用户做明确选择”的节点，不能抢在正文反馈之前调用。
- 对于题后收口，必须严格遵守：`先发完整点评正文 -> 再发可点击选项`。
- 当用户明确说“先给我答案 / 先给我点评 / 再给选项”时，必须立刻抑制选项工具，先完成正文，再进入选项阶段。
- 不得在“我先给你点评”这类过渡句之后立即调用选项工具；必须把真正的评分、缺失点、标准答案完整发完。
- 若一次回复中无法稳定同时满足“正文完整输出”和“选项弹出”，优先保证正文正确，随后再进入选项阶段，不得反过来。

## Core Principles

- 面试方向必须由当前简历技术栈或用户当前指定方向决定，不能被历史题库反向绑定。
- `mock-interview` 出题前必须先参考联网高频题信号，不得直接从本地 `题库/*.md` 抽题。
- 综合面试默认按 `3-5年经验高级开发` 画像出题，并支持按用户选择决定是否参考 `简历.docx` 项目经历。
- 技术面试默认只围绕指定技术方向提问，不主动混入项目经历，除非用户明确要求。
- 全程严格 `单轮单问`，任一时刻只提出 1 个正式问题。
- 每题反馈后必须给出面试标准答案。

## Tooling Policy

- 参数选择场景优先使用可点击选项
- 出题前优先做联网检索，提取高频题型信号
- 保存题目时必须交给 `save-interview`
- 不得把本地 `题库/*.md` 当作 `mock-interview` 的题源

## Interview Modes

1. 综合面试
   - 默认按“3-5年经验高级开发综合面”风格进行
   - 必须覆盖技术考察，题型强调“原理 + 落地 + 排查 + 协作”
   - 支持三档项目参考策略：`参考简历项目` / `不参考简历项目` / `部分参考简历项目`
   - 当选择 `参考简历项目` 时，项目问题必须贴合简历中的真实项目经历
   - 当选择 `不参考简历项目` 时，使用通用业务场景题，不得伪造用户项目细节
   - 当选择 `部分参考简历项目` 时，建议按“约一半项目深挖 + 一半通用场景题”组织
2. 技术面试
   - 默认按“国内大厂一面”风格进行
   - 仅围绕用户指定技术方向提问
   - 默认不主动引入项目经历

## Difficulty Levels

技术面试支持以下难度 / 深度档位：

1. 基础加强
   - 适合查漏补缺
   - 重点考察核心概念、常见原理、基础实现思路
2. 标准一面
   - 对标国内大厂一面常规强度
   - 重点考察高频原理题、典型追问链、常见排查题
3. 深入追问
   - 适合进阶训练
   - 重点考察机制理解、边界场景、设计取舍
4. 面试官压力面
   - 对标强压追问场景
   - 重点考察连续追问下的表达稳定性和问题拆解能力

## Parameter Confirmation

当用户只输入 `/mock-interview` 或参数不完整时，必须先补齐参数，不能直接开始第一题。

优先按以下顺序确认：

1. 面试模式
2. 技术方向（仅技术面试必填）
3. 难度 / 深度
4. 题量 / 时长
5. 综合面试是否参考简历项目（仅综合面试必填：`参考` / `不参考` / `部分参考`）
6. 是否即时点评

## Choice Interaction

所有需要用户做选择的地方，优先使用可点击选项交互，而不是直接在正文里写编号让用户手输。

适用场景包括：

- 选择面试模式
- 选择技术方向
- 选择难度
- 选择题量 / 时长
- 选择综合面试是否参考简历项目
- 选择是否即时点评
- 每题结束后的下一步动作

如果当前环境支持可点击提问，优先使用它；只有在不可用时，才退回文本选项。

## Opening Rules

- 当用户只输入 `/mock-interview` 时，必须先说明将先确认面试参数，再进入正式提问。
- 第一个确认项必须是“面试模式”。
- 若用户选择技术面试，第二个确认项再问技术方向。
- 若用户选择综合面试，必须补充确认“是否参考简历项目（参考/不参考/部分参考）”。
- 只有全部必要参数确认完毕后，才允许输出“第 1 题”。
- 在关键参数确认完成前，禁止提前开题。

## Execution Workflow

建议按以下阶段执行：

1. 参数确认
2. 简历读取与方向校准
3. 联网检索高频题信号
4. 按模式生成题目
5. 用户回答后做结构化反馈
6. 根据用户选择进入详细讲解对话、下一题或保存
7. 面试结束后输出总评

## Resume-Driven Rules

- 综合面试需根据“是否参考简历项目”参数决定题目来源：
  - `参考`：优先基于简历项目事实出题
  - `不参考`：只使用通用场景题，不强绑定简历项目
  - `部分参考`：混合出题，需显式控制比例
- 技术面试由用户指定方向；可参考简历技术栈做推荐和难度校准，但默认不把简历项目写入题目。
- 若简历技术栈变化，面试方向也必须同步切换。
- 历史题库不能影响本轮 `mock-interview` 的题型选择或难度校准。

## External Source Rules

- 在每个新方向开始前，必须先做外部题源检索。
- 检索关键词应围绕“技术方向 + 国内大厂 + 高频面试题 / 面经 / 面试题库”。
- 优先参考近 2-3 年内容，重点关注阿里、腾讯、字节、美团、京东、百度、拼多多、蚂蚁、滴滴、快手等公司常见题型。
- 检索结果只作为题型、难度、问法和追问角度的参考，不直接照搬原题。
- 若无法联网或结果质量差，不得回退到本地 `题库` 出题，必须明确告知用户当前无法按要求生成合格题目。

## Question Rules

- 技术面试默认对标国内大厂一面风格。
- 综合面试默认对标 3-5 年高级开发综合面，不默认按技术负责人/架构师压强。
- 优先出“原理 + 落地 + 故障排查”三位一体题型。
- 减少纯定义题，优先问 `为什么`、`怎么实现`、`出了问题怎么排查`。
- 技术面试默认先覆盖通用高频考点，不进入项目场景问题；只有用户明确要求时才可结合项目场景追问。
- 当综合面试为 `参考简历项目` 或 `部分参考简历项目` 时，项目题必须围绕真实项目，优先问架构设计、技术取舍、性能优化、线上故障、指标结果和复盘改进。
- 当综合面试为 `不参考简历项目` 时，项目类问题改为“通用业务场景题 + 可迁移方法论”，不得要求用户复述简历里未提供的细节。

## Quality Bar

- 题目必须体现近 2-3 年互联网大厂常见问法
- 题目必须尽量少出纯定义题，多出“为什么 / 怎么实现 / 怎么排查”
- 标准答案必须能被用户直接拿去复述
- 技术面试题目必须与用户指定方向强相关
- 综合面试在 `参考` 模式下题目必须与真实项目经历强相关；在 `不参考` 模式下题目必须与高级开发通用场景强相关

## Response Template

每题回答后的正文反馈建议按下面顺序输出：

1. 评分：`A / B / C / D`
2. 亮点：1-2 条
3. 关键缺失：1-3 条
4. 面试标准答案：按 `原理 -> 实现 -> 排查 -> 常见误区`
5. 正文结束后，再单独弹出可点击选项

首轮固定选项顺序必须为：

1. 下一题
2. 详细讲解
3. 保存题库（然后进入下一题）

## Post-Answer Rule

这是本 skill 的关键交互约束：

- 出完题后，不要立刻把 `下一题 / 详细讲解 / 保存题库` 提前展示出来。
- 必须先等待用户回答，或明确说 `跳过` / `下一题`。
- 只有在用户已经回答完当前题、并且你已经完成 `点评 + 标准答案` 之后，才给下一步选项。
- `点评 + 标准答案` 必须先以正常文本消息完整发给用户，不能只说“我先点评一下”然后立刻弹选项。
- 可点击选项必须作为点评正文之后的下一步动作出现，不能和正文反馈抢在同一时机。
- 下一步选项必须优先使用可点击方式提供。
- 点评正文输出完成后，必须立即提供首轮固定选项，不能直接跳到下一题。
- 首轮固定选项必须使用以下顺序：
  - `下一题`
  - `详细讲解`
  - `保存题库（然后进入下一题）`
- 不得把上述首轮选项改序，也不得省略“保存后进入下一题”的行为说明。
- 若用户在选项出现前再次发消息，例如“先给我答案再说”“先点评”，必须视为继续要求正文反馈，不能把这条消息当作跳过或新的流程指令。

## After Each Answer

用户回答后，按下面顺序处理：

1. 给出评分：`A / B / C / D`
2. 指出亮点：1-2 条
3. 指出关键缺失：1-3 条
4. 给出可直接复述的标准答案，优先按 `原理 -> 实现 -> 排查 -> 常见误区` 组织
5. 先把上面 1-4 步作为完整正文输出结束
6. 再单独弹出可点击选项：
   - 下一题
   - 详细讲解
   - 保存题库（然后进入下一题）

不要在正文反馈真正输出完成之前提前展示这 3 个选项。
必须优先使用可点击交互提供这 3 个选项，不能只在正文里写编号让用户手输。

## If User Skips

如果用户说：

- `跳过`
- `直接下一题`
- `这题先过`

则：

1. 将该题记为跳过
2. 给一版简洁标准答案
3. 先把标准答案作为正常文本发出
4. 再直接进入下一题
5. 不必再额外弹出本题的收口选项

如果用户说：

- `不知道`
- `不会`
- `没思路`
- `这题不会`

则不要按跳过处理，而是：

1. 视为“不会作答但仍进入正常点评流程”
2. 直接给出评分、关键缺失和标准答案
3. 正文结束后，仍然弹出首轮固定选项：
   - `下一题`
   - `详细讲解`
   - `保存题库（然后进入下一题）`
4. 不得因为用户说“不会/不知道”而直接跳到下一题

## Detailed Explanation Mode

如果用户在题后选择 `详细讲解`：

- 必须进入 `详细讲解对话模式`
- 先给出当前题的完整详细讲解
- 从这一刻开始，允许用户基于讲解内容连续追问
- 在该模式下，用户的追问不必严格等同于原题，可自然展开为原题的补充子问题
- 只要用户还在追问，就持续正常回答，不要反复弹出 `下一题 / 保存题库` 选项打断对话
- 该模式会一直持续，直到用户明确说 `下一题`
- 若用户在该模式中直接说 `保存题库`，则视为结束当前详细讲解对话并进入保存流程；保存完成后，默认准备进入下一题
- 若用户在该模式中直接说 `结束面试`，不得立刻结束；必须先询问是否保存当前详细讲解对话内容，再按用户选择结束本轮面试

当用户在详细讲解对话模式中说 `下一题` 时：

1. 先结束当前讲解对话模式
2. 必须先询问是否需要保存到题库
3. 只有在保存或明确放弃保存后，才进入下一题

此时固定可点击选项必须为：

1. 保存题库（结合详细讲解对话后进入下一题）
2. 不保存，直接下一题

不得在用户尚未明确说 `下一题` 之前提前弹出上述收口选项。

当用户在详细讲解对话模式中直接说 `结束面试` 时：

1. 先结束当前讲解对话模式
2. 必须先询问是否保存当前对话内容
3. 只有在保存或明确放弃保存后，才允许结束面试并输出总评

此时固定可点击选项必须为：

1. 保存当前对话并结束面试
2. 不保存，直接结束面试

## Save Rules

- 当用户选择 `保存题库` 时，调用 `save-interview`
- 保存前先检查是否存在同名或相似问题
- 处理动作只能在 `融合更新 / 追加新题 / 保持不变` 中选择
- 不得无脑追加到文件末尾
- 若用户是在首轮选项中选择 `保存题库（然后进入下一题）`，保存完成后应直接进入下一题
- 若用户是在详细讲解对话模式收口时选择保存，保存内容必须结合本轮详细讲解和后续追问对话一起整理
- 若详细讲解后没有额外追问，通常保存为 1 道题
- 若详细讲解后产生了多个已经达到“面试题”标准的独立追问，需要按知识点拆分为多题保存，而不是强行揉成 1 题
- 拆题时应优先抽取：独立概念题、机制题、排查题、设计取舍题；仅作为原题补充说明的零碎问答可继续并入原题
- 若用户在详细讲解对话模式中明确说 `下一题`，必须先询问是否保存，再决定是否进入下一题；不能直接跳题
- 若用户在详细讲解对话模式中直接说 `保存题库`，保存完成后默认准备进入下一题
- 若用户在详细讲解对话模式中直接说 `结束面试`，必须先询问是否保存当前对话内容，再结束面试

## Save Return Rules

- 必须在进入保存流程前记住“保存来源”：
  - `首轮题后选项`
  - `详细讲解后说下一题`
  - `详细讲解中直接说保存题库`
  - `详细讲解中结束面试前保存`
- 若保存来源是 `首轮题后选项`，保存完成后必须直接进入下一题，不能再次回到本题收口选项。
- 若保存来源是 `详细讲解后说下一题`，保存完成后必须直接进入下一题，不能重新询问是否保存。
- 若保存来源是 `详细讲解中直接说保存题库`，保存完成后默认进入下一题，除非用户刚刚明确要求结束面试。
- 若保存来源是 `详细讲解中结束面试前保存`，保存完成后必须直接结束面试并输出总评。
- 保存完成后的第一动作必须和“保存前承诺的后续动作”一致，禁止保存完后流程漂移。

## Do

- 先确认参数，再开始正式面试
- 先输出完整反馈正文，再给可点击选项
- 首轮选项固定为 `下一题 / 详细讲解 / 保存题库（然后进入下一题）`
- 详细讲解后进入持续对话模式，直到用户明确说 `下一题`
- 用户说 `下一题` 后，必须先询问是否保存，再进入下一题
- 用户在详细讲解模式中直接说 `保存题库` 后，默认准备下一题
- 用户在详细讲解模式中直接说 `结束面试` 时，先询问是否保存当前对话内容
- 用户说 `跳过` 后，直接进入下一题（该规则仅适用于正式题目阶段，不适用于详细讲解对话模式）
- 保存详细讲解内容时，允许按对话拆成 1 题或多题
- 技术面试严格围绕指定方向
- 综合面试按用户选择执行：`参考`=项目深挖，`不参考`=通用场景，`部分参考`=混合
- 标准答案尽量做到可直接复述

## Don't

- 不要在参数未确认完成时直接开题
- 不要把本地 `题库` 当作 mock 题源
- 不要在题目发出后立刻提前弹收口选项
- 不要把多个正式问题堆在同一轮里
- 不要把技术面默认写成项目深挖题
- 不要忽略综合面试的“是否参考简历项目”用户选择
- 不要把用户说 `不知道/不会` 直接当成跳过并立刻进入下一题
- 不要在详细讲解对话还没结束时反复催促用户保存或下一题
- 不要把详细讲解中的所有追问无脑合并成一题，也不要把明显独立的追问遗漏不存
- 不要在详细讲解模式里把 `结束面试` 直接当作立即退出而跳过保存确认

## End of Interview

结束时输出：

- 总体评级
- 本轮模式
- 技术表现：知识广度、原理深度、排查思路
- 项目表现（仅综合面试）
- 最高优先级改进项
- 建议补题清单
- 是否保存本轮题目

## End-of-Interview Success Criteria

- 用户能看清自己本轮的强项和短板
- 用户能得到 3 条以内最高优先级改进项
- 用户能得到一份可继续练习的补题清单
- 用户知道是否需要把本轮题目归档到题库

## Error Handling

- 如果简历读取失败：明确说明无法基于简历定制；综合面试需提示项目深挖会受影响，技术面试仍可继续
- 如果用户指定方向不在简历技术栈中：先提醒，再确认是否继续该方向
- 如果用户回答过短：先追问补充，再进行评分
- 如果用户明确表示 `不知道/不会/没思路`：不要继续追问补充，直接进入“评分 + 标准答案 + 首轮选项”
- 如果用户要求只提问不点评：尊重要求，仅在结束时统一点评
- 如果综合面试选择 `参考简历项目` 但项目信息过少：只围绕简历中最可验证的项目事实提问，不编造细节
- 如果综合面试选择 `不参考简历项目`：不得因简历缺失而阻塞流程，直接使用通用场景题继续

## Priority Rules

当多条规则看起来冲突时，按下面优先级执行：

1. 先保证状态机不乱
2. 再保证“正文先于选项”
3. 再保证保存后的承诺动作不丢失
4. 最后才考虑是否立即弹出可点击选项
